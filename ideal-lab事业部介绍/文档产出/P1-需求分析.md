# P1 需求分析

## 文档信息
- 文档名称：ideal-lab Marketplace 与最佳实践介绍
- 文档类型：产品介绍文档（推广使用）
- 目标读者：事业部同事（技术+管理+产品混合背景，了解AI辅助开发但不熟悉Claude Code生态）
- 核心目标：让事业部同事了解并开始使用 ideal-lab，包含实操指南
- 分析时间：2026-05-11

## 功能需求

### 场景一：认知 ideal-lab — "这是什么？为什么需要它？"

- **目标**：让读者理解 ideal-lab 的定位、价值和解决的问题，建立"我需要这个工具"的认知
- **当前状态**：事业部同事对 ideal-lab 一无所知或仅有模糊概念，不清楚它与日常工作的关系
- **输入**：ideal-lab 仓库的 README、marketplace.json、9个插件的元数据
- **输出**：清晰的 ideal-lab 概念介绍，包含：
  - 什么是 Claude Code Plugin Marketplace
  - ideal-lab 在生态中的定位（最佳实践插件库）
  - 核心价值主张：将团队最佳实践标准化为可复用的 Skill
  - 解决的痛点：重复劳动、质量不一、经验难传承
- **价值**：建立第一印象，激发兴趣，回答"这跟我有什么关系"

### 场景二：理解插件架构 — "怎么组织的？"

- **目标**：让读者理解 ideal-lab 的组织方式和插件架构，建立结构化认知
- **当前状态**：读者不了解 Plugin/Skill 的概念和组织方式
- **输入**：仓库目录结构、plugin.json 格式、SKILL.md 格式、AGENT.md 治理规则
- **输出**：清晰的架构图解，包含：
  - Plugin → Skill → References/Scripts 的三层结构
  - 每个 Plugin 的自包含性（技能+参考文档+脚本+模板）
  - Marketplace 分发机制（添加源 → 按需安装 → 斜杠命令调用）
  - 版本管理策略（changeset 自动管理）
- **价值**：让读者理解"这不是一个黑箱，而是透明、可审计、可定制的工具集"

### 场景三：深入了解 ideal-document-workflow — "典型工作流长什么样？"

- **目标**：以 ideal-document-workflow 为案例，展示 ideal-lab 工作流插件的核心设计理念
- **当前状态**：读者不了解工作流编排的细节，不理解"12阶段流程"意味着什么
- **输入**：ideal-document-workflow 的 SKILL.md、各阶段 Skill 的 SKILL.md、参考文档
- **输出**：深入但易懂的工作流解析，包含：
  - 12阶段流程全景图（P1需求→P3大纲→P5拆分→P7写作→P9配图→P11渲染，穿插评审）
  - 编排器模式：主智能体不做任何工作，只读取状态、调用Skill、更新状态
  - 评审团队机制：读者/作者/专家三方评审 + 修改者整合 + 熔断保护
  - 两种执行模式：手动模式 vs YOLO模式
  - 关键最佳实践：What-Why-How 写作模式、按需调研、写作风格规范
- **价值**：让读者理解 ideal-lab 不仅提供工具，更提供方法论和流程保障

### 场景四：纵览9大插件 — "还有什么能力？"

- **目标**：快速展示全部9个插件的能力全景，让读者发现与自身工作相关的工具
- **当前状态**：读者只知道名字，不知道各插件的具体能力
- **输入**：9个插件的 plugin.json、各 Skill 的 SKILL.md frontmatter
- **输出**：结构化的插件能力矩阵，按使用场景分类：
  - **开发全流程**（ideal-dev-workflow）：15阶段从需求到交付
  - **PPT全链路**（ideal-ppt-suite）：13阶段从调研到导出PPTX
  - **文档写作**（ideal-document-workflow）：12阶段结构化写作
  - **知识库构建**（ideal-knowledge-base）：材料分析到文档生成
  - **深度调研**（ideal-deep-research）：4种深度模式，10+信息源
  - **AI图片生成**（baoyu-image-gen）：多Provider支持
  - **Dify工作流**（ideal-dify-generator）：DSL生成与校验
  - **任务澄清**（ideal-ralph）：苏格拉底式迭代验证
  - **仓库维护**（ideal-lab-maintainer）：自动化提交/发布/校验
- **价值**：让每位读者都能找到与自己工作相关的工具

### 场景五：实操入门 — "我怎么开始用？"

- **目标**：提供清晰的从零到一的入门指南，降低使用门槛
- **当前状态**：读者想用但不知道从何开始
- **输入**：README.md 的快速开始部分、安装命令、使用示例
- **输出**：分步骤的实操指南，包含：
  - 前置条件：Claude Code 安装和配置
  - 第一步：添加 Marketplace 源
  - 第二步：选择并安装插件
  - 第三步：调用 Skill（斜杠命令示例）
  - 典型使用场景的完整示例（如用 document-workflow 写一份文档）
  - 常见问题和排错
- **价值**：让读者在15分钟内完成首次使用，建立信心

## 非功能需求

| 需求类型 | 描述 | 度量指标 |
|----------|------|----------|
| **可读性** | 技术与管理混合背景都能读懂，避免过度技术化 | 非技术读者能理解80%以上内容 |
| **可操作性** | 读完就能上手操作，不需要额外查阅资料 | 从读到用<30分钟 |
| **准确性** | 所有命令、路径、描述与仓库实际一致 | 0错误 |
| **完整性** | 覆盖 ideal-lab 核心能力和典型使用场景 | 9个插件均有涉及 |
| **视觉辅助** | 关键概念配合图表说明 | 至少3处架构/流程图 |
| **篇幅控制** | 足够详细但不过长，避免信息过载 | 总篇幅6000-8000字 |

## 技术约束

| 约束类型 | 描述 |
|----------|------|
| **输出格式** | Markdown，支持 Mermaid 图表 |
| **语言** | 中文为主，技术术语保留英文原文 |
| **信息来源** | 仅使用仓库内的实际代码和文档，不虚构功能 |
| **受众适配** | 技术读者需要看到足够的架构细节；管理读者需要看到价值主张和ROI |
| **风格** | 产品介绍风格，不是技术文档也不是营销材料 |

## 复杂度评估

- **功能模块数**：5个核心场景（认知→架构→工作流详解→全景→实操）
- **预估总篇幅**：7000-8000字
- **建议拆分**：5-6个写作任务
  - 任务1：ideal-lab 概述与价值主张（~1200字）
  - 任务2：插件架构与组织方式（~1000字）
  - 任务3：ideal-document-workflow 深度解析（~2500字，核心章节）
  - 任务4：9大插件能力纵览（~1500字）
  - 任务5：实操入门指南（~1000字）
  - 任务6（可选）：图表与视觉辅助（3-5个 Mermaid 图）
- **信息充分度**：高（仓库内资料完整，无需额外调研）

## 关键写作要点

1. **开头必须有 hook**：不要从定义开始，从痛点或场景切入（"你有没有遇到过……"）
2. **理想读者旅程**：认知（为什么）→ 理解（是什么）→ 信任（怎么做到的）→ 行动（怎么用）
3. **ideal-document-workflow 是核心卖点**：用最多篇幅展示，这是最能体现 ideal-lab 能力的案例
4. **实操部分必须有完整示例**：不能只有命令，要展示一个从输入到输出的完整流程
5. **避免内部术语**：不要使用"sub-agent""编排器""熔断"等术语而不解释，或者用通俗语言替代
