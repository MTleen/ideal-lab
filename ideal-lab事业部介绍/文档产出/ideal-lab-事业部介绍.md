# ideal-lab：从日常工作中沉淀的 Claude Code 最佳实践插件库

## 1. 从痛点出发：ideal-lab 是怎么来的

### 1.1 真实场景里的反复困扰

写一份技术方案，从打开空白文档到完成初稿，往往需要大半天。中间反复修改结构、补充论据、调整措辞，改完一版回头看，发现逻辑链条又断了。PPT 更是如此——从零开始排版、选风格、配图，一份 20 页的汇报材料能吃掉两三天。

代码评审看似有流程，实际常常变成"扫一眼、打个 LGTM"的走过场。不是不想认真审，而是没有结构化的评审框架，评审质量完全依赖个人经验和投入时间。更棘手的是经验传承问题：一个项目中踩过的坑、总结出的写法、验证过的方案，到了下一个项目常常要从头再来一遍，因为之前的经验没有被系统化地保留下来。

这些不是假设性的场景，是我在日常工作中反复遇到的困扰。

### 1.2 从重复中提炼，从迭代中沉淀

ideal-lab 的起点，就是在解决这些问题的过程中逐步形成的。最初没有"设计一个插件库"的想法，只是在一个个具体项目里，反复经历同一个模式：用 AI 辅助完成某项任务，发现输出质量不稳定，于是总结经验、写下来、固化成可复用的流程，下一次直接调用。

第一次用 AI 写技术文档时，发现生成的内容结构松散、深度不足。总结原因后发现，问题出在提示词不够具体——没有告诉 AI 应该遵循什么结构、达到什么深度、面向什么读者。于是把这些要求整理成一份写作规范，下次直接喂给 AI，输出质量立刻提升了一个档次。再后来发现，光有写作规范还不够，还需要需求分析、大纲评审、多轮校验这些前置和后置环节。每发现一个薄弱点，就补充一个对应的环节，逐步形成完整的写作工作流。

每一个 Skill 背后都有真实项目的影子。ideal-ppt-suite 里 17 种预设风格的来源，是在实际做 PPT 的过程中，发现不同场景需要不同的视觉策略，于是逐一沉淀下来的。ideal-deep-research 的 4 种深度模式，是因为调研任务的时间预算和深度要求差异很大，quick 模式和 ultradeep 模式面向的是完全不同的使用场景。

这个"踩坑—总结—固化—复用—再迭代"的循环，是 ideal-lab 成长的核心路径。不是先画架构图再填内容，而是从一个个具体的痛点出发，逐步生长出一套完整的工具体系。

### 1.3 ideal-lab 今天的样子

经过多轮迭代，ideal-lab 目前包含 **9 个插件**、**43 个 Skill**，覆盖开发流程、PPT 生成、文档写作、知识库构建、深度调研等场景。

两个核心概念先说清楚：

- **插件（Plugin）** 是能力单元。每个插件对应一个独立的工作领域，插件之间完全独立，按需安装，用哪个装哪个。
- **技能（Skill）** 是执行单元。每个 Skill 做一件事，定义了"做什么"和"怎么做"。一个插件包含若干 Skill，它们按流程编排形成完整工作流。Skill 除了跟随工作流自动运行，大部分也可以单独调用。

ideal-lab 的 9 个插件按用途分为三类：4 个工作流套件（覆盖文档、PPT、开发、知识库四大场景）、3 个独立工具（调研、图片生成、Dify 工作流生成）、2 个开发辅助（任务澄清、仓库维护）。

## 2. 工作流套件：四个完整流程

工作流套件是 ideal-lab 的核心。每个套件都是一个"编排器 + 多个 Phase Skill"的组合——编排器负责调度流程，Phase Skill 负责执行具体阶段。你可以调用编排器一键走完全流程，也可以单独调用某个 Phase Skill 做特定环节。

四个工作流套件共享同一个设计哲学：**质量来自流程，而非来自灵光一现**。每个套件都是"产物阶段 → 评审阶段"交替推进，产物阶段生成内容，评审阶段把关质量。这不是为了增加流程的仪式感，而是在实践中反复验证过的事实——没有质量关卡的 AI 输出，大概率是结构混乱、内容遗漏、风格不统一的。评审关卡让问题在早期被发现和修正，而不是累积到最终交付物上才暴露。

另一个共享的设计是**两种运行模式**：手动模式（每个阶段完成后停下来等你确认）和 YOLO 模式（自动推进直到完成或触发熔断）。手动模式适合首次使用或重要文档，YOLO 模式适合日常任务或时间紧迫的场景。底层逻辑完全一致，区别仅在于是否在每个阶段停一停。

### 2.1 ideal-document-workflow — 结构化文档写作

**做什么：** 从一句话需求出发，经过需求分析、大纲生成、任务拆分、并行写作、智能配图、渲染输出，自动完成一篇结构完整的文档。你的参与集中在"确认需求"和"审核大纲"两个节点，其余环节由工作流自动完成。

**适用场景：** 技术方案、产品文档、汇报材料、知识沉淀等任何需要结构化写作的场景。

**设计理念：** 这个工作流的核心思路是"先锚定需求，再围绕大纲写作"。文档质量的决定因素不是 AI 的生成能力，而是前期需求的清晰度和大纲结构的合理性。一份需求明确、大纲扎实的文档，写作阶段几乎不需要人工干预；反过来，如果需求模糊就开始写，后期返工的代价远大于前期多花半小时理清需求。

另一个关键设计是"并行写作 + 统一规范"。把大纲拆成独立章节、分配给多个子智能体同时写作，可以大幅提升速度。但并行写作的固有风险是风格不统一——五个智能体写出来的东西拼在一起像五个人的作品。解决方案是所有智能体遵循同一份写作风格规范（`writing-skills` Skill 提供），确保最终文档在风格上保持一致。

**包含 8 个 Skill：**

| Skill | 做什么 | 能否独立使用 |
|-------|--------|-------------|
| `ideal-document-workflow` | 编排器：协调下面 7 个 Skill 自动走完 12 阶段流程 | 否（编排器） |
| `requirement-analysis` | 解析你的需求，明确文档类型、目标读者、内容范围 | 能 |
| `outline-generation` | 基于需求生成大纲，支持在线调研或本地材料分析 | 能 |
| `task-split` | 将大纲拆分为可并行执行的写作任务 | 能 |
| `document-writing` | 多个子智能体并行写作各章节，统一风格规范 | 能 |
| `illustration` | 分析内容自动配图（流程图用 Mermaid，架构图用 AI 生成） | 能 |
| `document-render` | 将各章节组装为最终文档，支持 Markdown / Word / PPT 格式输出 | 能 |
| `writing-skills` | 写作风格规范（元技能，被其他 Skill 引用） | 能 |

**完整流程（12 阶段）：**

整个流程是"产物阶段 → 评审阶段"交替推进。评审阶段的逻辑一致：从读者、作者、专家三个视角审核产出物质量，发现问题则修改后复查，最多 3 轮，仍不通过则触发熔断（停止自动推进，等待人工介入）。下面重点展开每个产物阶段的具体做法。

**P1 需求分析** — 有两种执行模式。**模式 A**：你提供了一份现成的需求文档（PRD、规格说明书等），Skill 解析文档内容，提取结构化字段——功能需求（场景名称、目标、现状、输入、输出、价值）、非功能需求（性能、安全、可靠性、兼容性）、技术约束（技术栈、集成要求、合规要求），生成一份标准化的 `P1-需求分析.md`。**模式 B**：你没有现成文档，只有一堆原材料（项目代码、设计稿、会议纪要等），Skill 会扫描这些材料，识别项目类型，判断知识覆盖情况（哪些信息已有、哪些还缺），输出需求分析报告，包含目标读者画像、读者读完后应该能回答的核心问题、知识缺口清单。

**P3 大纲生成** — 首先读取 P1 的需求分析，识别哪些部分需要额外调研。然后做"按需调研"：如果 P1 标记了"需要在线调研"，调用 `ideal-deep-research` 搜索相关信息；如果你提供了原材料路径，就读取源码和文档提取信息；如果两者都不需要，直接进入大纲生成。大纲按 1-3 级标题组织，每个标题都带有字数估算（一级标题基础 500-800 字 + 每个子节 200 字，二级标题基础 300-500 字 + 每个子节 150 字）。同时预判每个章节是否需要配图（流程图、架构图等），标注配图类型。输出 `P2-大纲.md`。

**P5 任务拆分** — 读取 P2 大纲，按一级标题拆分为独立的写作任务。如果某个一级标题估算超过 10000 字，就按二级标题进一步拆分。每个任务的目标体量是 3000-6000 字（目标 5000 字）。拆分的关键是**上下文打包**：每张任务卡片不只有章节标题，还携带了原需求文档的相关内容、父级大纲的描述、兄弟任务的标题（避免重复写同样内容）、特殊写作要求。这些上下文确保独立写作的章节拼在一起时逻辑连贯。输出 `P3-任务拆分.md`。

**P7 并行写作** — 读取任务卡片，为每个写作任务 spawn 一个子智能体。每个子智能体执行同样的流程：先判断当前任务是否需要调研（技术细节需要外部验证就调用 `ideal-deep-research`，需要从源码提取信息就读取源码文件，已有足够素材就跳过调研直接写），然后按写作规范执行写作。写作时强制使用几种模式：**What-Why-How 模式**（每个功能点先说是什么、再说为什么选它、最后说怎么做）、**对比模式**（传统方案 vs 新方案，用量化数据对比）、**分层架构模式**（总体 → 第一层 → 第二层 → 层间交互）。所有子智能体遵循同一份写作风格规范（无第一人称、无问句标题、无口语化表述、段落以叙述为主、数值标注来源），确保并行写作的各章节风格统一。输出 `sections/*.md`。

**P9 智能配图** — 扫描所有章节文件，检测四类关键词：流程描述关键词（触发流程图）、架构描述关键词（触发架构图）、数据描述关键词（触发数据图）、已有的 Mermaid 代码块。生成一个"配图候选列表"，向你确认哪些位置需要配图、用什么类型的图。配图生成按内容类型选择工具：**流程图和时序图**用 Mermaid CLI 将文本描述转换为 PNG（可编辑，白底）；**架构图、数据图、路线图**用 `baoyu-image-gen` AI 生成（图中的文字标签必须是中文）。生成后将图片引用插入对应章节。原则是"内容决定形式"——不是每段都配图，只在图片比文字更有效地传达信息时才配。

**P11 渲染输出** — 根据你在流程状态中指定的输出格式组装最终文档。**Markdown 格式**：直接保留各章节文件，不额外处理。**Word 格式**：先将所有 .md 按文件名顺序合并，预处理 Markdown 格式（清理换页符、水平线、空行、表格格式），通过 Pandoc 转换为 Word（LaTeX 公式转为 OMML 格式），再用 python-docx 做中文排版后处理（正文明宋/Times New Roman 12pt 1.5 倍行距首行缩进 2 字符、一级标题黑体 16pt、二级标题黑体 14pt、A4 页面 2.54cm 边距）。**PPT 格式**：通过 PptxGenJS 将内容转换为幻灯片。

**两种运行模式：**

| 维度 | 手动模式 | YOLO 模式 |
|------|---------|----------|
| 控制粒度 | 每阶段确认 | 全自动或熔断时介入 |
| 适用场景 | 首次使用、重要文档 | 日常文档、时间紧迫 |
| 人工参与 | 6-12 次确认 | 0-1 次介入（仅熔断时） |

### 2.2 ideal-ppt-suite — PPT 全链路生成

**做什么：** 从一句话需求出发，经过调研、策略规划、大纲设计、提示词工程、HTML 生成、AI 配图、PPTX 导出，最终输出一份可编辑的 PPT 文件。支持 17 种预设视觉风格（科技蓝、极简白、数据驱动等），也可以自定义风格。

**适用场景：** 方案汇报、产品路演、培训课件、竞品分析、季度总结等需要制作 PPT 的场景。

**设计理念：** PPT 制作的核心矛盾是"内容质量"和"视觉设计"经常互相拖后腿。内容好但排版丑，没人看得下去；排版漂亮但内容空洞，看完等于没看。ideal-ppt-suite 的核心思路是**把内容决策和视觉决策分开处理**——先通过调研和策略规划把"说什么"和"怎么说"定清楚，再通过提示词工程和 HTML 执行把视觉呈现做到位。

这个设计的关键在"八项确认"（策略规划阶段）：在 AI 开始生成任何幻灯片之前，强制和你确认 8 项设计决策。先把这些决策明确下来，后续生成才能有章可循，而不是 AI 随机发挥。

另一个关键设计是"HTML 作为中间格式"。HTML 提供了像素级的视觉控制——CSS 可以精确控制位置、大小、颜色，而 PPTX 的生成库在这方面约束很多。生成 HTML 幻灯片后，再通过自动化工具将 HTML 转换为可编辑的 PPTX（保留原生形状和文本），既保证了视觉效果，又保证了下游可编辑。

**包含 9 个 Skill：**

| Skill | 做什么 | 能否独立使用 |
|-------|--------|-------------|
| `ideal-ppt-workflow` | 编排器：协调下面 8 个 Skill 走完 13 阶段流程 | 否（编排器） |
| `ideal-ppt-research` | 通过对话理解你的需求，确认 PPT 的受众、目的、内容来源 | 能 |
| `ideal-ppt-strategist` | "八项确认"：和你确认格式、页数、受众、风格、配色、图标、字体、图片策略 | 能 |
| `ideal-ppt-outline` | 用金字塔原理构建叙事结构，设计每页内容布局 | 能 |
| `ideal-ppt-prompt` | 将大纲转化为精确的 HTML 生成指令 | 能 |
| `ideal-ppt-image` | 根据设计规格生成 AI 配图（背景、插画、图标等） | 能 |
| `ideal-ppt-executor` | 将每页指令渲染为独立 HTML，包含内联样式和 SVG 图形 | 能 |
| `ideal-ppt-export` | 将 HTML 转换为可编辑的 PPTX（原生形状/文本）和 PDF | 能 |
| `ideal-ppt-generator` | 简化路径：跳过完整流程，直接从内容生成幻灯片图片并合成 PPTX | 能 |

**完整流程（13 阶段）：**

**P1 需求研究** — 分四步执行。**第一步：源内容处理**，接受 PDF、Word、URL、Markdown、口头描述甚至图片作为输入，全部转换为 Markdown 文本，做质量检查（内容 >100 字、结构保留、无乱码）。**第二步：内容信号提取**，自动分析出语言（中/英/日）、内容长度映射到建议页数（<1K 字 = 5-10 页，1K-3K = 10-18 页，3K-5K = 15-25 页，>5K = 20-30 页）、内容类型（数据分析/技术方案/战略规划/产品发布/教育/叙事）和推荐的视觉风格。**第三步：苏格拉底式提问**，根据信息充分度决定提问数量（信息充足 = 1-2 个确认性问题，部分充足 = 2-3 个场景问题，信息不足 = 3-4 个全面问题），一次只问一个问题，每个问题附选项，最多 3 轮提问，你可以跳过任何问题。**第四步：生成调研报告**，综合分析结果和你的回答，输出一份 8 节结构的 `research.md`（核心信息、受众画像、演示目的、内容信号、推荐风格、页数范围、画布格式、来源摘要）。

**P3 策略规划（八项确认）** — 分三个阶段。首先解读 `research.md`，提取主题、受众、场景、内容框架和用户偏好。然后一次性呈现 8 项设计决策（不是逐个问，而是一起呈现让你批量确认）：**画布格式**（16:9/4:3 PPT、小红书、故事模式、方形、微信头图等）、**页数范围**（根据内容框架自动计算：开场 1-2 页 + 目录 1 页 + 核心章节每章 2-5 页 + 结尾 1-2 页）、**受众与目的**（受众类型、演示目的、认知负荷级别）、**视觉风格**（模式 A：17 种预设风格选一个；模式 B：按纹理/情绪/排版/密度 4 个维度自定义组合）、**配色方案**（60-30-10 规则：主色 60%、辅色 30%、强调色 10%）、**图标策略**（Emoji/AI 生成/内置图标库/无图标）、**字体方案**（中文 + 英文字体配对，分标题/正文/强调三种）、**图片策略**（无图片/用户提供/AI 生成/占位符）。全部确认后，生成 `design-spec.md`（12 节，涵盖画布规格、视觉主题、字体系统、布局原则、图标规格、图表参考、图片资源清单等）。

**P5 大纲生成** — 读取 `research.md` 和 `design-spec.md`，用金字塔原理构建叙事结构。叙事流分三段：**开场**（1-2 页：悬念钩子 + 背景上下文 + 内容预告）、**正文**（N 页，选用 Problem→Solution、SCR 情境-冲突-解决、What/Why/How、时间线、并列对比等叙事模式）、**结尾**（1-2 页：综合总结 + 行动呼吁 + 收束）。每页用 **Bento Grid 卡片布局**分解内容块——确定核心内容块、映射到卡片类型、规划卡片大小和排列，通过卡片大小区分层级关系。同时生成 `STYLE_INSTRUCTIONS`：从 design-spec 提取颜色系统、字体栈、字号层级、间距系统等 CSS 自定义属性，供后续生成 HTML 时直接引用。每页的元数据包含类型（封面/章节/内容/数据/对比/流程/引言/结尾）、叙事目标、Bento Grid 卡片布局、视觉描述。输出 `outline.md`。

**P7 提示词工程** — 将大纲中的每一页转化为一条自包含的 HTML 生成指令。每条指令包含完整的 `STYLE_INSTRUCTIONS`（从 design-spec 提取的 CSS 变量：颜色、字体、间距）加上该页的具体内容。这个阶段执行**内容深度检查**：禁止抽象标签（如"展示关键指标"、"显示流程的流程图"），要求具体数据（如"三列数字卡片：左列 #2196F3 蓝色、值 2,380 万、标签 Q1 营收"）。检查不通过则重新生成（最多重试 3 次）。自包含验证：检查每条指令是否包含完整的风格参数、具体内容、完整数据、具体的视觉描述、无外部依赖、页面元数据。最后运行脚本生成 `html-generation-manifest.json`，汇总所有指令。

**P9 AI 配图** — 从 design-spec 的图片资源清单中提取状态为"待生成"的条目，按类型生成优化的图片提示词：**背景图**（抽象渐变、低对比度、预留文字区域）、**摄影图**（真实场景、专业灯光）、**插画**（扁平矢量、线条干净）、**图示**（流程图/架构图风格）、**装饰纹理**（细微质感）。通过 AI 图片生成工具（优先 Gemini API，备选 DALL-E）按 design-spec 的配色方案和比例生成图片，保存到 `images/` 目录。这个阶段是条件执行的：如果策略规划时选了"无图片"、"用户提供"或"占位符"，自动跳过。

**P11 HTML 执行** — 逐页生成 HTML 幻灯片（不批量，一页一页来）。每页的生成过程：读取该页的提示词指令 → 构建 HTML5 结构 + 内联 CSS + CSS 变量 + 内容（内联 SVG）+ 图片引用 → 保存到 `html_output/` 目录 → 确认无错误后继续下一页。HTML 的三种渲染模式：**HTML+CSS**（默认，文字和布局用 CSS 控制）、**内联 SVG**（图表和示意图直接画在 SVG 里）、**AI 图片**（通过 `<img>` 标签引用）。每页强制高信息密度（至少 3 个信息块，画布利用率 >70%），禁止使用 filter/mask/clipPath/foreignObject/animate/script 等 SVG 元素。全部生成后运行后处理管道：`validate`（校验 HTML 格式、viewport、CSS 变量、无禁止元素）→ `screenshot`（无头浏览器截图到 PNG）→ `finalize`（确认所有输出齐全）。同时为每页生成演讲备注（标题 + 2-5 句话的讲稿 + 3 个要点 + 预估时长 + 页间过渡语），保存到 `notes/total.md`。

**P13 导出交付** — 分三步输出。**PPTX 导出**（核心）：通过 Playwright 渲染每页 HTML，用 JS `getBoundingClientRect()` 提取元素坐标和尺寸，映射 HTML 元素为 PPTX 原生元素（`<h1>` → TextBox + 大号粗体、`.card` → Rectangle + TextBox、`<table>` → 原生 Table、内联 SVG → 截图后嵌入为 Image、`<img>` → Image 嵌入），坐标转换公式为 `px * 9525`（96 DPI）映射到 EMU 单位。最终输出可编辑的 PPTX，你拿到后可以自由修改文字、调整布局、更换颜色。**演讲备注拆分**：将 `total.md` 拆分到每页对应的备注中。**PDF 导出**：通过 Playwright 将每页 HTML 渲染为单页 PDF，再合并为完整文件。

**简化路径：`ideal-ppt-generator`**

如果你不需要完整 13 阶段流程，`ideal-ppt-generator` 提供了一条简化路径：直接提供内容 → 确认风格和页数 → 生成幻灯片图片 → 合成 PPTX。它用 AI 直接生成每页幻灯片的图片（而非 HTML），适合对定制化要求不高、快速出图的场景。

### 2.3 ideal-dev-workflow — 15 阶段开发流程

**做什么：** 从需求澄清到代码提交的完整开发流程。覆盖需求分析、技术方案设计、编码计划、测试用例、编码执行、代码评审、测试执行、维基更新、成果提交。每个阶段都有对应的专用 Skill，按 TDD 原则组织。

**适用场景：** 功能开发、Bug 修复、代码重构等软件开发任务。

**设计理念：** 这个工作流的核心哲学是**"先想清楚再动手，验证了再往前走"**。软件开发中最大的浪费不是写代码的时间，而是返工的时间——需求没想清楚就写代码，写完发现方向错了；没有测试用例就编码，测的时候发现逻辑不对；代码写完不做评审，合入之后给别人捅出 bug。每一步的额外投入，都是在为后续步骤节省几倍的时间。

具体来说有三条贯穿始终的设计原则：

**第一，TDD（测试驱动开发）。** 先写测试用例（P7），再编码实现（P9）。测试用例先于代码存在，意味着每写完一段代码都有现成的验证标准。

**第二，编码计划优于直接编码。** 把技术方案拆解为带依赖关系的原子任务（P5），按拓扑顺序执行（P9），而不是想到哪写到哪。

**第三，两层评审。** 代码评审分两个阶段：先检查"是否按计划构建"（功能正确性），再检查"代码质量"（风格、性能、安全性）。分开评审比混在一起评审效率更高，因为评审者的注意力不会被分散。

**包含 15 个 Skill：**

| Skill | 做什么 | 能否独立使用 |
|-------|--------|-------------|
| `ideal-flow-control` | 编排器：管理 15 阶段的状态转换，支持手动/YOLO 模式 | 否（编排器） |
| `ideal-yolo` | 自动化多视角评审：架构师 + 开发者 + QA + 修改者四角色 | 否（由编排器调用） |
| `ideal-init` | 项目初始化：检测语言/框架/工作类型，生成项目配置 | 能 |
| `ideal-requirement` | 苏格拉底式需求收集：识别任务类型，生成需求文档 | 能 |
| `ideal-dev-solution` | 技术方案生成：6 个子智能体并行分析架构、技术选型、接口设计等 | 能 |
| `ideal-dev-plan` | 编码计划：将方案拆解为原子任务，按依赖关系排列执行顺序 | 能 |
| `ideal-test-case` | 测试用例生成：提取验收标准，生成功能/边界/异常测试 | 能 |
| `ideal-dev-exec` | 编码执行：按拓扑顺序实现每个任务，每步做代码自审 | 能 |
| `ideal-code-review` | 两阶段代码评审：先检查是否按计划构建，再检查代码质量 | 能 |
| `ideal-test-exec` | 测试执行：运行测试用例，生成测试报告，失败时自动调试 | 能 |
| `ideal-debugging` | 系统化调试：根因分析 → 模式识别 → 假设验证 → TDD 修复 | 能 |
| `ideal-wiki` | 维基更新：生成项目文档，包含架构图和使用说明 | 能 |
| `ideal-delivery` | 成果提交：创建 PR、清理工作目录、归档 | 能 |
| `ideal-deep-research` | 内嵌调研工具：用于技术方案阶段的技术调研 | 能 |

**完整流程（15 阶段）：**

**P0 项目初始化** — 检测项目环境，分三步：读取检测规则库，自动识别项目名称、工作类型（软件开发/文档写作/演示/论文/专利/调研报告/项目建议书/投标等）和技术栈（如果是软件开发）。如果项目已有 `.claude/project-config.md`，询问是重新配置还是保留；如果没有，生成配置文件。同时处理 CLAUDE.md：已有则追加项目配置段，没有则从模板生成。输出 `project-config.md`。

**P1 需求编写** — 先读取项目上下文（CLAUDE.md 或 README.md）——这是铁律：**不了解项目上下文之前不允许问任何需求问题**。然后询问是个人迭代还是项目迭代（目录结构不同）。接着做 **worktree 检查**：如果当前不在 worktree 中，自动通过 `git worktree add` 创建隔离的开发环境并切换进去；子迭代则继承父 worktree，不独立创建。接下来进入苏格拉底式对话：由 `pm` 子智能体逐个提问，每次只问一个问题、附 2-4 个选项、逐步深入细节、聚焦用户价值。你可以用 Skip（跳过）、Generate Directly（跳过剩余问题直接生成）、Go Back To XX（回到之前的问题）、Cancel（取消）来控制节奏。对话完成后识别任务类型（新功能/Bug 修复/重构），判断是否需要拆分为子迭代（标准：涉及多个独立模块 / 超过 2 人天 / 有明确依赖关系 / 跨多个技术栈），生成 `P1-需求文档.md`（正式学术风格）和 `流程状态.md`。

**P3 技术方案** — 所有工作由子智能体完成，主智能体不直接读取文件或写文档。第一步：spawn `context-loader` 子智能体读取 P1 需求文档、CLAUDE.md、项目配置和已有调研报告。第二步：调用 `ideal-deep-research` 做技术调研（竞品实现方案、技术趋势、可复用的开源项目、已知坑点），输出到 `research/P3-技术调研_{timestamp}.md`。第三步：**6 个子智能体并行工作**——`agent-requirements`（功能分解、依赖梳理、优先级排序）、`agent-architecture`（系统架构、组件划分、Mermaid 架构图）、`agent-tech-selection`（候选方案打分，维度权重为功能 30% / 团队熟悉度 25% / 社区活跃度 15% / 性能 15% / 学习成本 10% / 维护成本 5%）、`agent-data-model`（核心数据模型、ER 图、数据流向）、`agent-interface-design`（内外部接口规格）、`agent-risk-analysis`（风险矩阵 + 缓解策略）。第四步：spawn `doc-writer` 子智能体整合所有结果，生成 `P3-技术方案.md`（8 节：概述、架构、功能模块、数据模型、接口设计、风险分析、实施计划、参考资料）。

**P5 编码计划** — 读取 P3 技术方案和 P1 需求文档。第一步：spawn `architect` 子智能体做模块分解，标注依赖关系，生成依赖图。第二步：spawn `pm` 子智能体将模块拆解为原子任务（粒度 2-5 分钟），每个任务包含验收标准。任务按 TDD 步骤组织（RED-GREEN-REFACTOR），禁止循环依赖。第三步：生成编码计划，每个任务格式为 T001/T002...，包含功能描述、涉及文件、前置条件、变更内容、验证方式、依赖关系、时间估算。第四步：spawn `architect` 生成上下文隔离的 Story 文件（`stories/0XX-*.md`，每个包含上下文片段、任务列表、验收标准，不引用完整文档），以及 Story 索引（`stories/index.md`，包含任务列表 + 状态 + Mermaid 依赖图 + 执行顺序）。

**P7 测试用例** — 读取 P1 需求文档（验收标准、功能需求）和 P5 编码计划（任务列表、功能映射）。spawn `qa` 子智能体分四轮工作：**第一轮**提取验收标准，映射为可测试用例；**第二轮**生成功能测试用例（每个功能至少一个正向用例，覆盖主流程）；**第三轮**生成边界测试用例（输入边界、状态边界、流程边界）；**第四轮**生成异常测试用例（错误输入、异常状态、外部错误）。如果 P5 包含前端任务，还会生成**前端截图测试用例**（通过 Chrome DevTools MCP 驱动浏览器做截图，覆盖表单组件的空/填写/错误/成功态、数据展示的空/加载/正常/错误态、交互组件的初始/交互后/动画态、页面全视口）。测试用例编号格式为 TC-{类型}-{ID}（类型 F=功能/B=边界/E=异常/S=截图），覆盖比例目标为 F 50-60% / B 20-30% / E 15-20%。输出 `P7-测试用例.md`。

**P9 开发执行** — 读取 `stories/index.md`，按拓扑排序确定执行顺序。首先验证环境：确认当前目录匹配 worktree 路径，运行测试套件确认基线全部通过。然后按依赖层级逐步执行：无依赖的任务（第一层）可以并行执行，有依赖的任务等前置任务完成后执行。每个任务的执行循环：spawn `implement` 子智能体按 TDD 协议实现代码 → spawn `check` 子智能体做两阶段代码评审（**第一阶段检查规范符合性**：所有计划功能都实现了？没有多余功能？只修改了计划内的文件？测试覆盖了验收标准？接口签名匹配方案？→ **第二阶段检查代码质量**：命名、格式、可读性、函数长度、性能、安全性、可维护性。问题分三级：Critical 必须立即修复阻塞后续、Important 应在继续前修复、Minor 记录后续处理）。评审发现问题则 spawn `debug` 子智能体自动修复后重新评审。全部任务完成后运行完整测试套件 + 构建验证。

**P11 测试执行** — 读取 P7 测试用例，按优先级执行（P0 → P1 → P2）。spawn `qa` 子智能体执行测试并记录结果。前端截图测试通过 Chrome DevTools MCP 直接控制浏览器：导航 → 等待渲染 → 交互 → 截图 → 保存（验证截图存在且 >10KB）。发现缺陷时，按严重程度分类（致命：系统崩溃/数据丢失；严重：核心功能不可用；一般：功能受限/体验问题；轻微：UI 问题/排版错误），spawn `debug` 子智能体做根因分析并修复，修复后重新验证。输出测试报告（总/通过/失败/跳过数、每条用例结果、缺陷记录、结论）。

**P13 维基更新** — 分析项目代码结构、配置文件、已有文档和 P1 需求文档。spawn `tech-writer` 子智能体生成维基大纲 → **暂停等用户确认大纲** → 确认后生成内容 → **第一篇文章完成后暂停，让用户确认写作风格**。内容写作完成后，spawn `dev` 子智能体做一致性审查（文档是否与实际代码匹配），spawn `qa` 子智能体做角色模拟审查（分别模拟新用户和开发者视角）。所有图表使用 Mermaid（不用 ASCII），要求正式技术风格，流程描述必须完整（前置条件 → 步骤 → 预期 → 异常），代码示例必须可执行。

**P15 成果提交** — 首先读取流程状态，检测是否为子迭代（通过 `parent` 字段区分）。**子迭代**的处理：在 worktree 中提交代码到共享分支，然后直接跳到本地归档（不创建 PR、不删除 worktree 和特性分支），同时更新父迭代的子迭代状态。**父迭代**的处理：切换到主仓库（非 worktree），提交所有变更，推送并通过 `gh pr create` 创建 PR → **暂停等用户合并 PR** → 合并后清理：checkout main → pull → 删除特性分支 → 删除 worktree。最后做本地归档：将流程状态更新为 completed，将目录从 `[进行中]` 重命名为 `[已完成]`。

### 2.4 ideal-knowledge-base — 知识库构建

**做什么：** 从现有项目材料（代码、文档、配置文件等）出发，自动分析内容、规划文档结构、并行生成知识库文档、汇总评审。适合将零散的项目资料系统化为结构化的知识库。

**适用场景：** 项目知识沉淀、团队 Wiki 构建、技术积累整理。

**设计理念：** 知识库构建的核心挑战不是"写不出来"，而是"不知道该写什么、写到什么深度"。一个项目有成百上千个文件，哪些信息值得沉淀？以什么结构组织？写到什么粒度？ideal-knowledge-base 的设计思路是**"先系统性扫描，再结构化提取，最后按标准模板生成"**。

分三个大阶段：第一阶段串行处理（材料收集 → 深度分析 → 文档规划），先对原材料做全面扫描和深度分析，确定知识库的范围和结构；第二阶段并行生成，为每份文档调用 `ideal-document-workflow` 走完整的写作流程，确保每份知识库文档都达到独立的文档质量标准；第三阶段汇总评审，做 5 维度质量审查。

核心洞察：**知识库文档不应该比普通文档的质量低**。很多知识库的问题是"有但不好用"——文档是生成了，但内容浅、结构乱、没人愿意看。通过复用 `ideal-document-workflow` 的完整写作质量保障机制，确保每份知识库文档都经得起阅读。

**包含 6 个 Skill：**

| Skill | 做什么 | 能否独立使用 |
|-------|--------|-------------|
| `build-knowledge-base-workflow` | 编排器：协调材料分析 → 文档规划 → 并行生成 → 汇总评审 | 否（编排器） |
| `kb-material-scope` | 扫描项目原材料，识别文件类型，评估覆盖维度 | 能 |
| `kb-deep-analysis` | 从材料中深度提取知识（7 个维度：项目定位、技术架构、功能模块等） | 能 |
| `kb-doc-planning` | 规划知识库文档结构（4 份核心文档 + 可选扩展文档） | 能 |
| `kb-doc-writing` | 已废弃，现由 `ideal-document-workflow` 替代完成文档生成 | — |
| `kb-index-review` | 更新索引 + 5 维度质量审查（完整性、准确性、深度、可读性、可维护性） | 能 |

**完整流程（3 个大阶段）：**

**第一阶段（串行）：材料收集 → 深度分析 → 文档规划**

材料收集（`kb-material-scope`）扫描你提供的项目材料（本地目录或 GitHub 仓库），识别文件类型（README、源码、文档、配置、容器配置、测试、CI/CD、数据文件等），对 6 个覆盖维度逐一评估（充足/部分/缺失）：项目定位、技术架构、功能实现、部署运维、应用成效、领域专项维度（按项目类型不同：数据治理 / 模型训练 / 知识管理 / 智能体）。如果是 GitHub 仓库还会提取语言统计、近期活跃度、Stars/Forks、Topics。最后识别知识缺口，按优先级排序（高/中/低），输出 `P1-材料清单.md`。

深度分析（`kb-deep-analysis`）分 7 个阶段逐层深入：**① 项目定位分析**（从 README/文档/首页提取产品名、一句话定位、目标客户、核心痛点 2-4 个、核心价值主张）；**② 技术架构分析**（从目录结构、配置文件、入口文件识别架构模式——单体/微服务/流水线/智能体——和技术栈、模块划分、部署方式、外部依赖）；**③ 功能模块分析**（从源码目录结构、入口文件、路由、API 定义提取模块层级、每模块核心功能、模块间依赖、实现状态）；**④ 技术细节分析**（从核心源码、算法实现深入——算法分析要求 300+ 字解释原理、设计思路、实现步骤、核心代码和参数，禁止只贴代码不解释）；**⑤ 领域专项分析**（按项目类型不同做针对性分析）；**⑥ 部署运维分析**（从 Dockerfile、docker-compose、部署脚本提取部署方案、环境依赖、配置项、监控日志）；**⑦ 应用成效分析**（从 CHANGELOG、Issue、PR 描述提取应用案例、量化指标、已知问题、版本历史）。项目较大时（50+ 核心文件），按模块 spawn 并行子智能体加速处理。关键原则：读取源码入口和核心文件而不只是 README，`pyproject.toml`/`package.json`/`requirements.txt` 是理解技术栈的捷径，测试文件往往包含使用示例和边界条件。输出 `P2-分析报告.md`。

文档规划（`kb-doc-planning`）基于分析结果确定知识库文档结构。**4 份核心文档**（必选）：项目概述、技术架构、功能清单、部署与运维。**3 份领域专项文档**（按项目类别自动选择，如智能体类 = 智能体设计 + 工具集成 + 应用成效）。**可选扩展文档**（如核心算法详解——P2 分析显示算法复杂度高时推荐）。你可以追加自定义文档。每份文档规划了内容要点（从 P2 分析提取）、信息来源标注、充分度评级、配图需求（Mermaid 图表类型）。文档之间的依赖关系也明确了：概述 → 技术架构 → 功能清单；概述 → 领域专项文档；部署运维（独立）。包含一段机器可读的 YAML 块供编排器解析。输出 `P3-文档规划.md`。

**第二阶段（并行）：为每份文档调用 `ideal-document-workflow`**

每一份知识库文档都走完整的写作流程（需求分析 → 大纲 → 任务拆分 → 并行写作 → 配图 → 渲染），多份文档并行生成。复用 `ideal-document-workflow` 的完整质量保障机制，确保知识库文档的质量不打折扣。

**第三阶段：合并索引 + 质量审查**

`kb-index-review` 分两部分。**索引更新**：更新分类索引（按日期倒序，每条包含日期、记录人、来源路径、知识库位置、内容摘要、关键词、关联产出清单）和主索引（跨类别，多类别项目做交叉引用）。**5 维度质量审查**：完整性（所有规划文档都生成了？模板章节齐全？必填字段填写了？无空章节？需要配图的地方有图？）、准确性（信息可追溯到源文件？技术描述与实际代码一致？版本信息正确？跨文档信息一致？无未经证实的主张？）、深度（架构分层描述？算法有原理+输入输出+复杂度分析？数据流覆盖完整链路？性能数据有具体数值？边界条件覆盖？）、可读性（结构清晰？术语一致？Mermaid 图表渲染正确？列表数据用表格？语言专业？）、可维护性（有来源标注？有版本关联？易于更新？文档边界模块化？）。严重问题必须修复后才能交付。

## 3. 独立工具：三个专项能力

独立工具不需要编排器，每个工具就是一个完整的 Skill，直接调用即可。

### 3.1 ideal-deep-research — 企业级深度调研

**做什么：** 从一个调研问题出发，自动搜索多源信息、交叉验证、引用追踪、可信度评分，生成一份带完整引用的调研报告。

**设计理念：** 调研的核心挑战不是"找不到信息"，而是"信息太多，真假难辨，难以形成结论"。ideal-deep-research 的设计思路是**"多源交叉验证"**——每条结论至少需要 3 条来自不同信息源的引用支撑，并且对每个信息源做可信度评分。

**4 种深度模式：**

| 模式 | 耗时 | 适用场景 |
|------|------|---------|
| `quick` | 2-5 分钟 | 快速扫描，了解概况 |
| `standard` | 5-10 分钟 | 标准调研，日常使用（默认） |
| `deep` | 10-20 分钟 | 深度分析，技术选型 |
| `ultradeep` | 20-45 分钟 | 全面审查，竞品深度分析 |

**调研管道（8 个阶段）：** 确定范围 → 制定计划 → 检索信息 → 交叉验证（用多源比对筛掉不可靠信息）→ 生成大纲 → 综合分析 → 批判性审视（用反方视角检查是否在信息茧房里）→ 精炼输出。要求覆盖 10+ 信息源，每条结论至少 3 条引用，对信息源做可信度评分。

**调用方式：** `/ideal-deep-research:deep-research`

---

### 3.2 baoyu-image-gen — AI 图片生成

**做什么：** 通过 AI 生成图片，支持三个 Provider：OpenAI (DALL-E)、Google (Imagen)、DashScope (通义万相)。支持文本生成图片、参考图片生成、宽高比设置、质量预设、并行生成。

**设计理念：** 不同 Provider 各有优势——DALL-E 擅长创意类图片，Imagen 在写实场景表现好，通义万相对中文 prompt 理解更准确。支持三个 Provider 让你根据具体需求选择最合适的工具。

**适用场景：** 文档配图、PPT 插图、营销素材等需要 AI 生成图片的场景。

**调用方式：** `/baoyu-image-gen:baoyu-image-gen`

---

### 3.3 ideal-dify-generator — Dify 工作流生成

**做什么：** 从自然语言描述的需求出发，自动生成符合 Dify DSL 规范的 YAML 工作流配置文件，并做语法校验和逻辑检查。

**设计理念：** Dify 工作流的 YAML 配置语法虽然不复杂，但手写容易出错。ideal-dify-generator 的设计思路是"你只需要描述业务逻辑，DSL 语法由工具处理"。内置 7 种场景模板（简单 LLM 对话、错误处理、条件分支、OCR 处理、数据查询、智能体编排、迭代循环）。

**适用场景：** 需要快速搭建 Dify 工作流或智能体的场景。

**调用方式：** `/ideal-dify-generator:ideal-dify-generator`

## 4. 开发辅助：两个效率工具

### 4.1 ideal-ralph — 苏格拉底式任务澄清

**做什么：** 通过对话引导你明确任务的输入、输出、验收标准和实施方式，生成"任务合约"，然后以持久迭代循环逐步执行，每一步都验证是否满足验收标准。核心理念是"先想清楚再动手"。

**设计理念：** Ralph 解决的是一个被严重低估的问题：**很多人不是做得不好，而是没想清楚要做什么**。Ralph 的设计是苏格拉底式引导——不是替你想，而是通过提问帮你把模糊的想法变成明确的合约。

合约（contract）包含四个维度：输入（你有什么素材/前提条件）、输出（最终交付物长什么样）、验证（怎么判断做完了）、实施（用什么方法做）。合约一旦确认，Ralph 进入 LOOP 阶段，持久迭代执行，每步完成后对照验收标准验证。不通过就继续改，通过才进入下一步。

Ralph 有 6 条铁律：永不停止（除非所有验收标准通过）、永不缩减范围（做不完可以上报，不能偷偷砍需求）、永不空洞承诺（说做到就必须做到）、小步前进（每步只做一件事）、上报障碍（遇到解决不了的问题及时上报，而不是死磕）、永不代理（自己做，不把任务转包给其他智能体然后假装完成了）。

**执行分两阶段：**
1. **CLARIFY（澄清）**：在输入、输出、验证、实施四个维度上通过对话明确任务，生成合约
2. **LOOP（迭代执行）**：持久循环，每步完成后验证验收标准，直到全部通过

**适用场景：** 任何需要先想清楚再动手的任务。Ralph 不局限于开发，也可以用于文档写作、调研等场景。

**调用方式：** `/ideal-ralph:ideal-ralph`

## 5. 如何开始使用

### 5.1 安装（两步）

**第一步：添加 Marketplace 源**（只需一次）

```bash
claude plugin marketplace add https://github.com/MTleen/ideal-lab
```

**第二步：按需安装插件**

```bash
# 文档写作（推荐入门）
claude plugin install ideal-document-workflow@ideal-lab

# PPT 生成
claude plugin install ideal-ppt-suite@ideal-lab

# 深度调研
claude plugin install ideal-deep-research@ideal-lab

# 开发流程
claude plugin install ideal-dev-workflow@ideal-lab

# 知识库构建
claude plugin install ideal-knowledge-base@ideal-lab
```

每个插件独立安装，用哪个装哪个。

### 5.2 第一次使用

安装完成后，Skill 自动加载，直接在对话中通过斜杠命令调用。几个快速上手的例子：

**写一份文档：**
```
/ideal-document-workflow:ideal-document-workflow 帮我写一份"团队代码评审规范"，面向开发团队，1500 字左右
```

**做一次调研：**
```
/ideal-deep-research:deep-research 调研 2025 年主流 LLM 应用开发框架对比，standard 模式
```

**生成一份 PPT：**
```
/ideal-ppt-suite:ideal-ppt-workflow 生成一份"项目季度总结"PPT，10 页，科技蓝风格
```

**只使用某个阶段的 Skill（以文档大纲为例）：**
```
/ideal-document-workflow:outline-generation 帮我生成一份技术方案的大纲
```

### 5.3 插件组合使用

ideal-lab 的 Skill 可以自由组合。一个典型的工作流是先调研再输出：

```
# 第一步：深度调研
/ideal-deep-research:deep-research 调研 XX 主题

# 第二步：基于调研结果生成 PPT
/ideal-ppt-suite:ideal-ppt-workflow 基于上面的调研结果，生成汇报 PPT
```

### 5.4 按角色推荐

| 角色 | 推荐起步 | 推荐插件 |
|------|---------|---------|
| 开发工程师 | 用 ralph 澄清一个任务的验收标准 | ideal-ralph → ideal-dev-workflow |
| 产品经理 | 用 deep-research 做一次竞品调研 | ideal-deep-research → ideal-ppt-suite |
| 技术写作 | 用 document-workflow 写一份短文档 | ideal-document-workflow → ideal-knowledge-base |
| 项目经理 | 用 deep-research 做项目前期调研 | ideal-deep-research → ideal-ppt-suite → ideal-document-workflow |

## 6. 下一步

### 6.1 推荐起步路径

**开发工程师：**
1. 安装 `ideal-ralph`，用它澄清一个你手头任务的验收标准——体验"先想清楚再动手"
2. 安装 `ideal-dev-workflow`，选一个小功能任务走一遍完整流程
3. 体验 `ideal-yolo` 的多视角自动评审，对比人工评审的差异

**产品/管理：**
1. 安装 `ideal-deep-research`，选一个你关心的行业话题做一次 standard 模式调研
2. 安装 `ideal-ppt-suite`，基于调研结果生成一份汇报 PPT
3. 尝试 YOLO 模式，体验从调研到 PPT 的全自动流程

**技术写作：**
1. 安装 `ideal-document-workflow`，选一个你正在写的文档走一遍完整流程
2. 重点关注评审环节——读者、作者、专家三个视角分别给出了什么反馈
3. 安装 `ideal-knowledge-base`，尝试从现有项目材料中构建知识库

### 6.2 反馈与共建

ideal-lab 是持续迭代的活系统。使用过程中如果你发现问题或有改进建议：

- **问题反馈**：通过 GitHub Issue 描述你遇到的问题
- **Skill 共建**：如果你在日常工作中沉淀出了一套好用的流程，可以按照 ideal-lab 的 Plugin 结构封装为新的 Skill，通过 PR 提交
- **经验分享**：使用 ideal-lab 产出的文档、PPT、调研报告，本身就是最好的经验沉淀

最后一句话：ideal-lab 的每一个 Skill 都是从真实项目中长出来的，不是拍脑袋设计的。你在使用中发现更好的做法，就是下一次迭代的起点。
