# P3 任务拆分计划

## 拆分规则
- 目标字数：每任务约 3000-6000 字（本文档总篇幅 6800 字，单个章节均不足 3000 字，需合并）
- 拆分策略：按逻辑单元合并一级标题，确保每个任务是完整的叙事单元
- 并行友好：任务间尽量独立，通过上下文卡片传递必要信息

## 任务列表

### 任务 1: 认知与架构 — 从痛点到工具体系
- **文件**：`sections/1-认知与架构.md`
- **预估字数**：1900 字
- **章节层级**：第 1 章 + 第 2 章（1.1 ~ 1.3 + 2.1 ~ 2.3）
- **上下文**：
  - 来源需求（P1 场景一+场景二）：让读者从痛点共鸣到建立结构化认知，理解 ideal-lab 的定位和架构
  - 上级概述：ideal-lab 是从日常工作中沉淀的 Claude Code 最佳实践插件库，9 个插件 42 个 Skill，采用 Plugin → Skill → References 三层架构
  - 同级任务标题：任务 2（document-workflow 深度解析）、任务 3（全景与上手）
- **写作要点**：
  - 语气是"同事间分享实战经验"，不是产品推销
  - 1.1 痛点要具体、真实，每个场景 2-3 句话，点到为止
  - 1.2 沉淀叙事是关键：第一次用 AI → 发现问题 → 总结经验写成 Skill → 复用 → 迭代。用时间线讲，不要用抽象概念
  - 1.3 自然过渡到当前状态，一句话定位 + 核心数字 + 核心价值
  - 2.1 三层结构要配 Mermaid 架构图（Plugin 包含 Skills，Skill 包含 SKILL.md + references/ + scripts/）
  - 2.2 分发机制给实际命令：`claude plugin marketplace add`、`claude plugin install`、斜杠命令调用
  - 2.3 版本管理侧重管理读者关心的：可追溯、可回滚、更新可控
- **配图**：三层结构示意图（Mermaid）
- **避免**：不要在第 2 章展开任何工作流细节，那是任务 2 的内容

### 任务 2: 核心案例 — ideal-document-workflow 深度解析
- **文件**：`sections/2-深度解析.md`
- **预估字数**：2200 字
- **章节层级**：第 3 章（3.1 ~ 3.6）
- **上下文**：
  - 来源需求（P1 场景三）：以 ideal-document-workflow 为案例，展示编排器模式、评审机制、YOLO 模式等核心设计理念
  - 上级概述：ideal-document-workflow 是 12 阶段结构化写作工作流，包含 6 个产物阶段、5 个评审阶段、1 个交付阶段
  - 前文摘要（任务 1 已写）：读者已知 ideal-lab 是什么、Plugin/Skill 架构是什么
  - 同级任务标题：任务 1（认知与架构）、任务 3（全景与上手）
- **写作要点**：
  - 3.1 用一个完整案例串联："帮我写一份技术方案" → 需求分析 → 大纲 → 任务拆分 → 写作 → 配图 → 渲染，每步展示输入和输出
  - 3.2 十二阶段流程图（Mermaid）：产物阶段（蓝）和评审阶段（灰）交替出现，一眼看出节奏
  - 3.3 编排器模式用通俗语言：主智能体 = 项目经理，只分配任务不亲自动手。补充"为什么这样设计"——AI 一次写完质量不可控，所以拆成小步
  - 3.4 评审团队用类比：读者评审 = 用户验收、作者评审 = 自检、专家评审 = 同行评审、修改者 = 整合修改的人。熔断 = 连续 3 轮不通过就停下来等老板拍板
  - 3.5 两种模式对比用手动挡 vs 自动挡的比喻
  - 3.6 What-Why-How 模式要举一个简短示例（2-3 句话），让读者直观理解
  - 所有内部术语（编排器、sub-agent、熔断）首次出现时用括号注释通俗解释
- **配图**：12 阶段流程图（Mermaid）、评审循环流程图（Mermaid）
- **避免**：不要展开其他 8 个插件的细节，那是任务 3 的内容
