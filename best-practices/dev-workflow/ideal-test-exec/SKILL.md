# ideal-test-exec（P11 测试执行）

## 角色定位

Phase Skill — **执行协调者**。

职责：
1. 加载所需上下文（P7 测试用例 + 项目代码）
2. 调度子智能体执行测试
3. 将报告写入文件系统
4. 返回执行摘要

**不负责**：更新 flow state、验证前置条件，协调评审。

---

## 输入

| 来源 | 内容 |
|------|------|
| P7-测试用例.md | 功能/边界/异常/截图用例 |
| 项目代码 | 待测试的代码 |

## 输出

| 文件 | 路径 |
|------|------|
| 测试报告 | `docs/迭代/{需求名称}/P11-测试报告.md` |

---

## 子智能体调度

| 调用时机 | 子智能体 | 任务 |
|----------|----------|------|
| 测试执行 | `qa` | 按优先级执行用例，记录结果 |
| 缺陷记录 | `qa` | 分类严重程度，关联相关用例 |
| 调试修复 | `debug` | 测试失败时的根因分析和修复 |

---

## 前端截图测试（必须）

> **强制要求**：当 P7 测试用例中包含前端截图用例时，必须通过 Chrome DevTools MCP 工具驱动浏览器执行截图，不得仅依赖 TypeScript 编译检查或手工截图。

### 截图方式

**Chrome DevTools MCP**：Agent 直接通过 MCP 工具控制浏览器，执行导航、交互和截图操作。无需编写任何测试脚本，零额外依赖。

### 截图保存规范

| 项目 | 规范 |
|------|------|
| 截图保存目录 | `docs/迭代/{需求名称}/screenshots/` |
| 截图命名规则 | `{用例编号}_{截图名称}.png` |
| 视口大小 | 自动使用当前屏幕分辨率（无需配置） |
| 截图方式 | Agent 通过 MCP 工具操控浏览器，全页面截图或指定元素截图 |
| 工具 | Chrome DevTools MCP（Agent 直接调用，零代码） |

### 截图执行流程

```
1. 确认 P7 中存在前端截图用例（TC-S-* 类型）
2. 创建截图保存目录：docs/迭代/{需求名称}/screenshots/
3. Agent 通过 Chrome DevTools MCP 工具操控浏览器：
   - 导航到目标页面
   - 等待页面渲染完成
   - 执行交互操作（点击、输入等）
   - 调用截图功能，保存到指定路径
4. 验证截图文件已保存到 screenshots/ 目录（文件存在 + 大小 > 10KB）
5. 在测试报告中引用截图路径
```

### 截图验证规则

| 检查项 | 说明 |
|--------|------|
| 文件存在 | 每个 TC-S-{id} 用例必须有对应的截图文件 |
| 文件大小 | 截图文件 > 10KB（小于则可能为空白页） |
| 命名规范 | 符合 `{TC编号}_{名称}.png` 格式 |
| 视觉完整性 | 页面无空白区域、无渲染错误（人工抽检） |

## 测试结果状态

| 状态 | 说明 |
|------|------|
| PASS | 测试通过 |
| FAIL | 测试失败 |
| SKIP | 跳过测试 |
| BLOCK | 被阻塞无法执行 |

## 缺陷严重程度

| 级别 | 说明 |
|------|------|
| 致命 | 系统崩溃、数据丢失 |
| 严重 | 核心功能不可用 |
| 一般 | 功能受限、体验问题 |
| 轻微 | 界面问题、文字错误 |

---

## 执行流程

```
Step 1: 读取 P7-测试用例.md
Step 2: 准备测试环境
Step 3: 按优先级执行用例
  └─ Task(qa) → 记录执行结果，截图/日志留存
Step 3.5: 前端截图测试（仅当 P7 包含截图用例时）
  ├─ 创建截图目录 docs/迭代/{需求名称}/screenshots/
  ├─ Agent 通过 Chrome DevTools MCP 操控浏览器
  ├─ 逐个执行截图用例：导航 → 交互 → 截图 → 保存
  ├─ 验证截图文件已保存（文件存在 + 大小 > 10KB）
  └─ 将截图路径写入测试报告
Step 4: 发现缺陷时
  └─ Task(debug) → 根因分析，修复后重新验证
Step 5: 生成测试报告
Step 6: 更新流程状态
```

详见 `references/test-exec-flow.md`。
