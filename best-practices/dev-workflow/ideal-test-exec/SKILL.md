# ideal-test-exec（P11 测试执行）

## 角色定位

Phase Skill — **执行协调者**。

职责：
1. 加载所需上下文（P7 测试用例 + 项目代码）
2. 调度子智能体执行测试
3. 将报告写入文件系统
4. 返回执行摘要

**不负责**：更新 flow state、验证前置条件，协调评审。

---

## 输入

| 来源 | 内容 |
|------|------|
| P7-测试用例.md | 功能/边界/异常用例 |
| 项目代码 | 待测试的代码 |

## 输出

| 文件 | 路径 |
|------|------|
| 测试报告 | `docs/迭代/{需求名称}/P11-测试报告.md` |

---

## 子智能体调度

| 调用时机 | 子智能体 | 任务 |
|----------|----------|------|
| 测试执行 | `qa` | 按优先级执行用例，记录结果 |
| 缺陷记录 | `qa` | 分类严重程度，关联相关用例 |
| 调试修复 | `debug` | 测试失败时的根因分析和修复 |

---

## 测试结果状态

| 状态 | 说明 |
|------|------|
| PASS | 测试通过 |
| FAIL | 测试失败 |
| SKIP | 跳过测试 |
| BLOCK | 被阻塞无法执行 |

## 缺陷严重程度

| 级别 | 说明 |
|------|------|
| 致命 | 系统崩溃、数据丢失 |
| 严重 | 核心功能不可用 |
| 一般 | 功能受限、体验问题 |
| 轻微 | 界面问题、文字错误 |

---

## 执行流程

```
Step 1: 读取 P7-测试用例.md
Step 2: 准备测试环境
Step 3: 按优先级执行用例
  └─ Task(qa) → 记录执行结果，截图/日志留存
Step 4: 发现缺陷时
  └─ Task(debug) → 根因分析，修复后重新验证
Step 5: 生成测试报告
Step 6: 更新流程状态
```

详见 `references/test-exec-flow.md`。

```markdown
| 缺陷编号 | 严重程度 | 描述 | 状态 |

{结论和建议}
```
