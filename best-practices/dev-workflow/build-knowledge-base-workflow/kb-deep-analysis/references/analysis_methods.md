# 分析方法论参考（Analysis Methods Reference）

本文件提供 P1-P2 阶段的分析方法论，指导如何高效、深入地分析不同类型的原始材料。

---

## 一、不同文件类型的分析策略

### 1.1 README 文件

**分析重点：** 项目概览、快速定位、核心概念

| 分析动作 | 具体方法 |
|----------|----------|
| 项目定位 | 从 README 开头 1-3 段提取项目的一句话定位 |
| 快速安装 | 从安装/快速开始部分了解项目的技术栈和运行方式 |
| 功能概览 | 从 Features/功能列表部分建立初步功能模块认知 |
| 架构线索 | 寻找 Architecture/架构相关章节或链接 |
| 依赖分析 | 从 Requirements/Dependencies 了解技术选型和依赖关系 |
| 关键入口 | 从 Usage/使用示例中找到核心 API 或入口文件 |

**提示：** README 往往信息密度不均匀，开头部分最有价值。注意区分"项目描述"和"使用教程"，前者对知识库更有价值。

### 1.2 源代码文件

**分析重点：** 实现细节、架构逻辑、算法实现

| 分析动作 | 具体方法 |
|----------|----------|
| 入口定位 | 找到 `main.py`、`app.py`、`index.js`、`cmd/` 等入口文件 |
| 模块划分 | 通过目录结构理解模块划分（每个子目录通常对应一个功能模块） |
| 核心逻辑 | 重点分析业务逻辑层（`service/`、`core/`、`handler/`、`logic/` 等） |
| 数据模型 | 分析 `model/`、`schema/`、`types/`、`dto/` 等数据定义文件 |
| 接口定义 | 分析 API 路由文件、接口定义文件（`api/`、`routes/`、`controller/` 等） |
| 工具函数 | 浏览 `utils/`、`helpers/`、`common/` 了解通用能力 |

**提示：** 不需要逐行阅读所有代码。优先看入口文件、核心模块的公开接口、数据模型定义，建立整体认知后再按需深入。

### 1.3 配置文件

**分析重点：** 技术栈、部署架构、运行参数

| 文件类型 | 分析重点 |
|----------|----------|
| `requirements.txt` / `pyproject.toml` / `package.json` | Python/Node.js 依赖列表，了解技术栈和版本 |
| `Dockerfile` / `docker-compose.yml` | 容器化方案、基础镜像、暴露端口、服务编排 |
| `config.yaml` / `config.json` / `.env` | 运行时配置、环境变量、可调参数 |
| `Makefile` / `Taskfile` / `justfile` | 构建、测试、部署的快捷命令 |
| `helm/` / `k8s/` | Kubernetes 部署配置，了解生产部署架构 |
| `nginx.conf` / `Caddyfile` | 反向代理、负载均衡、路由配置 |

**提示：** 配置文件是了解项目技术栈和部署方式的最快途径，建议在 P1 阶段优先阅读。

### 1.4 测试文件

**分析重点：** 行为规范、边界条件、预期输出

| 分析动作 | 具体方法 |
|----------|----------|
| 功能边界 | 从测试用例了解功能的输入输出和边界条件 |
| 异常处理 | 从错误场景测试了解系统的异常处理策略 |
| 集成方式 | 从集成测试了解系统与外部组件的交互方式 |
| 性能基线 | 从性能测试了解系统的性能目标和基准数据 |
| 数据规格 | 从测试数据了解系统处理的数据格式和规模 |

**提示：** 测试文件是"活的文档"，往往比注释更准确地反映系统行为。优先看单元测试和集成测试的 test case 描述。

### 1.5 文档文件

**分析重点：** 设计意图、技术决策、使用指南

| 文件类型 | 分析重点 |
|----------|----------|
| `docs/` 目录 | 架构设计、API 文档、使用指南、FAQ |
| `CHANGELOG.md` | 版本演进、功能变更历史、重要里程碑 |
| `CONTRIBUTING.md` | 开发流程、代码规范、架构约定 |
| `design/` 或 `design-docs/` | 设计文档、RFC、技术方案 |
| `ADR/`（Architecture Decision Records） | 关键架构决策及其原因 |
| `.github/` 或类似目录 | Issue 模板、PR 模板、讨论记录 |

**提示：** 如果项目有 ADR（架构决策记录），这是理解"为什么这样设计"的宝贵资源。

### 1.6 CI/CD 配置

**分析重点：** 自动化流程、质量保障、发布策略

| 文件类型 | 分析重点 |
|----------|----------|
| `.github/workflows/` | GitHub Actions 工作流，了解自动化测试和部署流程 |
| `.gitlab-ci.yml` | GitLab CI/CD 配置 |
| `Jenkinsfile` | Jenkins 流水线配置 |
| `.circleci/` | CircleCI 配置 |

**提示：** CI/CD 配置可以揭示项目的质量保障手段（运行哪些测试、代码检查、构建步骤）和部署流程。

### 1.7 数据文件

**分析重点：** 数据结构、数据规模、数据关系

| 文件类型 | 分析重点 |
|----------|----------|
| `data/` 或 `dataset/` | 样本数据，了解数据格式和内容 |
| `migrations/` 或 `alembic/` | 数据库迁移，了解数据模型演进 |
| `schemas/` | 数据 Schema 定义（JSON Schema、Protobuf、Avro 等） |
| `examples/` | 示例数据，了解输入输出格式 |
| `prompts/` 或 `templates/` | Prompt 模板，了解 AI 交互设计 |

## 二、分类专项分析重点

### 2.1 数据治理（Data Governance）

#### 核心分析维度

1. **数据管线追踪**
   - 数据从哪些源头进入系统？
   - 数据经过哪些处理环节？每个环节做了什么？
   - 数据最终存储在哪里？格式是什么？
   - 是否有实时处理和批量处理两条路径？

2. **质量规则分析**
   - 定义了哪些数据质量维度（准确性、完整性、一致性、时效性、唯一性等）？
   - 每个维度有哪些具体的质量规则？
   - 质量规则是如何执行的（运行时检查、定时扫描、批处理校验）？
   - 质量阈值如何设定？不合格数据的处理策略是什么？

3. **治理策略分析**
   - 数据分类分级策略是什么？
   - 数据访问控制和权限管理方案是什么？
   - 数据生命周期管理（归档、清理、保留策略）是如何实现的？
   - 数据血缘追踪能力如何？
   - 元数据管理方案是什么？

#### 关键文件关注

```
优先关注：
├── 数据 Schema 定义（model/schema/migration）
├── 数据处理 Pipeline 代码（pipeline/etl/processor）
├── 质量规则配置（rules/quality/validation）
├── 数据血缘/元数据配置（metadata/lineage/catalog）
└── 治理策略文档（docs/policy/governance）
```

### 2.2 模型训练（Model Training）

#### 核心分析维度

1. **训练流程追踪**
   - 完整的训练流程是什么？（数据准备 -> 预处理 -> 特征工程 -> 训练 -> 评估 -> 导出）
   - 每个阶段的输入输出是什么？
   - 是否支持分布式训练？方案是什么？
   - 训练过程如何监控？支持断点续训吗？

2. **模型架构分析**
   - 使用了什么模型架构？为什么选择这个架构？
   - 模型的关键超参数是什么？默认值和调优范围是什么？
   - 模型的输入输出规格是什么？
   - 模型规模（参数量、FLOPs）如何？

3. **评估指标分析**
   - 使用了哪些评估指标？为什么选择这些指标？
   - 评估数据集是什么？如何划分的？
   - 基线模型是什么？当前模型相比基线提升了多少？
   - 是否有不同场景下的专项评估？

#### 关键文件关注

```
优先关注：
├── 模型定义文件（model/network/architecture）
├── 训练脚本（train.py/training/）
├── 配置文件（config/hyperparameters/args）
├── 评估脚本（evaluate/metrics/benchmark）
├── 数据处理 Pipeline（data/preprocess/dataset）
├── 实验记录（experiments/results/logs）
└── 模型导出/部署配置（export/deploy/serving）
```

### 2.3 知识库与知识治理（Knowledge Base & Governance）

#### 核心分析维度

1. **知识结构分析**
   - 知识是如何建模的？（实体-关系、本体、分类体系、层级结构等）
   - 知识模型/Schema 是如何定义的？
   - 知识的粒度是什么？（文档级、段落级、实体级、三元组级）
   - 知识的更新机制是什么？（全量更新、增量更新、版本管理）

2. **RAG 管线分析**
   - 文档分块（Chunking）策略：分块方式、分块大小、重叠策略、元数据关联
   - 向量化方案：Embedding 模型选择、向量维度、索引类型（HNSW/IVF/Flat）
   - 检索策略：检索方式（语义检索/关键词检索/混合检索）、Top-K、过滤条件
   - 重排策略：是否使用重排模型、重排算法
   - 生成策略：Prompt 模板设计、上下文窗口管理、参数配置、幻觉控制

3. **检索质量分析**
   - 如何评估检索质量？（召回率、精确率、MRR、NDCG）
   - 是否有检索质量的监控和反馈机制？
   - 如何处理检索失败或低质量检索的情况？
   - 是否支持多轮检索和查询改写？

#### 关键文件关注

```
优先关注：
├── 知识模型定义（schema/ontology/model）
├── 文档处理 Pipeline（pipeline/ingest/chunk）
├── 向量索引配置（index/embedding/vector）
├── 检索逻辑（retrieval/search/query）
├── 生成/Prompt 配置（generation/prompt/template）
├── 评估脚本（evaluate/metrics/retrieval_eval）
└── 知识管理接口（api/crud/management）
```

### 2.4 智能体（Agents）

#### 核心分析维度

1. **Agent 架构分析**
   - Agent 架构模式是什么？（单 Agent / 多 Agent / 层级式 / 网络式）
   - 每个 Agent 的角色和职责是什么？
   - Agent 的决策流程是什么？（ReAct、Plan-and-Execute、反思式等）
   - Agent 间如何协作和通信？

2. **决策流程分析**
   - Agent 的推理链（Chain of Thought）是如何设计的？
   - Agent 如何决定调用哪个工具？
   - Agent 如何处理工具调用失败？
   - Agent 的终止条件是什么？如何判断任务完成？

3. **Prompt 策略分析**
   - 系统提示（System Prompt）的设计思路是什么？
   - Prompt 中包含了哪些关键信息？（角色定义、行为规范、输出格式、示例等）
   - Prompt 的版本管理策略是什么？
   - 是否有 Prompt 的 A/B 测试或评估机制？

4. **工具链分析**
   - Agent 可以使用哪些工具？每个工具的接口和功能是什么？
   - 工具调用的参数验证和错误处理机制是什么？
   - 是否支持动态工具发现或工具注册？
   - 工具调用的结果如何反馈给 Agent？

5. **记忆机制分析**
   - 短期记忆（对话上下文）的管理策略是什么？
   - 长期记忆（持久化存储）的实现方式是什么？
   - 记忆的检索和遗忘策略是什么？
   - 不同 Agent 间的记忆是否共享？

#### 关键文件关注

```
优先关注：
├── Agent 定义文件（agent/bot/assistant）
├── Prompt 模板（prompts/templates/system_prompt）
├── 工具定义（tools/functions/plugins）
├── 决策/推理逻辑（reasoning/planning/chain）
├── 记忆管理（memory/context/state）
├── Agent 编排/调度（orchestrator/router/dispatcher）
├── 对话管理（dialog/conversation/session）
└── 安全和权限（auth/guardrail/safety）
```

## 三、知识缺口识别方法论

### 3.1 系统性缺口识别

按照以下维度逐一检查，确保不遗漏关键信息：

| 维度 | 检查问题 | 典型缺口 |
|------|----------|----------|
| **是什么** | 产品的核心功能和价值是什么？ | 只有功能列表，没有价值定位 |
| **为谁做** | 目标用户是谁？解决了什么痛点？ | 缺乏用户画像和痛点分析 |
| **怎么做** | 技术架构和实现方案是什么？ | 有架构图但缺关键设计决策的原因 |
| **做得怎样** | 效果指标和实际成效如何？ | 缺少量化的效果数据 |
| **如何用** | 部署、配置、使用方式是什么？ | 缺乏运维和使用指南 |
| **边界在哪** | 系统能力边界和限制是什么？ | 只描述能力，忽略限制 |

### 3.2 缺口严重程度评估

| 严重程度 | 定义 | 处理方式 |
|----------|------|----------|
| **严重** | 缺失核心信息，影响知识库的基本完整性 | 必须向用户确认或从其他渠道补充 |
| **一般** | 缺失重要细节，影响知识库的深度 | 在文档中标注"待补充"，建议用户后续提供 |
| **可忽略** | 缺失边缘信息，不影响知识库的核心价值 | 可暂时跳过，在维护阶段按需补充 |

### 3.3 缺口处理策略

1. **从代码推断** - 对于缺失的文档描述，尝试从代码逻辑中推断系统行为
2. **从测试推断** - 从测试用例中推断功能边界和预期行为
3. **从配置推断** - 从配置文件中推断部署架构和运行参数
4. **从依赖推断** - 从依赖列表中推断技术栈和集成关系
5. **标注待确认** - 对于无法推断的信息，明确标注为"待确认"并列出问题清单

## 四、大型项目并行分析策略

### 4.1 项目规模判断

| 规模 | 代码量 | 文件数 | 分析策略 |
|------|--------|--------|----------|
| 小型 | < 5K 行 | < 50 | 顺序分析，快速完成 |
| 中型 | 5K-50K 行 | 50-500 | 模块化分析，按模块逐个突破 |
| 大型 | 50K-500K 行 | 500-5000 | 分层分析，先架构后细节 |
| 超大型 | > 500K 行 | > 5000 | 重点分析，聚焦核心模块 |

### 4.2 并行分析策略

对于中大型项目，可以按以下方式组织并行分析：

```
第一轮（并行，建立全局认知）：
├── 分析入口文件 -> 了解启动流程和核心调用链
├── 分析目录结构 -> 了解模块划分和职责
├── 分析配置文件 -> 了解技术栈和部署方式
└── 分析 README/文档 -> 了解项目定位和设计意图

第二轮（并行，模块级深入）：
├── 分析核心模块 A -> 详细了解模块 A 的实现
├── 分析核心模块 B -> 详细了解模块 B 的实现
├── 分析核心模块 C -> 详细了解模块 C 的实现
└── 分析集成/接口层 -> 了解模块间交互

第三轮（串行，交叉验证和补充）：
├── 验证模块间接口一致性
├── 补充跨模块的数据流
├── 识别全局性的架构模式
└── 整理知识缺口清单
```

### 4.3 分析粒度控制

- **P1 阶段**：粗粒度分析，目标是"知道有什么"，不需要深入每个实现细节
- **P2 阶段**：中粒度分析，目标是"理解怎么做的"，需要了解核心模块的关键实现
- **P4 阶段**：按需细粒度分析，只对正在编写的文档相关部分进行深入分析

## 五、高效分析技巧

### 5.1 快速入口定位

**从 README 开始：** README 通常包含项目的最佳入口点信息。

**从 CI/CD 配置切入：** `.github/workflows/` 中的构建和测试命令可以快速了解项目的运行方式。

**从 Makefile/Taskfile 切入：** 如果项目有 Makefile，`make help` 或查看 all target 可以快速了解项目的主要操作。

### 5.2 配置文件捷径

| 想了解什么 | 先看什么文件 |
|------------|-------------|
| 技术栈 | `requirements.txt`、`package.json`、`go.mod`、`pom.xml` |
| 部署方式 | `Dockerfile`、`docker-compose.yml`、`helm/` |
| 运行参数 | `config.yaml`、`.env`、`settings.py` |
| API 接口 | `openapi.yaml`、`swagger.json`、路由文件 |
| 数据库 | `migrations/`、`prisma/schema.prisma`、SQL 文件 |
| 认证方式 | `auth/`、`middleware/auth`、`.env` 中的密钥配置 |

### 5.3 测试文件的洞察价值

测试文件是"可执行的文档"，往往包含以下有价值的信息：

- **输入输出规格**：从测试数据的 setup/teardown 了解系统接受和产生的数据格式
- **边界条件**：从边界测试用例了解系统的能力边界
- **错误场景**：从异常测试了解系统的错误处理策略
- **集成方式**：从集成测试了解系统与外部组件的交互协议
- **业务规则**：从业务逻辑测试了解系统的核心业务规则

### 5.4 代码注释和文档字符串

- Python 的 docstring（`"""..."""`）通常包含函数/类的用途说明
- Go 的 godoc 注释（`//`）紧跟声明之后，是官方文档来源
- Java 的 Javadoc（`/** ... */`）包含参数说明和返回值说明
- TypeScript 的 TSDoc（`/** ... */`）包含类型信息和用途说明

### 5.5 Git 历史分析

当静态分析不足以理解设计意图时，可以查看 Git 历史：

- **`git log --oneline -- <file>`** - 查看特定文件的变更历史，理解演进脉络
- **`git log --all --grep="<keyword>"`** - 搜索与特定功能相关的提交
- **`git blame <file>`** - 查看每一行代码的最后修改者和时间
- **`git log --merges --oneline`** - 查看合并历史，了解分支和功能开发节奏

### 5.6 常见陷阱提醒

| 陷阱 | 说明 | 应对 |
|------|------|------|
| 过度深入细节 | 在某个文件上花太多时间，忽略全局 | 设定时间限制，先建立全局认知 |
| 忽略测试文件 | 认为测试不重要而跳过 | 测试文件往往是最准确的"文档" |
| 信任 README 不加验证 | README 可能过时或夸大 | 用代码验证 README 中的描述 |
| 忽略配置文件 | 只看代码不看配置 | 配置文件往往包含关键的运行信息 |
| 遗漏非代码文件 | 忽略 docs/、examples/、prompts/ 等 | 全面的材料清单应包含所有类型 |
