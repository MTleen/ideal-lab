# KB Doc Planning (P3)

## Overview

根据 P2 分析报告，规划知识库的文档结构，确定要生成哪些文档、每个文档包含什么内容、以及写作任务的拆分。

## 输入

- P1-材料清单.md
- P2-分析报告.md
- 项目名称和分类

## 输出

`{分类}/知识库/{项目名称}/P3-文档规划.md`

## 执行步骤

### 1. 确定文档清单

基于项目分类，确定标准7文档 + 用户自定义文档：

**核心文档（4个，所有分类必选）：**

| # | 文档 | 必选 | 说明 |
|---|------|------|------|
| 1 | 概述.md | ✅ | 产品定位、目标客户、痛点、价值、竞品 |
| 2 | 技术架构.md | ✅ | 架构设计、架构图、技术栈、部署架构 |
| 3 | 功能清单.md | ✅ | 功能模块、功能点、优先级、依赖 |
| 4 | 部署与运维.md | ✅ | 部署方案、环境、配置、监控 |

**分类专属文档（3个，根据分类自动选择）：**

| 分类 | 文档5 | 文档6 | 文档7 |
|------|-------|-------|-------|
| 数据治理 | 数据流.md | 数据质量.md | 应用成效.md |
| 模型训练 | 训练管线.md | 模型评估.md | 应用成效.md |
| 知识库与知识治理 | 知识模型.md | 检索与生成.md | 应用成效.md |
| 智能体 | Agent设计.md | 工具与集成.md | 应用成效.md |

**自定义文档（用户在 P3 审核时指定）：**

追加到标准7文档之后。

> **可选额外文档**：`核心算法详解.md` — 当项目包含复杂算法（如LLM调用、Pipeline编排、数据处理管线等）时建议生成。此文档需要 P2 阶段深入源码提取算法细节才能正确生成。如果 P2 分析报告显示算法复杂度高，应主动建议生成此文档。

### 2. 规划各文档内容要点

根据 P2 分析报告，为每个文档列出具体内容要点：

- **从分析报告中提取**：每个文档应该包含哪些关键信息
- **标注信息来源**：哪些内容来自哪个分析维度
- **标注信息充分度**：充分 / 部分 / 需补充
- **标注图表需求**：是否需要 Mermaid 图表，什么类型

### 3. 拆分写作任务

将文档生成拆分为可并行执行的写作任务：

- 每个文档 = 1个写作任务
- 任务间依赖关系（如技术架构依赖概述中的定位信息）
- 建议的并行分组

## 写作任务拆分

### 任务依赖图

```
概述.md ──────→ 技术架构.md ──→ 功能清单.md
     │                │
     │                ↓
     └──→ 分类专属文档 ×3
部署与运维.md（独立）
```

### 任务列表

### 任务 1: 概述.md
- **文件**：`{分类}/知识库/{项目名称}/概述.md`
- **依赖**：无
- **信息充分度**：高
- **写作要点**：
  - 产品定位需从 README 和项目文档中提炼
  - 目标客户需从功能和设计文档中推断
  - 竞品对比标注"基于领域知识推断"

### 任务 2: 技术架构.md
- **文件**：`{分类}/知识库/{项目名称}/技术架构.md`
- **依赖**：任务1（产品定位影响架构描述视角）
- **信息充分度**：中
- **图表需求**：系统架构图(Mermaid)、部署架构图(Mermaid)
- **写作要点**：
  - 技术栈从配置文件中提取（含版本号）
  - 架构图需覆盖所有核心模块
  - 部署架构从 Dockerfile/docker-compose 中提取
