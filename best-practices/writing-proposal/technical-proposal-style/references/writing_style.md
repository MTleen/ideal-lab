# 写作风格规范

## 1. 人称使用规范

### 1.1 禁止使用第一人称

**问题表述**：
- ❌ "我们选择微服务架构"
- ❌ "我们评估了三种方案"
- ❌ "我们尝试过直接调用API"
- ❌ "我们设计了月度更新机制"

**改写方式**：

| 方式 | 示例 |
|-----|------|
| 被动语态 | "微服务架构被采用" → 不推荐，过于生硬 |
| 无主句 | "采用微服务架构" → 推荐 |
| 以方案为主语 | "本方案选择微服务架构" → 推荐 |
| 以系统为主语 | "系统采用微服务架构" → 推荐 |

**改写示例**：
- ❌ "我们选择微服务架构，因为..." → ✅ "微服务架构被选定，其优势在于..."
- ❌ "我们评估了三种方案" → ✅ "经对三种方案的综合评估"

### 1.2 第二人称使用

技术文档中应避免使用第二人称"你"、"你们"。

- ❌ "你需要配置以下参数"
- ✅ "需配置以下参数"
- ✅ "系统需配置以下参数"

## 2. 问句标题禁止

### 2.1 问题类型

禁止在标题中使用疑问句，包括：

| 问题类型 | ❌ 错误示例 | ✅ 正确示例 |
|---------|-----------|-----------|
| 为什么型 | "为什么需要这些数据？" | "数据需求分析" |
| 什么是型 | "什么是微服务架构？" | "微服务架构概述" |
| 如何型 | "如何实现数据融合？" | "数据融合实现方案" |
| 多少型 | "需要多少GPU资源？" | "GPU资源配置规划" |

### 2.2 改写模式

将问句转化为陈述性标题：

```
问句 → 分析类标题
"为什么需要X？" → "X的需求分析" / "X的必要性说明"
"X解决什么问题？" → "X解决的问题域"

问句 → 概述类标题
"什么是X？" → "X概述" / "X简介"
"X有哪些特点？" → "X的核心特性"

问句 → 方案类标题
"如何实现X？" → "X的实现方案" / "X的技术路线"
"X怎么做？" → "X的实施步骤"
```

### 2.3 内容组织替代

如果确实需要引出问题，使用以下方式：

**方式一：段落首句陈述问题背景**
```markdown
#### 数据需求分析

环境监测分析的核心是"关联"。要判断一次污染事件的来源，需要同时整合多类数据...
```

**方式二：背景-问题-方案结构**
```markdown
#### 数据治理策略

当前数据治理面临三个核心挑战：数据质量参差不齐、时空基准不统一、格式标准各异。
针对这些挑战，本方案采用...
```

## 3. 论证逻辑规范

### 3.1 正面表述优先

**问题**：过多使用"放弃的方案"、"放弃XXX的原因"等负面表述。

**改进**：采用对比分析模式，先说各方案优缺点，再说选择结果。

| ❌ 负面表述 | ✅ 正面表述 |
|-----------|-----------|
| "放弃的方案：直接使用各源系统的数据" | "方案对比：数据治理策略选择" |
| "放弃从头预训练的原因" | "方案选择依据：微调与预训练对比" |
| "放弃TensorFlow的原因" | "框架选型分析" |

### 3.2 What-Why-How 模式

每个技术选型或方案描述应包含三个要素：

```
**What（是什么）**：技术/方案的准确定义
**Why（为什么）**：选择的理由和依据
**How（怎么做）**：具体的实现方式
```

**示例**：
```markdown
#### 2.1.4.1 数据治理路线

**方案选择**：构建统一数据中台。

**选择依据**：试点结果表明，直接使用源系统数据存在质量参差不齐的问题，
数据质量仅达60%。统一数据中台可将数据质量提升至95%。

**实现方式**：采用ClickHouse处理时序数据，PostgreSQL+PostGIS处理空间数据，
通过Apache NiFi实现数据采集编排。
```

### 3.3 避免绝对化表述

| ❌ 禁止 | ✅ 替代 |
|--------|--------|
| 唯一、仅此一家 | 主要、核心 |
| 必然、一定 | 预计、预计可达到 |
| 完美、最优 | 较优、相对更优 |
| 彻底解决 | 有效缓解、显著改善 |
| 零缺陷、零误差 | 缺陷率低于X%、误差控制在Y范围内 |

## 4. 段落组织规范

### 4.1 段落长度

- 单段落不超过 200 字（约 4-6 句）
- 超过 200 字应拆分为多个段落
- 每个段落聚焦一个主题

### 4.2 段落结构

推荐采用"主题句 + 支撑句 + 总结句"结构：

```
【主题句】本系统采用微服务架构设计。
【支撑句】微服务架构支持服务独立部署和扩展，故障隔离能力强，团队可并行开发。
【总结句】这种架构设计确保系统的长期可维护性和可扩展性。
```

### 4.3 分隔线使用

- 避免频繁使用 `---` 分隔线
- 分隔线仅用于章节间必要分隔
- 同一章节内应通过段落过渡而非分隔线
