# 标书写作：去AI味风格指南

## 核心理念

**标书不是产品说明书，是一封"说服信"。**

评审专家看标书时在找三件事：
1. 你真的理解我的问题吗？
2. 你的方案靠谱吗？
3. 凭什么是你？

---

## 一、什么是"AI味"？

### 1.1 AI味的典型特征

| 特征 | 具体表现 | 问题本质 |
|------|---------|---------|
| 结构过于工整 | 永远是"首先、其次、最后"、"一是、二是、三是" | 像机器生成的，缺乏人的思考痕迹 |
| 形容词堆砌 | "先进、智能、高效、全面、精准、完善"轮番上阵 | 空洞，没有实质内容 |
| 空洞的完美感 | 方案看起来无懈可击，没有任何取舍和判断 | 缺乏专业思考，像模板套用 |
| 缺乏业务温度 | 像说明书，不像写给客户看的方案 | 没有站在客户角度思考 |
| 表格依赖症 | 能用表格的地方绝不多写一句解释 | 懒惰，缺乏深入思考 |
| 罗列式思维 | "支持A、B、C、D、E..." | 没有重点，没有判断 |

### 1.2 AI味示例与改写

#### 示例1：空洞的形容词

```
❌ AI味：
"本系统采用先进的深度学习技术，具有智能化、自动化、高效率的特点，
可为用户提供全面、精准的水质监测服务。"

✅ 人味：
"上海河网密布、水体类型复杂，传统地面监测点位有限，难以全面反映水质
空间分布。我们基于Sentinel-2卫星数据，针对上海Ⅱ类水体特征定制了反演
算法，叶绿素a反演R²达到0.80以上，可实现对全市主要水体的每周一次全覆盖
监测。"
```

**改写要点：**
- 删除空洞形容词（先进、智能、全面、精准）
- 从客户问题出发（上海河网密布、点位有限）
- 给出具体方案（Sentinel-2、定制算法）
- 用数据说话（R²≥0.80、每周一次）

#### 示例2：罗列式思维

```
❌ AI味：
"本系统支持Sentinel-2、Landsat-8/9、高分一号、高分二号、高分六号、
MODIS、FY-3系列等多源卫星数据..."

✅ 人味：
"卫星数据源的选择，需要在分辨率、重访周期、数据成本之间权衡。
Sentinel-2分辨率10米、重访周期5天、数据免费，是日常监测的主力数据源；
高分二号分辨率1米，用于重点区域的精细化监测；MODIS虽然分辨率较低，
但每日覆盖，适合大尺度的快速预警。三种数据源各司其职，形成互补。"
```

**改写要点：**
- 不罗列，而是说明选择逻辑
- 解释为什么选这个（分辨率、周期、成本权衡）
- 说明各自定位（主力、精细化、预警）

#### 示例3：缺乏场景感

```
❌ AI味：
"系统具备水华监测功能，可自动识别水华区域并生成报告。"

✅ 人味：
"每年7-9月是上海湖泊水华高发期。传统方式依赖人工巡湖，发现时往往已经
暴发。系统通过每日卫星过境自动扫描，一旦检测到叶绿素a异常升高，立即推送
预警到值班人员手机，同时生成水华范围专题图，供决策参考。"
```

**改写要点：**
- 描述真实场景（7-9月水华高发期）
- 对比传统方式的痛点（人工巡湖、发现滞后）
- 描述使用体验（推送预警、专题图）

#### 示例4：表格依赖症

```
❌ AI味：
**质检指标体系：**
| 指标类别 | 检测内容 | 质量要求 |
|----------|----------|----------|
| 覆盖完整性 | 影像覆盖范围 | 完全覆盖感兴趣区域 |
| 云量比例 | 云层覆盖面积 | 一般≤20% |
...

✅ 人味：
"数据质检是遥感数据标准化生产的第一道关口，直接影响后续分析产品的可靠性。
本系统建立了覆盖完整性、云量比例、几何精度等多维度的质检指标体系，通过
自动化质检算法实现影像质量的快速评估与筛选。

上海地区多云多雨，云检测是质检的核心难点。我们采用深度学习云检测模型
结合传统光谱阈值法的混合策略，云检测综合准确率可达93%以上，有效保障了
可用数据的筛选效率。

**质检指标体系：**
| 指标类别 | 检测内容 | 质量要求 |
..."
```

**改写要点：**
- 表格前增加引导段（为什么需要质检）
- 结合客户实际情况（上海多云多雨）
- 说明技术难点和解决方案
- 表格作为补充，不是主体

## 二、好标书的叙述逻辑

### 2.1 核心逻辑：问题→理解→方案→效果

```
┌─────────────────────────────────────────────────────────────┐
│  客户有什么问题？                                           │
│  ↓                                                         │
│  我们怎么理解这个问题？（体现专业洞察）                      │
│  ↓                                                         │
│  我们用什么方案解决？                                       │
│  ↓                                                         │
│  为什么这个方案是合适的？（有取舍、有判断）                  │
│  ↓                                                         │
│  预期能达到什么效果？（有数据、有场景）                      │
└─────────────────────────────────────────────────────────────┘
```

### 2.1.1 逻辑要自然，不要标签化

**核心原则**：叙述逻辑是"问题→理解→方案→效果"，但写作上要自然流畅，绝不能用【问题】【理解】【方案】这种机械化标签。

```
❌ 错误写法（机械化标签）：
【问题】上海监测覆盖不足
【理解】这是因为地面站点有限，传统手段难以全覆盖
【方案】因此我们建设遥感监测系统
【效果】可以实现全市域覆盖

✅ 正确写法（逻辑清晰，行文自然）：
上海陆域面积6340平方公里，但地面监测站点约200个，传统点位监测只能覆盖
不足1%的区域。崇明东滩等生态保护区，徒步采样一天只能覆盖5%面积，存在
大量监测盲区。

遥感监测能够突破地面站点的空间限制。我们基于Sentinel-2卫星数据，
针对上海Ⅱ类水体特征定制反演算法，可实现全市主要水体的每周一次全覆盖监测，
从根本上解决覆盖不足的问题。
```

**为什么不能用标签？**
1. 标签让文章像填空题，缺乏专业感
2. 标签打断了阅读的连贯性
3. 评审专家想看的是你的思考过程，不是你的模板

**如何做到逻辑清晰但行文自然？**
1. 用"但是""然而""因此"等连接词，不用标签
2. 每段开头直接陈述事实或问题，不用【问题】引导
3. 用"我们选择""本方案采用"引出方案，不用【方案】标签
4. 用数据、场景来呈现效果，不用【效果】标签

### 2.2 叙述逻辑示例

#### 技术选型的叙述逻辑

```
❌ 错误逻辑：先说技术，再说特点
"本系统采用PostgreSQL数据库，具有开源免费、性能稳定、社区活跃的特点..."

✅ 正确逻辑：先说问题，再说选择
"系统需要存储和查询海量的时空监测数据。传统关系型数据库在空间数据支持上
存在局限，专业GIS数据库又成本高昂。我们选择PostgreSQL配合PostGIS扩展，
既获得了完整的空间数据管理能力，又保持了开源的技术栈。经测试，百万级
空间查询响应时间低于100毫秒。"
```

#### 功能描述的叙述逻辑

```
❌ 错误逻辑：描述功能
"系统具备异常检测功能，可自动识别监测数据中的异常值。"

✅ 正确逻辑：描述场景和价值
"环境监测站点分布在全市各地，人工逐一核查数据异常效率低下。系统内置的
异常检测模块，能够自动识别监测值的突变、超标、缺失等问题，并将异常信息
按严重程度排序后推送给值班人员。试运行期间，异常发现时间从平均4小时
缩短到15分钟。"
```

### 2.3 有取舍，有判断

好标书不是"全面罗列"，而是"有判断的选择"。

```
❌ AI味：
"系统支持多种深度学习框架，包括TensorFlow、PyTorch、Keras、MXNet..."

✅ 人味：
"在深度学习框架的选择上，我们采用PyTorch作为主力框架。相比TensorFlow，
PyTorch的动态计算图更便于模型调试和快速迭代，这对于需要频繁调整的
遥感影像解译任务尤为重要。同时，PyTorch拥有丰富的预训练模型库，
可以显著加速模型开发。"
```

**关键点：**
- 说明选择的标准（调试便利性、迭代速度）
- 与竞品对比（相比TensorFlow）
- 结合具体场景（遥感影像解译任务）

## 三、好标书的语言风格

### 3.1 用数据说话，不用形容词

| ❌ AI味 | ✅ 人味 |
|--------|--------|
| 系统响应速度快 | 单景影像处理时间≤30秒 |
| 监测精度高 | 叶绿素a反演R²≥0.80，经200+样本验证 |
| 覆盖范围广 | 覆盖全市6340平方公里陆域及周边近海 |
| 系统稳定可靠 | 系统可用性≥99.5%，年故障时间≤44小时 |
| 数据处理能力强 | 日均处理卫星影像100+景，数据量500GB+ |
| 预警及时 | 从异常检测到预警推送≤5分钟 |

### 3.2 用专业判断，不用万能句式

| ❌ AI味 | ✅ 人味 |
|--------|--------|
| 本系统采用先进的深度学习技术 | 针对上海Ⅱ类水体光谱特征复杂的问题，我们采用深度学习方法 |
| 具有智能化、自动化特点 | 系统可自动完成从数据获取到报告生成的全流程，无需人工干预 |
| 为用户提供全面的服务 | 一次卫星过境，即可生成全市水质分布图、异常点位清单、趋势分析报告 |
| 实现了高效的数据处理 | 原需2人天完成的监测报告，现在10分钟内自动生成 |

### 3.3 有温度，有关怀

```
❌ AI味：
"系统采用B/S架构，支持多终端访问，用户可通过浏览器使用系统。"

✅ 人味：
"考虑到环境监测人员经常外出作业，系统专门优化了移动端体验。在手机上
就能查看最新监测结果、接收预警推送，不需要回到办公室打开电脑才能工作。"
```

```
❌ AI味：
"系统提供完善的权限管理功能，支持多级角色配置。"

✅ 人味：
"不同岗位对数据的访问需求不同。一线监测人员只需查看自己负责的站点，
科室主任需要掌握全区情况，领导层则关注整体态势。系统预设了三种角色
模板，也支持自定义配置，新员工入职5分钟即可完成权限设置。"
```

### 3.4 有画面感，有场景感

```
❌ AI味：
"系统支持应急监测功能，可在突发事件时快速响应。"

✅ 人味：
"当发生突发环境事件时，值班人员只需在系统中框选事发区域，系统即可
自动调取最近的卫星影像、周边监测站点数据、历史污染记录，3分钟内生成
初步态势研判报告，为应急决策争取宝贵时间。"
```

## 四、常见问题检查清单

### 4.1 自检清单

写作完成后，用以下问题自检：

| 检查项 | 问题 | 通过标准 |
|--------|------|---------|
| 叙述起点 | 是否从客户问题出发，而不是从技术出发？ | 每个功能描述前都有"为什么需要" |
| 形容词密度 | 是否有连续3个以上空洞形容词？ | 删除"先进、智能、高效、全面"等词 |
| 数据支撑 | 是否每个性能描述都有具体数字？ | 用"≤30秒"替代"快速" |
| 场景感 | 是否描述了具体的使用场景？ | 有"当...时，系统会..."的描述 |
| 取舍判断 | 是否说明了为什么选这个方案？ | 有"选择X而不是Y，是因为..." |
| 表格解释 | 每个表格前是否有引导性文字？ | 表格是补充，不是主体 |
| AI痕迹 | 是否有"首先、其次、最后"的固定句式？ | 改用更自然的过渡 |

### 4.2 典型问题段落改写练习

#### 练习1：改写以下AI味段落

```
原文：
"本系统采用先进的大数据技术，实现了海量数据的高效处理。系统具有智能化、
自动化、可视化的特点，为用户提供全面的数据分析服务。"

改写思路：
1. 删除空洞形容词（先进、高效、智能化、自动化、可视化、全面）
2. 从客户问题出发
3. 用具体数据替代
4. 描述使用场景

参考改写：
"系统每日接入全市200+个监测站点的实时数据，日均数据量达50万条。传统
方式下，人工整理一份全市监测日报需要2小时；系统上线后，数据自动汇聚、
自动分析、自动生成报告，值班人员只需10分钟审核即可发布。"
```

#### 练习2：为表格增加引导段

```
原文：
**技术选型：**
| 技术领域 | 选型方案 | 说明 |
|----------|----------|------|
| 数据库 | PostgreSQL | 开源关系型数据库 |
| 缓存 | Redis | 高性能内存数据库 |
| 消息队列 | Kafka | 分布式消息系统 |

改写思路：
1. 说明选型的整体原则
2. 解释为什么这样组合
3. 说明技术栈的一致性

参考改写：
"技术选型遵循'成熟稳定、开源优先'的原则。数据库选择PostgreSQL，既因为
其优秀的空间数据支持（PostGIS扩展），也因为与Python生态的无缝集成。
Redis作为缓存层，可显著提升热点数据的访问速度。Kafka负责数据流转，
其持久化特性确保监测数据不丢失。整套技术栈均为开源方案，无厂商锁定风险。

**技术选型：**
| 技术领域 | 选型方案 | 说明 |
..."
```

## 五、写作模式速查

### 5.1 功能描述模式

```
[客户痛点] → [我们的理解] → [解决方案] → [具体效果]

示例：
"上海每年7-9月水华高发（痛点），传统人工巡湖发现滞后（理解），
系统通过每日卫星扫描实现早期预警（方案），预警时间提前3天（效果）。"
```

### 5.2 技术选型模式

```
[选择标准] → [备选方案] → [选择结果] → [选择理由]

示例：
"框架选择以调试效率和社区生态为标准（标准），对比了TensorFlow和PyTorch
（备选），最终选择PyTorch（结果），因其动态图更适合频繁迭代的遥感任务
（理由）。"
```

### 5.3 性能描述模式

```
[具体指标] + [验证方式] + [适用条件]

示例：
"叶绿素a反演R²≥0.80（指标），经200+样本独立验证（验证），
适用于上海地区Ⅱ类水体（条件）。"
```

### 5.4 场景描述模式

```
[触发条件] → [系统行为] → [用户收获]

示例：
"当监测值超标时（触发），系统自动推送预警到值班人员手机（行为），
无需人工盯屏即可及时响应（收获）。"
```

## 六、标书写作的"三个一"原则

### 一个核心
每段话只说一个核心意思，不要堆砌。

### 一个场景
每个功能至少描述一个具体使用场景。

### 一个数据
每个性能描述必须有具体数字支撑。
