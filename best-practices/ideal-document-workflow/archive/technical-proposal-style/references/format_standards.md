# 格式规范

## 1. 标题层级规范

### 1.1 层级定义

| 层级 | Markdown | 用途 | 编号格式 |
|-----|----------|------|---------|
| 一级 | `#` | 章标题 | 第X章 |
| 二级 | `##` | 节标题 | X.Y |
| 三级 | `###` | 小节标题 | X.Y.Z |
| 四级 | `####` | 段落分组标题 | X.Y.Z.W |
| 五级 | `#####` | 仅用于极细分内容 | 尽量避免 |

### 1.2 标题编号规则

**章节编号**（使用中文数字）：
```markdown
# 第一章 采购需求分析
# 第二章 总体技术方案
# 第三章 生态环境遥感监测系统
```

**节编号**（二级标题，使用阿拉伯数字）：
```markdown
## 1.1 项目背景
## 1.2 核心业务需求
## 1.3 项目难点
```

**小节编号**（三级标题）：
```markdown
### 1.2.1 生态环境遥感监测
### 1.2.2 监测机构监管
```

**段落分组编号**（四级标题）：
```markdown
#### 1.2.1.1 数据采集模块
#### 1.2.1.2 数据存储模块
```

### 1.2.1 章节编号对照表

| 数字 | 中文 | 数字 | 中文 |
|-----|------|-----|------|
| 1 | 一 | 8 | 八 |
| 2 | 二 | 9 | 九 |
| 3 | 三 | 10 | 十 |
| 4 | 四 | 11 | 十一 |
| 5 | 五 | 12 | 十二 |
| 6 | 六 | 13 | 十三 |
| 7 | 七 | 14 | 十四 |

### 1.2.2 多文件章节编号规范

当技术方案拆分为多个 Markdown 文件时，每个文件的章节编号必须：

1. **保持连续性**：所有文件合并后，章节编号应从第一章连续编号到最后一章
2. **子标题对应章节**：二级及以下标题的编号前缀必须与所属章节一致
3. **文件名与内容对应**：文件名（如"第三章-xxx.md"）应与实际章节编号一致

**示例**：
```
文件：第三章-环境污染溯源分析系统.md
内容：
# 第五章 环境污染溯源分析系统    ← 错误！应为"第三章"
## 5.1 溯源分析概述              ← 正确，与章节号对应
```

**正确做法**：
```
文件：第三章-环境污染溯源分析系统.md
内容：
# 第三章 环境污染溯源分析系统
## 3.1 溯源分析概述
## 3.2 大气污染溯源分析
### 3.2.1 多源数据联动
```

### 1.3 标题使用禁忌

| ❌ 禁止 | 原因 |
|--------|------|
| 使用问句作为标题 | 不符合正式文档风格 |
| 五级标题作为段落引导 | 层级过深，应整合内容 |
| 标题末尾加句号 | 标题不是句子 |
| 标题中使用英文标点 | 使用中文标点 |
| 跳跃层级（如从##直接到####） | 保持层级连续 |

### 1.4 标题长度规范

- 一级标题：不超过15字
- 二级标题：不超过20字
- 三级标题：不超过25字
- 四级标题：不超过30字

## 2. 表格规范

### 2.1 表格结构

**表头**：
- 表头必须清晰、简洁
- 使用加粗格式（Markdown中自动处理）
- 避免合并单元格

**表格编号**：
- 格式：表X-Y（X为章节号，Y为序号）
- 示例：表2-1、表2-2

**表格标题**：
- 位置：表格上方
- 格式：**表X-Y 表格名称**

### 2.2 表格内容规范

| 要求 | 说明 |
|-----|------|
| 对齐方式 | 数字右对齐，文字左对齐 |
| 单位 | 在表头或列标题中标明 |
| 空值 | 用"—"或"N/A"表示，不空白 |
| 注释 | 置于表格下方，格式：*注：...* |

### 2.3 表格示例

```markdown
**表2-1 技术方案对比**

| 方案 | 准确率 | 响应时间 | 成本 | 适用场景 |
|-----|-------|---------|-----|---------|
| 方案A | 85% | 200ms | 中 | 通用场景 |
| 方案B | 92% | 500ms | 高 | 高精度场景 |
| 方案C | 78% | 50ms | 低 | 实时场景 |

*注：准确率基于测试集验证（n=1200）；成本为相对值*
```

### 2.4 表格使用原则

**核心原则**：表格是辅助手段，文字论述是主体。**严禁"以表代文"**。

**允许使用场景**：
- 数据汇总（如技术栈清单、性能指标）
- 配置信息（如参数表、版本表）
- 总结性对比（在充分文字论述后）

**禁止使用场景**：
- ❌ 直接用表格呈现技术选型论证
- ❌ 用表格替代方案对比的文字分析
- ❌ 表格前后缺乏充分的文字说明

### 2.5 "以表代文"问题示例

**❌ 错误示例（以表代文）**：
```markdown
经单体架构和微服务架构评估：

| 维度 | 单体架构 | 微服务架构 | 选择 |
|-----|---------|-----------|-----|
| 开发效率 | 初期快 | 初期慢 | - |
| 扩展性 | 差 | 好 | 微服务 |
| 运维复杂度 | 低 | 高 | - |

**选择微服务的原因**：业务差异大，需要独立扩展。
```

**问题分析**：
- 引入语仅1句话，无背景分析
- 对比分析全部在表格中
- 结论说明过于简略（仅1句话）

**✅ 正确示例（文字为主，表格为辅）**：
```markdown
本系统架构选型需综合考虑业务特点、团队能力、运维成本等因素。

**业务特点分析**：环境监测业务涵盖遥感监测、污染溯源、应急响应等多个相对独立的业务领域。各业务模块的数据模型、处理逻辑、访问模式差异显著。遥感监测以影像处理为主，计算密集；污染溯源涉及多源数据关联分析，IO密集；应急响应则需要高并发、低延迟的实时处理能力。

**单体架构分析**：单体架构将所有业务模块部署在同一个应用中，共享数据库和资源。其优势在于初期开发效率高，部署和运维相对简单，技术门槛较低。然而，随着业务规模增长，单体架构的局限性逐渐显现：任一模块的故障可能影响整体系统稳定性；无法针对特定业务独立扩容；代码耦合度高，维护成本随规模增长。

**微服务架构分析**：微服务架构将业务功能拆分为独立部署的服务单元，每个服务可独立开发、部署和扩展。其核心优势在于故障隔离——单个服务的故障不会波及其他服务；弹性扩展——可根据各业务的实际负载独立扩容；技术灵活——不同服务可采用最适合的技术栈。代价是运维复杂度显著增加，需要建立完善的服务治理体系。

**综合评估与选择**：基于上述分析，考虑到环境监测业务的模块独立性较强，且不同业务对计算资源的需求差异显著，本方案选择微服务架构。虽然初期投入和运维成本较高，但从长期可维护性和业务扩展性角度，微服务架构是更优的选择。

（表格作为辅助总结）
**表2-X 架构模式对比总结**

| 维度 | 单体架构 | 微服务架构 | 选择 |
|-----|---------|-----------|-----|
| 开发效率 | 初期快 | 初期慢 | - |
| 扩展性 | 差 | 好 | 微服务 |
| 运维复杂度 | 低 | 高 | - |
```

### 2.6 表格使用量化标准

| 检查项 | 标准 | 说明 |
|-------|------|-----|
| 表格前文字说明 | ≥3句 | 背景分析、问题引入 |
| 表格后文字说明 | ≥2句 | 结论阐述、理由说明 |
| 技术选型类表格 | 必须有充分论证 | 不能仅有引入语 |
| 单节对比表格数 | ≤2个 | 超过需合并或精简 |

## 3. 图表规范

### 3.1 图表编号

- 格式：图X-Y（X为章节号，Y为序号）
- 示例：图2-1、图2-2

### 3.2 图表标题

- 位置：图表下方
- 格式：**图X-Y 图表名称**

### 3.3 图表引用

在正文中引用图表时：
- ✅ "如图2-1所示"
- ✅ "（见图2-1）"
- ❌ "如下图所示"（应明确编号）

### 3.4 占位符规范

当图表暂未制作时，使用占位符：

```markdown
【图X-Y 图表名称】

*说明：图表内容描述，用于后续制作参考*
```

**示例**：
```markdown
【图2-1 系统总体架构图】

*说明：展示"四层三翼"架构的分层结构和组件关系*
```

## 4. 列表规范

### 4.1 无序列表使用原则

**核心原则**：正式技术标书应以段落叙述为主，无序列表仅作为辅助手段，**严禁过度使用**。

**允许使用场景**：
- 目录式列举（如技术栈汇总）
- 简短的并列要点（≤3条）
- 补充说明性内容

**禁止使用场景**：
- ❌ 技术选型论证（应改为段落叙述）
- ❌ 功能模块描述（应改为连贯段落）
- ❌ 架构层次说明（应改为层级段落）
- ❌ 配置参数列举（应整合为描述性文字）
- ❌ 连续超过3个要点的列举

### 4.2 无序列表改写示例

**❌ 错误示例（过度使用列表）**：
```markdown
Web端技术架构：
- 框架：Vue 3.x + TypeScript
- UI：Element Plus / Ant Design Vue
- 状态：Pinia
- 构建：Vite 5.x
- 地图：OpenLayers / Cesium
- 可视化：ECharts 5.x
```

**✅ 正确示例（段落叙述）**：
```markdown
Web端采用Vue 3.x框架结合TypeScript语言开发，UI组件库选用Element Plus或Ant Design Vue。状态管理通过Pinia实现，构建工具采用Vite 5.x以提升开发效率。地图展示层集成OpenLayers和Cesium，分别用于二维和三维地理信息可视化；数据可视化层采用ECharts 5.x实现图表渲染。
```

### 4.3 有序列表

用于有顺序关系的内容（如步骤、流程）：

```markdown
1. 第一步：描述
2. 第二步：描述
3. 第三步：描述
```

**注意**：有序列表也需控制使用，优先考虑段落叙述。

### 4.4 列表层级

```markdown
- 一级要点
  - 二级要点
    - 三级要点
```

**注意**：层级不超过三级。若需三级以上，应重新组织内容结构。

### 4.5 列表使用量化标准

| 检查项 | 标准 | 说明 |
|-------|------|-----|
| 单节无序列表数量 | ≤3处 | 超过则需整合 |
| 单个列表条目数 | ≤5条 | 超过则改段落 |
| 列表总占比 | ≤5%行数 | 相对全文总行数 |

## 5. 引用与注释规范

### 5.1 引用格式

**直接引用**：
- 使用引号标明
- 注明出处

**间接引用**：
- 用自己的话表述
- 可选择性注明出处

### 5.2 脚注与尾注

**脚注**（用于补充说明）：
```markdown
正文内容[^1]

[^1]: 补充说明内容
```

**尾注**（用于参考文献）：
- 按引用顺序编号
- 格式统一

### 5.3 注释块

用于重要提示或警告：

```markdown
> **注意**：此处需特别关注...

> **说明**：补充说明内容...

> **警告**：风险提示...
```

## 6. 分隔线使用

### 6.1 使用场景

分隔线 `---` 仅用于：
- 章与章之间的分隔
- 大节之间的分隔（可选）

### 6.2 禁止使用

同一章节内不应使用分隔线，应通过：
- 段落过渡
- 小标题分隔
- 空行分隔

## 8. 数字与单位规范

### 8.1 数字格式

| 类型 | 规范 | 示例 |
|-----|------|-----|
| 整数 | 直接书写 | 100 |
| 小数 | 保留必要精度 | 3.14 |
| 百分比 | 使用% | 95% |
| 大数 | 使用千分位 | 1,000,000 |
| 范围 | 使用波浪线 | 100~200 |

### 8.2 单位规范

- 使用国际标准单位
- 单位与数字之间留空格
- 示例：100 ms、50 GB、25 °C

### 8.3 时间格式

| 类型 | 格式 | 示例 |
|-----|------|-----|
| 日期 | YYYY-MM-DD | 2026-03-15 |
| 时间 | HH:MM:SS | 14:30:00 |
| 日期时间 | YYYY-MM-DD HH:MM | 2026-03-15 14:30 |
