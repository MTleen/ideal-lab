# Panel Review — 多维会审

从会话历史中识别上文产出的文档、代码、方案或 PPT，根据内容类型和复杂度动态组建专家审查团队（agent team），每个专家从独特的攻击面进行对抗性审查，最终去重、按严重度排序，输出攻击面报告。支持对同一对象发起多轮重审，逐轮收敛到高质量发现。

## 核心原则

1. **角色动态生成**：不预设固定角色列表。按固定的生成流程，从内容特征推导审查角色。
2. **攻击面驱动**：每个角色绑定一个具体的攻击面，不是"审文档"而是"审 X 维度有没有 Y 类问题"。
3. **专用团队优先**：优先使用 Agent Team 做并行审查，不可用时退化为顺序子 agent。
4. **深度自适应**：审查深度由内容长度和复杂度决定，短文快速模式，复杂系统深度分析。
5. **可迭代重审**：用户可以对报告中的任意角色要求重审，调整攻击面后重新审查。

## 五个阶段

```
Phase 1   Analyze    → 扫描会话历史，识别上文产出了什么（文档/代码/方案/PPT）
Phase 2   Assemble   → 按固定流程生成审查角色，每个绑定一个攻击面，用户确认
Phase 3   Attack     → 创建 agent team，每个角色并行执行自己的攻击面审查
Phase 4   Report     → 去重、合并同类发现、按严重度排序、输出攻击面报告
Phase 5   Iterate    → 用户对结果不满意时，调整角色或攻击面后重新审查
```

### 违反以下规则的后果

以下反模式必须在对应阶段被显式检查。Phase 4 汇总裁决时，发现违反反模式的条目应直接降级或丢弃：

| # | 反模式 | 检查点 | 处理 |
|---|--------|--------|------|
| A1 | 预设固定角色列表 | Phase 2 | 每个审查任务的角色必须由内容特征驱动，不得从"常用角色"中直接搬用 |
| A2 | 角色数 > 7 | Phase 2 | 超过 7 个时强制精简：只保留攻击面最不重叠的 7 个 |
| A3 | 攻击面太宽泛 | Phase 2 呈现前 | 不得出现"审文档质量"这种表述，必须是"审 X 维度有没有 Y 类问题" |
| A4 | 审查 prompt 跨界 | Phase 3 角色 prompt | 角色 prompt 中显式声明"只审查你的攻击面，不要提及安全性/性能/..." |
| A5 | 去重草率 | Phase 4 | 同一问题多个角度应合并保留多视角依据，不得简单丢弃 |
| A6 | 跳过用户确认 | Phase 2 | 角色清单和攻击面描述必须经用户确认后再执行 Phase 3 |
| A7 | 沉默即安全 | Phase 3 角色 prompt | 不确定但可疑的发现必须输出，标注"低置信度（confidence: low）"并附原因 |
| A8 | 审查对象切片不当 | Phase 3 | 审查内容过长时必须分段喂给每个 teammate，不得依赖 teammate 自行搜索 |

### Phase 1: 分析上下文

#### Step 1: 定位审查对象

扫描会话历史，按以下优先级定位需要审查的内容：

1. 用户在调用本 skill 时**显式指定的**文件/文件夹/对话片段
2. 会话中最近 **20 轮对话**内由 agent 产出并保存到文件的内容（通过 Write/Edit 工具调用记录识别）
3. 用户在上文明确提到"这个方案"、"刚才的代码"等指代的内容

以下内容**不计入**审查对象：
- agent 的推理过程和中间思考（thinking 块）
- 用户截图/图片（除非用户明确要求审查图片内容）
- 会话历史本身的结构性内容（工具调用参数等）

#### Step 2: 空场景处理

如果未识别到任何产出：

> 我没有在最近的会话历史中找到可以被审查的产出（新写入的文件、生成的方案、代码等）。
>
> 请告诉我你想审查什么——可以直接粘贴内容、指定文件路径，或者重新执行需要审查的工作后再调用我。

等待用户提供审查对象。最多重试 3 轮交互。如果用户 3 轮后仍不提供有效审查对象，输出以下信息并终止：

> 未收到有效的审查对象，多维会审终止。你可以随时重新调用我，并直接指定要审查的内容。

#### Step 3: 提取特征

对每个审查对象提取以下特征，填入内部清单（不呈现给用户）：

| 特征 | 提取方法 | 取值示例 |
|------|---------|---------|
| 产出类型 | 文件扩展名 + 内容结构 | 技术方案、代码 PR、PPT、API 文档 |
| 产出规模 | 字数/WC | short (<2000)、medium (2000-8000)、long (>8000) |
| 结构化程度 | 是否有标题层级、表格、代码块 | high (有结构化章节) / low (散文式) |
| 领域信号 | 关键词扫描 | 微服务、金融、医疗、电信、AI |
| 受众 | 语气和术语级别 | 技术受众、管理受众、客户 |

#### Step 4: 呈现清单

输出一份用户可见的清单：

> 我识别到上文产出了以下内容：
>
> 1. **技术方案**：《XX 系统架构设计》— 约 5000 字，涉及微服务、数据库设计、安全模型
> 2. **代码实现**：3 个 Python 模块，约 800 行，核心是 API 层 + 数据访问层
>
> 请确认：这些就是需要审查的内容吗？有没有需要增删的？

用户确认后进入 Phase 2。

---

### Phase 2: 组建会审团

按以下固定流程生成审查角色。不得跳过任何步骤。

#### Step 1: 列出能力声称

对每个审查对象，提取它**声称自己做到了什么**——不是它实际写了什么，而是它在表达什么承诺。

> 这个技术方案声称：
> 1. 支持日均 100 万订单的并发处理
> 2. 实现了多机房容灾，RTO < 5 分钟
> 3. 通过了等保三级合规要求
> ...

呈现给用户确认："我理解这个方案在做这些承诺。有没有漏掉的关键承诺？"

#### Step 2: 生成攻击面

对每个能力声称，生成对应的攻击面。规则：

- 一个能力声称至少产生 1 个攻击面
- 攻击面 = 该声称被证伪的维度
- 从以下攻击面维度库中选择最匹配的，允许组合：

| 攻击面维度 | 含义 | 适用场景 |
|-----------|------|---------|
| 完整性 | 有没有遗漏的关键要素 | 方案、文档、代码 |
| 一致性 | 不同部分是否有矛盾 | 方案、文档、代码 |
| 安全性 | 是否存在被攻击/被绕过路径 | 代码、方案 |
| 可落地性 | 能否真的被执行 | 方案、PPT |
| 边界条件 | 极端情况下是否仍成立 | 方案、代码 |
| 逻辑闭合 | 推理链是否有跳跃 | 方案、标书 |
| 数据准确性 | 数字和事实是否可验证 | 标书、PPT |
| 受众适配 | 对目标受众是否恰当 | PPT、文档 |
| 成本合理性 | 投入产出比是否成立 | 方案、标书 |
| 竞争力 | 与替代方案/竞品比是否站得住 | 标书、方案 |
| 合规性 | 是否满足法规/行业标准 | 方案、标书、代码 |
| 可维护性 | 未来修改/扩展成本是否可控 | 代码、方案 |

#### Step 3: 去重 + 归组

合并高度重叠的攻击面，将相似攻击面归组为一个角色。归组原则：

- 同一攻击面维度、同一审查对象 → 合并
- 不同攻击面但审查的技能/知识重叠 → 可归为一个角色
- 攻击面过窄（只覆盖一两句话的内容）→ 合并到相邻攻击面

#### Step 4: 生成角色清单

为每个归组后的攻击面生成一个审查角色：

- **角色名**：一个具体的身份标签，暗示审查视角（如"分布式一致性审查"、"甲方合规审计"）
- **攻击面**：从 Step 2 选择的维度 + 具体描述
- **审查指令**：一句精确的可执行 prompt，包含攻击面维度和期望的攻击方向

#### Step 5: 数量控制

| 内容规模 | 建议角色数 | 模式 |
|---------|-----------|------|
| 短文（<2000字）/ 简单代码（<300行） | 2-3 个 | 快速模式 |
| 中等（2000-8000字）/ 中等项目 | 3-5 个 | 标准模式 |
| 长文（>8000字）/ 复杂系统 | 5-7 个 | 深度模式 |

**硬上限 7 个**。如果 Step 4 生成的角色超过 7 个，只保留攻击面最不重叠的 7 个（反模式 A2）。

#### Step 6: 呈现并确认

将角色清单呈现给用户（注意：不包含"严重度预期"——审查前无法预判）：

> 基于上述 [N] 个能力声称，我生成了 [M] 个攻击面，归组后组建以下会审团：
>
> | # | 角色 | 攻击面 |
> |---|------|--------|
> | 1 | 架构一致性审查 | 分布式事务边界、服务间契约是否自洽 |
> | 2 | 安全审计 | 认证链路、权限模型、数据加密是否留绕过路径 |
> | 3 | 运维可落地性 | 部署方案、回滚策略、监控覆盖是否真的可执行 |
> | 4 | 成本合理性 | 资源估算、弹性策略的投入产出是否成立 |
>
> 有没有需要增加、删除或调整的角色？也可以直接告诉我你想手动指定角色。

用户确认后进入 Phase 3。如果用户手动指定角色（覆盖配置项），跳过程序化生成，直接使用用户指定的角色和攻击面。

---

### Phase 3: 并行攻击

#### Step 1: 检查 Agent Team 可用性

尝试创建 agent team：

```
TeamCreate("review-panel-{timestamp}", "多维会审团")
```

如果 TeamCreate 成功，走团队模式（Step 2）；如果 TeamCreate 不可用或失败，跳过团队模式，直接走子 Agent 模式（Step 3）。

#### Step 2: 团队模式（优先）

```
1. 为每个审查角色 spawn 一个 agent：
   - 使用 Agent tool（subagent_type="general-purpose"）
   - name 设为角色名（如 "逻辑闭合审查"）
   - prompt 使用下方角色审查 prompt 模板（填入角色专属内容）
   - 所有 agent 并行 spawn，不需要等待上一个完成
2. 等待所有 teammate 返回结果
3. 收集各角色报告，按角色名归档
```

#### Step 3: 子 Agent 模式（退化）

如果 TeamCreate 不可用，使用 Agent tool 逐个 spawn 子 agent，顺序执行。每个子 agent 使用相同的角色 prompt 模板。

#### Step 4: 失败兜底

如果某个子 agent 超时（120s 无响应）或返回错误：

- 单个角色失败 → 在报告中标记该角色为"审查失败"，不阻塞其他角色
- 全部角色失败 → 主 agent 自行执行所有攻击面的审查（降级为单 agent 模式）
- 降级模式下，主 agent 依次以每个角色的身份审查，产出简化版报告，标注 `fallback_mode: true`

审查对象硬上限：总长度超过 32K tokens 或分段数超过 5 段时，拒绝全文审查，提示用户缩小审查范围。

```
你是会审团的 [角色名]。

你的攻击面：[攻击面——具体的一个维度，如"分布式事务边界是否在所有异常路径下都有处理"]

你要攻击的审查对象：
--- [审查内容：将产出内容的完整原文直接粘贴到这里] ---

审查要求：
1. 只审查你的攻击面，不要评论以下不属于你领域的内容：[列出其他角色的攻击面名称]
2. 只输出你有把握的弱点。不确定但可疑的发现也写出来，但标注"低置信度（confidence: low）"并附上你为什么觉得可疑（反模式 A7）
3. 每个攻击点必须标注严重度：
   - 致命：上线即灾难 / 方案推翻重来
   - 严重：有实际风险 / 必须修改
   - 一般：改进空间大 / 建议修改
   - 建议：锦上添花 / 可选修改
4. 每个攻击点包含：具体引用原文位置 + 问题是什么 + 为什么是问题 + 修正方向
5. 如果攻击面内没有发现任何弱点，直接输出"本攻击面无发现"，不要为了凑数而制造问题
6. 每发现一个问题，必须指出是违反了哪个原则/声称/最佳实践，不得无依据攻击
```

#### 审查对象交接协议

- 主 agent 必须**将原文内容直接粘贴**到每个 role prompt 中（`--- [审查内容] ---` 占位处）
- 如果原文超过 8000 tokens，分段贴入，每段附页码标记，确保每个 teammate 收到完整内容
- 不得依赖 teammate 自行搜索文件或通过读取工具获取——审查依赖于完整、一致的输入
- 所有 teammate 收到的是同一份审查对象，确保攻击面之间可对齐

#### Step 3: 子 Agent 模式（退化）

如果 TeamCreate 不可用，逐个 spawn 子 agent，顺序执行。每个子 agent 使用相同的角色 prompt 模板。

#### Step 4: 完成检查

所有角色返回结果后，检查：

- 有角色产出为空？→ 标记"本攻击面无发现"（是好消息，不是故障）
- 有角色产出仅含低置信度条目？→ 保留，但最终报告中标注"低置信度"
- 有角色明显跨界到其他角色的攻击面？→ 标记供 Phase 4 去重，跨界发现如果严重度 ≥ 严重则保留为独立条目
- 有角色产出异常短或格式不符？→ 如果是软违规，保留；如果是硬违规（如完全没遵循格式），标记重跑

---

### Phase 4: 汇总裁决

#### Step 1: 违反反模式检查

在去重之前，遍历所有角色产出，检查是否违反 Phase 开头定义的反模式。违反 A3（攻击面宽泛）的发现降一级严重度。违反 A4（跨界）的发现标注来源但移到对应攻击面的正确角色下。违反 A5（去重草率）→ 执行 Step 2。

#### Step 2: 去重

多个角色可能发现同一个问题。按以下规则合并：

- 同一位置、同一类问题 → 合并为一条，保留严重度更高的标注
- 同一问题、不同角度 → 合并为一条，列出所有角度的依据
- 表面相似但本质不同 → 分开保留

#### Step 3: 按严重度排序

```
致命问题
  1. ...
  2. ...

严重问题
  3. ...
  4. ...

一般问题
  5. ...

建议
  6. ...
```

如果用户设置了 `only_critical=true`，跳过"一般"和"建议"栏目，但保留栏目标题并注明"已省略"。

#### Step 4: 输出报告

```markdown
# 多维会审报告

## 审查概览

- 审查对象：XX 技术方案（5000 字） + 3 个 Python 模块（800 行）
- 会审团：[N] 个角色
- 发现总数：[M] 个（其中 [L] 个低置信度）
- 致命：[A] | 严重：[B] | 一般：[C] | 建议：[D]

## 攻击面发现

### 致命

**F1: [标题]**
- 审查角色：[角色名]
- 置信度：high / low
- 位置：[原文引用或章节/文件/行号]
- 问题：[描述]
- 根因：[为什么是问题——违反了什么原则/声称/最佳实践]
- 修正方向：[建议方案]

**F2: ...**

### 严重

**S1: ...**

### 一般

（如果 only_critical=true，此栏目省略）

### 建议

（如果 only_critical=true，此栏目省略）

## 各角色审查统计

| 角色 | 发现数 | 致命 | 严重 | 一般 | 建议 |
|------|--------|------|------|------|------|
| ... | ... | ... | ... | ... | ... |

## 未被攻击的维度

以下攻击面在会审中未发现明显问题：
- [维度]: [角色名] 审查后认为无明显弱点
- ...

## 结论

[一段话总结最关键的问题和整体评价]

---

如有任何角色的攻击面需要调整并重新审查，告诉我角色编号和新的攻击方向即可。
```

---

### Phase 5: 迭代重审

用户在看到报告后，可以对任意角色要求重审：

- 调整该角色的攻击面（收窄、放宽、换方向）
- 追加新角色
- 删除产出为空的角色

响应格式：

> 好的，我将对 [角色名] 以新的攻击面 [新攻击面] 重新审查。
> [执行 Phase 3 Step 2，仅重跑该角色]
> [将新结果插入原报告对应栏目，更新统计]

如果用户要求重审 ≥3 个角色，直接重新执行 Phase 3 全部角色（而非逐个重跑）。

## 配置项

以下配置可由用户在调用时覆盖。当多个配置同时指定并发生冲突时，优先级为：

```
手动指定角色 > 指定角色数 > 深度模式 > 仅致命+严重
```

即：用户手动指定角色时不执行 Phase 2 自动生成，指定的角色数设置被忽略。

| 配置 | 默认值 | 说明 |
|------|--------|------|
| 角色数 | 自动 | 手动指定角色数（2-7），仅当未手动指定角色时生效 |
| 深度 | 自动 | `fast` 快速模式（2-3 角色）/ `deep` 深度模式（5-7 角色） |
| 指定角色 | 无 | 用户手动指定审查角色和攻击面（跳过 Phase 2，优先级最高） |
| 仅致命+严重 | false | 设为 true 时只输出致命和严重问题，跳过一般和建议 |
| 审查对象 | 自动 | 用户显式指定要审查的文件/文本/会话片段（跳过 Phase 1 扫描） |
| 重审轮数 | 1 | 允许 Phase 5 最多重审几轮，防止无限循环 |
