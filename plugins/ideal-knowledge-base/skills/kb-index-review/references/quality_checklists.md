# 知识库质量检查清单

本文件包含知识库构建各阶段（P1-P5）的质量检查清单，用于系统性确保知识库质量。

---

## P1：原始材料分析（Material Analysis）

### 材料覆盖度

- [ ] 已识别并列出所有原始材料文件（代码、文档、配置、数据文件等）
- [ ] 已确认原始材料的版本/提交信息并记录
- [ ] 已识别材料中的核心文档（README、设计文档、API 文档等）
- [ ] 已识别关键源代码目录和入口文件
- [ ] 已识别配置文件（环境配置、构建配置、部署配置等）
- [ ] 已识别测试文件和测试覆盖范围
- [ ] 已识别 CI/CD 配置和自动化流程

### 分类识别

- [ ] 已正确判断项目所属分类（数据治理/模型训练/知识库与知识治理/智能体）
- [ ] 如涉及多分类，已识别主要分类和次要分类
- [ ] 已确认分类判断依据（基于项目核心功能和目标）

### 知识缺口识别

- [ ] 已列出原始材料中缺失的关键信息
- [ ] 已区分"信息缺失"和"信息未找到"（前者是材料中没有，后者是有但未定位到）
- [ ] 已为每个知识缺口标注严重程度（严重/一般/可忽略）
- [ ] 已列出需要向用户确认的问题清单

---

## P2：深度分析（Deep Analysis）

### 项目定位准确性

- [ ] 已准确理解项目的核心目标和价值主张
- [ ] 已识别目标用户/客户群体
- [ ] 已识别项目解决的核心痛点
- [ ] 已理解项目在整体技术方案中的定位和角色
- [ ] 已识别与同类方案的差异化特征

### 架构理解深度

- [ ] 已理解系统的整体架构分层
- [ ] 已识别核心组件及其职责
- [ ] 已理解组件间的交互关系和数据流向
- [ ] 已识别外部依赖和集成点
- [ ] 已理解部署架构和运行环境要求
- [ ] 已识别架构中的关键设计决策及其原因

### 模块完整性

- [ ] 已识别所有功能模块并列出模块层次结构
- [ ] 已理解每个模块的核心功能和职责边界
- [ ] 已识别模块间的依赖关系
- [ ] 已识别核心模块和辅助模块
- [ ] 已识别模块的优先级和成熟度

### 分类专项分析

- [ ] 已完成对应分类的专项分析（数据治理/模型训练/知识库与知识治理/智能体）
- [ ] 分类专项分析覆盖了该分类的关键技术维度
- [ ] 分类专项分析的深度达到了知识库标准（非浅层概述）

---

## P3：文档生成计划（Generation Planning）

### 文档选择

- [ ] 已根据项目分类正确选择需要生成的文档类型
- [ ] 已确认每份文档的生成必要性和优先级
- [ ] 已确认文档间的依赖关系和生成顺序

### 内容要点

- [ ] 每份文档已列出详细的内容要点清单
- [ ] 内容要点覆盖了 P2 分析中的所有关键发现
- [ ] 已为每个内容要点标注信息来源（来自哪个原始材料）
- [ ] 已识别需要特别关注的内容难点

### 任务拆分

- [ ] 已将文档生成任务合理拆分为可独立执行的子任务
- [ ] 已评估每个子任务的工作量和复杂度
- [ ] 已识别可以并行处理的任务
- [ ] 已制定合理的生成顺序（考虑依赖关系）

---

## P4：文档生成（Document Generation）

### 通用文档检查

以下检查适用于所有生成的知识库文档。

- [ ] 文档结构完整，包含模板要求的所有章节
- [ ] 无空章节（只有标题没有内容的章节）
- [ ] 无占位符文本（如"待填写"、"TODO"等）
- [ ] 内容基于原始材料分析，非主观臆断
- [ ] 术语使用一致，同一概念全文统一用词
- [ ] 表格用于展示列表型信息，格式规范
- [ ] Mermaid 图表语法正确，可正常渲染
- [ ] 代码示例准确，可运行或接近可运行状态

### 概述.md 专项检查

- [ ] 产品名称、版本、分类信息完整准确
- [ ] 产品定位描述清晰（一句话能说清"这是什么"）
- [ ] 目标客户/用户群体已明确列出
- [ ] 核心痛点已具体描述（非泛泛而谈）
- [ ] 核心价值主张已清晰阐述
- [ ] 与竞品/同类方案的对比分析有依据
- [ ] 内容简洁但信息密度高

### 技术架构.md 专项检查

- [ ] 包含完整的系统架构图（Mermaid 格式）
- [ ] 架构图层次清晰，标注了各层/各组件名称
- [ ] 技术栈表格完整（框架、语言、中间件、数据库等）
- [ ] 部署架构已描述（单机/分布式/云原生等）
- [ ] 外部依赖和集成点已列出
- [ ] 架构设计的关键决策和权衡已说明
- [ ] 从宏观到微观逐层展开，有层次感

### 功能清单.md 专项检查

- [ ] 功能模块层次结构清晰（模块 > 子模块 > 功能点）
- [ ] 每个模块有优先级标注（核心/重要/辅助）
- [ ] 每个模块有清晰的职责描述
- [ ] 功能点列表覆盖全面，无遗漏
- [ ] 模块间依赖关系已标注
- [ ] 模块状态已标注（已实现/开发中/规划中）

### 部署与运维.md 专项检查

- [ ] 环境要求已列出（硬件、软件、网络等）
- [ ] 部署步骤清晰，可按步骤操作
- [ ] 部署架构图已包含（如有分布式部署）
- [ ] 监控和告警方案已描述
- [ ] 扩缩容策略已说明
- [ ] 常见故障和处理方法已列出
- [ ] 备份和恢复方案已描述（如适用）

### 数据流.md 专项检查

- [ ] 包含端到端的数据流图（Mermaid 格式）
- [ ] 数据输入源已完整列出
- [ ] 数据处理流程已逐步描述
- [ ] 数据输出和消费方已说明
- [ ] 数据存储方案已描述（存储类型、存储位置、保留策略）
- [ ] 数据质量保障措施已说明
- [ ] 数据格式和Schema信息已描述（如适用）

### 数据质量.md 专项检查（数据治理分类）

- [ ] 数据质量维度已全面覆盖（准确性、完整性、一致性、时效性、唯一性等）
- [ ] 质量规则已具体列出（规则名称、规则逻辑、阈值）
- [ ] 质量检测流程已描述
- [ ] 质量问题的发现、告警、处理流程已说明

### 训练管线.md 专项检查（模型训练分类）

## 常见错误（Common Mistakes）

### 内容层面

- [ ] **未避免：将 README 的营销性语言直接搬入知识库** - 知识库应客观描述，而非复制宣传文案
- [ ] **未避免：只有组件列表，没有组件间关系描述** - 架构描述应包含组件交互和数据流
- [ ] **未避免：性能指标只说"提升显著"而无具体数据** - 应给出具体数值和对比基准
- [ ] **未避免：忽略能力边界和限制条件** - 应坦诚说明系统不能做什么
- [ ] **未避免：不同文档对同一信息的描述矛盾** - 应保持全局信息一致性
- [ ] **未避免：技术细节文档变成代码注释搬运** - 应提炼为可理解的技术说明

### 格式层面

- [ ] **未避免：Mermaid 图表语法错误导致无法渲染** - 生成后应验证图表可渲染
- [ ] **未避免：表格列对齐不一致** - 使用统一的 Markdown 表格格式
- [ ] **未避免：章节标题层级跳跃** - 应按 h1 > h2 > h3 顺序逐层递进
- [ ] **未避免：代码块未指定语言类型** - 应标注 ```python、```yaml 等语言标识

### 流程层面

- [ ] **未避免：跳过 P1-P3 直接生成文档** - 缺乏分析基础的文档质量不可控
- [ ] **未避免：P2 分析深度不足就开始生成** - 深度分析是文档质量的基础
- [ ] **未避免：P5 审查流于形式** - 每个检查项都应认真核实
- [ ] **未避免：发现问题不修正就标记通过** - 严重问题必须修正才能通过
- [ ] **未避免：索引更新遗漏** - 分类索引和主索引必须同步更新
