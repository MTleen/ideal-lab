# P5：索引更新与质量审查（Index Update & Quality Review）

## 概述

P5 阶段是知识库构建工作流的最终阶段，负责两项关键任务：

1. **索引更新**：将新构建的知识库项目条目更新到分类索引（`索引.md`）和主索引（`项目索引.md`），维护索引体系的完整性和一致性。
2. **质量审查**：对已生成的全部知识库文档进行系统性质量检查，确保内容完整、准确、深入、可读、可维护。

本阶段是知识库交付前的最后把关环节，所有发现的问题必须在本阶段修正后方可完成交付。

## 输入（Input）

| 输入项 | 说明 |
|--------|------|
| 知识库文档 | P4 阶段生成的全部知识库文档（`知识库/<项目名称>/` 下的所有 .md 文件） |
| 分类索引 | 当前分类的 `索引.md` 文件 |
| 主索引 | 仓库根目录的 `项目索引.md` 文件 |
| P1 分析报告 | P1 阶段的原始材料分析报告（用于核对信息覆盖度） |
| P2 分析报告 | P2 阶段的深度分析报告（用于核对技术准确性） |
| P3 生成计划 | P3 阶段的文档生成计划（用于核对文档完整性） |
| 质量检查清单 | `references/quality_checklists.md` 中的检查清单 |

## 输出（Output）

| 输出项 | 说明 |
|--------|------|
| 更新后的分类索引 | 更新了新项目条目的 `索引.md` |
| 更新后的主索引 | 更新了新项目条目的 `项目索引.md` |
| 质量审查报告 | 系统性的质量检查结果，包含通过/未通过项及修正建议 |
| 最终知识库 | 修正后达到质量标准的完整知识库 |

## 第一部分：索引更新（Index Update）

### 1.1 更新分类索引（索引.md）

在对应分类的 `索引.md` 文件中添加新项目条目，使用以下标准格式：

```markdown
### 项目名称
- **录入日期**：YYYY-MM-DD
- **录入人**：
- **原始材料来源**：（项目仓库路径 / 文件位置）
- **知识库位置**：`知识库/项目名称/`
- **内容概要**：
  -
- **关键词**：,,,
- **关联输出**：
  - [ ] 销售一页纸
  - [ ] 技术文档
  - [ ] 案例素材
  - [ ] 汇报材料
```

**更新规则：**

1. 条目按录入日期倒序排列（最新条目在最前）
2. "知识库位置" 使用相对于分类目录的路径
3. "内容概要" 用 2-3 个要点概括项目核心内容
4. "关键词" 选取 3-5 个最能代表项目特征的关键词
5. 如果分类索引不存在，则创建并添加标准的分类标题和说明

### 1.2 更新主索引（项目索引.md）

在仓库根目录的 `项目索引.md` 中添加跨分类的项目条目。主索引作为全仓库的统一入口，条目格式如下：

```markdown
### 项目名称
- **所属分类**：数据治理 / 模型训练 / 知识库与知识治理 / 智能体
- **录入日期**：YYYY-MM-DD
- **知识库位置**：`<分类>/知识库/项目名称/`
- **内容概要**：（1-2 句话概述）
- **关键词**：,,,
- **关联输出**：
  - [ ] 销售一页纸
  - [ ] 技术文档
  - [ ] 案例素材
  - [ ] 汇报材料
```

**更新规则：**

1. 主索引按分类分组，分类内按录入日期倒序排列
2. 如果项目涉及多个分类（如同时涉及数据治理和模型训练），在主要分类下列出，其他分类中以引用方式注明
3. 检查主索引中的分类分组是否与实际目录结构一致

### 1.3 更新日志

每次索引更新时，在主索引文件末尾维护更新日志：

```markdown
---
## 更新日志

| 日期 | 操作 | 项目 | 操作人 |
|------|------|------|--------|
| YYYY-MM-DD | 新增知识库 | 项目名称 | |
```

## 第二部分：质量审查（Quality Review）

### 2.1 完整性检查（Completeness）

检查知识库文档是否完整覆盖项目信息：

| 检查项 | 说明 |
|--------|------|
| 文档齐全性 | 对照 P3 生成计划，确认所有规划文档均已生成 |
| 模板遵循性 | 每份文档是否包含模板要求的所有章节 |
| 必填字段完整性 | 产品名称、版本、分类、技术栈等关键字段是否已填写 |
| 章节无空缺 | 是否存在只有标题无内容的空章节 |
| 图表完备性 | 需要架构图、流程图的文档是否已包含 Mermaid 图表 |

### 2.2 准确性检查（Accuracy）

检查内容是否准确反映项目实际情况：

| 检查项 | 说明 |
|--------|------|
| 信息可追溯 | 关键信息是否可追溯到原始材料（代码、文档、配置） |
| 技术描述正确 | 技术架构、技术栈、算法描述是否与代码实际一致 |
| 版本信息准确 | 产品版本、依赖版本等信息是否正确 |
| 数据一致性 | 同一信息在不同文档中的描述是否一致 |
| 无主观臆断 | 是否存在没有依据的推测性描述 |

### 2.3 深度检查（Depth）

检查分析的深度是否达到知识库标准：

| 检查项 | 说明 |
|--------|------|
| 架构描述深度 | 是否从宏观到微观逐层描述，而非仅列出组件名 |
| 算法说明深度 | 关键算法是否有原理说明、输入输出、复杂度分析 |
| 数据流完整度 | 是否覆盖了从输入到输出的完整链路 |
| 性能数据具体 | 性能指标是否有具体数值、基准、对比 |
| 边界条件覆盖 | 是否说明了能力边界、限制条件、异常处理 |

### 2.4 可读性检查（Readability）

检查文档的格式和表达是否便于阅读：

| 检查项 | 说明 |
|--------|------|
| 结构清晰度 | 章节层次是否合理，逻辑是否通顺 |
| 术语一致性 | 同一概念是否使用统一术语 |
| 图表质量 | Mermaid 图表是否正确渲染、标注是否清晰 |
| 表格使用 | 列表型信息是否使用表格呈现 |
| 语言风格 | 是否客观专业，避免营销性或模糊性表述 |

### 2.5 可维护性检查（Maintainability）

检查知识库是否便于后续更新和维护：

| 检查项 | 说明 |
|--------|------|
| 信息来源标注 | 关键信息是否标注了来源（代码文件、配置路径等） |
| 版本关联 | 是否记录了对应的原始材料版本/提交信息 |
| 可更新性 | 当项目有更新时，是否能方便地定位和更新对应内容 |
| 模块化程度 | 各文档之间是否有合理的职责边界，避免过度交叉引用 |

## 质量审查报告输出格式

完成质量审查后，输出如下格式的报告：

```markdown
# 知识库质量审查报告

## 基本信息
- **项目名称**：
- **所属分类**：
- **审查日期**：YYYY-MM-DD
- **审查范围**：P1-P5 全阶段

## 审查结果汇总

| 检查维度 | 通过项 | 未通过项 | 通过率 |
|----------|--------|----------|--------|
| P1 原始材料分析 | -/- | -/- | -% |
| P2 深度分析 | -/- | -/- | -% |
| P3 生成计划 | -/- | -/- | -% |
| P4 文档生成 | -/- | -/- | -% |
| P5 索引与审查 | -/- | -/- | -% |
| **总计** | **-/-** | **-/-** | **-%** |

## 未通过项详情

### [严重] 问题标题
- **所属文档**：
- **检查项**：
- **问题描述**：
- **修正建议**：

### [一般] 问题标题
- **所属文档**：
- **检查项**：
- **问题描述**：
- **修正建议**：

## 优化建议

1. ...
2. ...

## 审查结论

- [ ] **通过** - 所有检查项通过，可交付
- [ ] **条件通过** - 存在一般问题，修正后可交付
- [ ] **不通过** - 存在严重问题，需返工后重新审查
```

## 执行流程

```
开始 P5
  │
  ├─ 1. 加载输入文件
  │    ├─ 读取 P1 分析报告
  │    ├─ 读取 P2 分析报告
  │    ├─ 读取 P3 生成计划
  │    └─ 读取已生成的知识库文档
  │
  ├─ 2. 索引更新
  │    ├─ 更新分类索引（索引.md）
  │    ├─ 更新主索引（项目索引.md）
  │    └─ 记录更新日志
  │
  ├─ 3. 质量审查
  │    ├─ 完整性检查
  │    ├─ 准确性检查
  │    ├─ 深度检查
  │    ├─ 可读性检查
  │    └─ 可维护性检查
  │
  ├─ 4. 生成质量审查报告
  │
  ├─ 5. 问题修正（如有未通过项）
  │    ├─ 修正严重问题（必须）
  │    └─ 修正一般问题（建议）
  │
  └─ 6. 最终确认，完成交付
```

## 注意事项

1. **质量标准不可妥协**：严重问题必须修正后才能通过审查，不能带着已知严重问题交付。
2. **索引一致性**：分类索引和主索引的信息必须保持一致，任何更新需同步到两处。
3. **可追溯性**：质量审查报告中引用的每个问题都应指向具体的文档和章节。
4. **审查独立性**：质量审查应独立于文档生成过程，避免"自己审查自己"的偏差。
5. **持续改进**：每次审查中发现的问题应反馈到检查清单的持续完善中。
