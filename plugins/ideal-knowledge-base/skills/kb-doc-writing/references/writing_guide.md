# 知识库文档写作指南

本指南为知识库各类文档的标准化写作提供模式、规范与示例参考。

---

## 一、写作模式（Writing Patterns）

### 1.1 概述模式（Product Positioning Pattern）

用于描述产品定位、目标客户与核心价值。

**标准模板：**

```
[产品名称] 是面向 [目标客户群] 的 [产品类别] 产品/平台/系统，定位于 [一句话定位]。

核心解决 [客户群] 在 [业务场景] 中面临的 [痛点1]、[痛点2]、[痛点3] 等问题。

与 [传统方案/竞品] 相比，[产品名称] 的核心差异化在于 [差异化点1]、[差异化点2]。

当前版本为 [版本号]，主要功能覆盖 [功能范围]。
```

**示例：**

> 智能数据治理平台是面向电信运营商的企业级数据治理平台，定位于提供从数据采集到数据应用的全链路治理能力。核心解决数据部门在海量数据处理中面临的数据质量参差不齐、治理流程依赖人工、数据资产难以追溯等问题。与传统数据治理工具相比，智能数据治理平台的核心差异化在于内置行业数据质量规则库、支持自然语言驱动的数据质量检测、以及端到端的血缘追踪能力。当前版本为 V2.1，主要功能覆盖数据探查、质量检测、血缘分析、资产目录与治理工作流编排。

---

### 1.2 架构描述模式（Layered Architecture Pattern）

用于分层描述系统架构，强调各层职责与层间关系。

**标准模板：**

```
系统采用 [N] 层架构设计，自下而上依次为：

**第1层：[层名称]**
- 职责：[该层的核心职责]
- 核心组件：[组件1]、[组件2]
- 技术选型：[技术/框架]

**第2层：[层名称]**
- 职责：[该层的核心职责]
- 核心组件：[组件1]、[组件2]
- 技术选型：[技术/框架]

...（逐层展开）

**层间交互**
- [层A] 通过 [接口/协议] 向 [层B] 提供 [能力]
- [层C] 依赖 [层B] 的 [接口] 实现 [功能]
```

**示例：**

> 系统采用四层架构设计，自下而上依次为：
>
> **第1层：数据接入层**
> - 职责：对接多源异构数据，完成数据采集与标准化接入
> - 核心组件：Kafka 集群、Flink CDC、REST 数据适配器
> - 技术选型：Apache Kafka 3.x、Flink 1.17
>
> **第2层：数据处理层**
> - 职责：执行数据清洗、转换、质量检测与血缘采集
> - 核心组件：Spark ETL 引擎、规则引擎、血缘采集器
> - 技术选型：Apache Spark 3.4、Drools 规则引擎
>
> **第3层：知识管理层**
> - 职责：构建知识图谱、向量化索引与知识资产目录
> - 核心组件：Neo4j 图数据库、Milvus 向量库、Elasticsearch
> - 技术选型：Neo4j 5.x、Milvus 2.3
>
> **第4层：应用服务层**
> - 职责：对外提供 API、可视化界面与智能问答服务
> - 核心组件：API Gateway、前端门户、RAG 问答引擎
> - 技术选型：Spring Cloud Gateway、React 18

### 1.3 功能描述模式（Feature Module Pattern）

用于结构化描述功能模块及其能力点。

**标准模板：**

```
**[模块编号] [模块名称]**（优先级：[高/中/低]）

模块概述：[一句话描述模块做什么、解决什么问题]

核心功能点：
1. **[功能点1名称]**：[功能描述]，支持 [能力细节]
2. **[功能点2名称]**：[功能描述]，支持 [能力细节]
3. **[功能点3名称]**：[功能描述]，支持 [能力细节]

依赖关系：
- 上游依赖：[依赖的模块/系统/数据]
- 下游被依赖：[使用该模块的模块/系统]

约束与限制：
- [约束条件1]
- [约束条件2]
```

**示例：**

> **M03 数据质量检测**（优先级：高）
>
> 模块概述：自动化执行数据质量规则检测，生成质量报告并触发告警。
>
> 核心功能点：
> 1. **规则配置**：支持通过可视化界面或 YAML 文件配置质量规则，涵盖完整性、一致性、时效性、准确性、唯一性五大维度
> 2. **批量检测**：支持定时批量检测与手动触发，单次可处理亿级行数据，检测结果写入质量报告表
> 3. **异常告警**：质量分低于阈值时自动触发告警，支持钉钉、邮件、Webhook 三种通知渠道
>
> 依赖关系：
> - 上游依赖：M01 数据探查（元数据信息）、M02 数据接入（数据源连接）
> - 下游被依赖：M05 治理工作流（质量门禁）、M08 可视化看板（质量趋势展示）
>
> 约束与限制：
> - 单条规则执行超时上限为 30 分钟
> - 复杂跨表规则不支持实时检测，仅支持 T+1 批量模式

### 1.4 数据流描述模式（Data Flow Pattern）

用于描述数据从输入到输出的端到端流转过程。

**标准模板：**

```
**数据流名称：** [流程名称]

**数据源：**
- [数据源1]：[数据类型]、[更新频率]、[数据量级]
- [数据源2]：[数据类型]、[更新频率]、[数据量级]

**处理流程：**
1. [阶段1名称]：[处理动作描述]，输入 [输入数据] → 输出 [输出数据]
2. [阶段2名称]：[处理动作描述]，输入 [输入数据] → 输出 [输出数据]
3. [阶段3名称]：[处理动作描述]，输入 [输入数据] → 输出 [输出数据]

**数据产出：**
- [产出1]：[格式]、[存储位置]、[消费方]

**数据存储：**
- [存储层1]：[存储技术]、[存储内容]、[保留策略]
```

**示例：**

> **数据流名称：** 客户画像数据构建流程
>
> **数据源：**
> - CRM 系统：结构化客户信息，每日全量同步，约 500 万条/日
> - 计费系统：通话与流量详单，实时流接入，峰值约 2000 万条/小时
> - 投诉工单：工单记录，T+1 批量导入，约 3 万条/日
>
> **处理流程：**
> 1. 数据采集：通过 Kafka Connect 从各源系统拉取数据，统一写入 ODS 层（Raw 格式）
> 2. 数据清洗：Spark ETL 执行去重、空值填充、格式标准化，输出到 DWD 层（明细格式）
> 3. 特征工程：基于 Flink 实时计算通话频次、流量偏好等特征，输出到 DWS 层（汇总格式）
> 4. 标签生成：规则引擎 + ML 模型联合生成客户标签，写入标签库（HBase）
>
> **数据产出：**
> - 客户标签库：Parquet 格式，存储于 HDFS /dws/customer_tags/，供营销系统与客服系统消费
> - 客户画像报告：JSON 格式，通过 API 实时查询，供前端门户展示
>
> **数据存储：**
> - ODS 层：Hive 外部表，原始数据保留 90 天
> - DWD/DWS 层：Hive 内部表 + Iceberg 表，增量数据保留 1 年

### 1.5 对比模式（Comparison Pattern）

用于对比本方案与传统方案，突出差异化优势。

**标准模板：**

```
| 对比维度 | [传统方案] | [本方案] |
|---------|-----------|---------|
| [维度1] | [描述] | [描述] |
| [维度2] | [描述] | [描述] |
| [维度3] | [描述] | [描述] |

**关键差异分析：**
- [差异点1]：传统方案 [局限性]，本方案 [改进方式]，带来 [具体收益]
- [差异点2]：传统方案 [局限性]，本方案 [改进方式]，带来 [具体收益]
```

**示例：**

> | 对比维度 | 传统规则引擎方案 | 智能质量检测方案 |
> |---------|----------------|----------------|
> | 规则配置方式 | 人工编写 SQL/脚本，需开发人员参与 | 自然语言描述 + 可视化配置，业务人员自助完成 |
> | 规则覆盖率 | 依赖经验积累，约覆盖 60% 质量场景 | 内置 200+ 行业规则模板，覆盖 90%+ 常见场景 |
> | 异常发现时效 | T+1 批量检测，次日才能发现问题 | 支持准实时检测（分钟级），异常 15 分钟内告警 |
> | 根因定位 | 需人工逐层排查，平均 2-4 小时 | 自动血缘追溯 + 根因推荐，平均 15 分钟定位 |
>
> **关键差异分析：**
> - 规则配置方式：传统方案高度依赖开发人员，规则维护成本高；本方案通过 NLU 理解业务意图自动生成规则，业务人员可独立完成配置，开发介入量降低 80%
> - 异常发现时效：传统方案的 T+1 模式无法满足实时业务需求；本方案基于 Flink 流式处理，可在数据入库后 15 分钟内完成质量检测并告警，大幅缩短问题发现窗口

### 1.6 量化证据模式（Quantitative Evidence Pattern）

用于以具体数据支撑成效描述。

**标准模板：**

```
**[成效维度]**
- 指标：[具体指标名称]
- 改进前：[基线数值]（来源：[数据来源说明]）
- 改进后：[当前数值]（来源：[数据来源说明]）
- 提升幅度：[百分比/倍数]
- 计算方式：[简要说明计算口径]
```

**示例：**

> **数据质量检测效率**
> - 指标：单次全量质量检测耗时
> - 改进前：约 6 小时（来源：旧系统运行日志，2024年Q1平均值）
> - 改进后：约 45 分钟（来源：新系统生产环境监控，2024年Q4平均值）
> - 提升幅度：降低 87.5%
> - 计算方式：从定时任务触发到质量报告生成的端到端耗时

## 二、语言规范

### 2.1 语言选择

- **中文为主**：所有描述性文本、标题、说明均使用中文
- **英文用于技术术语**：产品名、框架名、协议名、命令等保留英文原名
  - 正确示例：使用 Apache Kafka 进行消息传输
  - 错误示例：使用阿帕奇卡夫卡进行消息传输
- **首次出现时附注说明**：对于非通用缩写，首次出现时给出全称
  - 示例：基于 RAG（Retrieval-Augmented Generation，检索增强生成）架构...

### 2.2 语气与风格

- **客观陈述**：描述事实与能力，避免主观评价
  - 正确示例：系统支持最大 10 亿条记录的数据处理
  - 错误示例：系统拥有强大的数据处理能力，令人惊叹
- **避免营销语言**：不使用"业界领先"、"独一无二"、"革命性"等修饰词
- **准确量化**：尽可能使用具体数值替代模糊描述
  - 正确示例：支持 50+ 种数据源类型
  - 错误示例：支持丰富的数据源类型

### 2.3 不确定信息的处理

当原始材料中信息不完整或不确定时，按以下规范处理：

1. **使用 `[待补充]` 标记**，并在其后用括号说明缺失原因
   - 示例：系统日均处理数据量约为 [待补充：原始材料中未提供具体数据量指标]
2. **区分"未提供"与"不存在"**：
   - 未提供：`[待补充：原始材料中未提及]`
   - 明确不存在：直接说明"该系统不支持 XXX 功能"
3. **基于上下文的合理推断**：若可根据已有信息合理推断，可写推断结果但标注
   - 示例：预计支持并发用户数约 500（[待确认：基于单节点 4C8G 配置推算，未经压测验证]）

### 2.4 信息来源标注

对于关键数据与事实，标注信息来源以便后续核查：

```
[来源：文件路径/页面位置/沟通记录]
```

- 示例：
  - `系统采用 Kubernetes 部署，集群规模为 5 个节点`[来源：部署文档 deploy/README.md 第2节]
  - `当前日活用户约 3000 人`[来源：2024年Q4运营报告 P12]
- 对于综合多个来源的信息：
  - `综合 [来源A] 与 [来源B] 信息整理`
- 来源不明确但可信度高：
  - `[来源：代码仓库 main 分支，2024-12 最新提交]`

## 三、各文档写作要点

### 3.1 概述.md

- **核心目标**：让读者在 2 分钟内理解"这是什么、给谁用、解决什么问题"
- **写作要点**：
  - 开篇用概述模式一句话定位产品
  - 目标客户与痛点要具体，避免泛化描述
  - 核心价值用 3-5 个要点概括，每个要点一句话
  - 版本信息与功能范围要准确
  - 竞品对比如信息不足，可仅列本产品特性，不强行对比
- **常见问题**：写成营销文案而非客观描述；功能范围与功能清单不一致

### 3.2 技术架构.md

- **核心目标**：让技术人员理解系统的整体结构与关键技术选型
- **写作要点**：
  - 使用架构描述模式分层说明
  - 必须包含 Mermaid 架构图（参考 mermaid_guide.md）
  - 技术选型表需注明具体版本号
  - 部署架构与逻辑架构分开描述
  - 外部依赖与集成点要完整列出
- **常见问题**：架构图过于简略；技术选型缺少版本号；忽略外部系统依赖

### 3.3 功能清单.md

- **核心目标**：提供完整的功能模块清单，便于需求对齐与工作量评估
- **写作要点**：
  - 使用功能描述模式逐模块展开
  - 模块编号保持层级一致（M01, M01-01, M01-01-01）
  - 每个模块标注优先级（高/中/低）
  - 依赖关系要双向标注（上游 + 下游）
  - 功能点描述要可验证（避免"提供良好体验"这类不可验证的描述）
- **常见问题**：功能粒度不一致（有的太粗有的太细）；缺少模块间依赖关系

### 3.4 部署与运维.md

- **核心目标**：提供完整的部署方案与运维指南
- **写作要点**：
  - 环境要求要具体（CPU、内存、磁盘、网络、OS 版本）
  - 部署步骤要可执行，包含完整的命令或配置文件
  - 监控指标与告警阈值要明确
  - 扩缩容策略要量化（何时扩容、如何扩容）
  - 故障处理要有分级（P0/P1/P2）和响应时效要求
- **常见问题**：部署步骤缺少前置条件；监控指标不完整；缺少回滚方案

### 3.5 数据流.md

- **核心目标**：让读者理解数据的完整生命周期
- **写作要点**：
  - 使用数据流描述模式逐流程展开
  - 每个数据流必须包含 Mermaid 数据流图
  - 数据量级与频率要量化
  - 数据格式与存储位置要明确
  - 异常数据处理策略要说明（丢弃/重试/死信队列）
- **常见问题**：数据流图缺少存储节点；数据格式描述不清；忽略异常场景

### 3.6 数据质量.md（数据治理分类专属）

- **核心目标**：说明数据质量管理的方法论、规则体系与执行机制
- **写作要点**：
  - 质量维度要完整（完整性、一致性、准确性、时效性、唯一性、合法性）
  - 规则分类体系要清晰（字段级/表级/跨表级）
  - 质量评分算法要说明计算口径
  - 质量问题处理流程要闭环（发现→定级→派单→修复→验证）
  - 内置规则模板要列出分类与数量
- **常见问题**：缺少质量评分定义；规则体系不够结构化；忽略质量问题闭环流程

### 3.7 训练管线.md（模型训练分类专属）

- **核心目标**：描述模型训练的完整 Pipeline
- **写作要点**：
  - 训练流程要分阶段说明（数据准备→特征工程→模型训练→评估→部署）
  - 每个阶段的输入/输出要明确
  - 训练配置参数要列出关键项（学习率、批大小、迭代轮次等）
  - 数据集划分策略要说明（训练/验证/测试比例）
  - 训练资源消耗要量化（GPU 类型、训练时长）
- **常见问题**：训练配置参数不完整；缺少数据集描述；训练时长信息缺失

### 3.8 模型评估.md（模型训练分类专属）

- **核心目标**：说明模型评估体系与评估结果
- **写作要点**：
  - 评估指标要与任务类型匹配（分类/回归/生成/排序）
  - 基线模型与改进模型要对比展示
  - 评估数据集构成要说明（来源、规模、分布）
  - 消融实验结果如有，需完整列出
  - 评估结果的置信区间或统计显著性要注明
- **常见问题**：评估指标与任务不匹配；缺少基线对比；评估数据集描述不足

### 3.9 知识模型.md（知识库与知识治理分类专属）

- **核心目标**：描述知识图谱的模型设计、本体定义与知识组织方式
- **写作要点**：
  - 本体定义要列出核心实体类型、关系类型、属性定义
  - 知识建模方法论要说明（自顶向下/自底向上/混合）
  - 实体与关系的规模要量化
  - 知识更新机制要说明（增量更新/全量重建策略）
  - 知识质量保障措施要列出
- **常见问题**：本体定义过于抽象缺少示例；实体规模信息缺失；忽略知识更新策略

### 3.10 检索与生成.md（知识库与知识治理分类专属）

- **核心目标**：描述 RAG 系统的检索策略、生成策略与优化手段
- **写作要点**：

### 3.11 Agent设计.md（智能体分类专属）

- **核心目标**：描述 Agent 的架构设计、决策机制与工具编排
- **写作要点**：
  - Agent 角色定义要清晰（职责、能力边界、约束条件）
  - 决策流程要可追溯（输入→推理→行动→观察→循环/终止）
  - 工具集要完整列出，每个工具的输入/输出/异常处理要说明
  - 多 Agent 协作模式如有，要描述通信协议与任务分配机制
  - 安全与权限控制要说明
- **常见问题**：Agent 行为边界不清晰；工具集描述不完整；忽略异常处理与回退策略

### 3.12 工具与集成.md（智能体分类专属）

- **核心目标**：描述 Agent 可调用的工具集与外部系统集成方案

  - 工具调用的限流与熔断策略要描述

- **常见问题**：工具接口描述不规范；缺少错误处理说明；忽略调用频率限制

### 3.13 应用成效.md
