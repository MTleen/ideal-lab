# KB Material Scope (P1)

## Overview

扫描项目原始素材全貌，评估材料覆盖度，识别知识缺口，确认项目分类和知识库文档清单。

## 输入

- 原始材料路径（本地路径或 GitHub 仓库）
- 项目名称（可选，默认从仓库名推断）
- 分类（可选，默认从材料推断）

## 输出

`{分类}/知识库/{项目名称}/P1-材料清单.md`

## 执行步骤

### 1. 扫描目录结构

识别以下关键文件类型：

| 文件类型 | 典型路径 | 价值 |
|----------|----------|------|
| 项目说明 | README.md, README.rst | 项目定位、功能概述 |
| 源码目录 | src/, lib/, app/ | 核心实现 |
| 文档目录 | docs/, doc/ | 设计文档、API文档 |
| 配置文件 | pyproject.toml, package.json, requirements.txt | 技术栈、依赖 |
| 容器配置 | Dockerfile, docker-compose.yml | 部署方式 |
| 测试文件 | tests/, test/ | 使用示例、边界条件 |
| CI/CD | .github/workflows/, .gitlab-ci.yml | 自动化流程 |
| 数据文件 | data/, datasets/ | 数据结构和样本 |
| 许可证 | LICENSE | 开源状态 |

### 2. GitHub 仓库元信息（如适用）

若原始材料为 GitHub 仓库，获取：
- 语言统计（Languages）
- 最近活跃度（Recent commits, Last updated）
- Stars/Forks（社区活跃度）
- Topics/Tags（项目标签）

### 3. 评估材料覆盖度

对以下维度评估覆盖情况（✅充分 / ⚠️部分 / ❌缺失）：

| 维度 | 说明 | 评估 |
|------|------|------|
| 项目定位 | 产品是什么、为谁服务、解决什么问题 | |
| 技术架构 | 系统设计、技术选型、模块关系 | |
| 功能实现 | 核心功能、算法、数据处理逻辑 | |
| 部署运维 | 部署方式、环境要求、监控 | |
| 应用成效 | 实际案例、性能数据、客户反馈 | |
| 分类专属 | 取决于分类（见下） | |

**分类专属覆盖维度：**

| 分类 | 专属维度 |
|------|----------|
| 数据治理 | 数据流链路、质量规则、治理策略 |
| 模型训练 | 训练流程、模型架构、评估体系 |
| 知识库与知识治理 | 知识结构、RAG管线、检索策略 |
| 智能体 | Agent架构、决策逻辑、工具链 |

### 4. 确认项目分类

根据材料内容判断所属分类：

| 分类 | 典型特征 |
|------|----------|
| 数据治理 | 数据清洗、标注、质量管控、ETL、数据目录 |
| 模型训练 | 训练脚本、微调配置、评估指标、模型导出 |
| 知识库与知识治理 | 向量数据库、RAG、知识图谱、分块策略 |
| 智能体 | Agent框架、工具调用、Prompt、对话管理 |

若项目跨多个分类，选择**主要分类**并在材料清单中标注次要分类。

### 5. 识别知识缺口

列出原始材料无法覆盖的知识维度，按重要性排序：

```markdown
### 知识缺口
1. **[高]** 竞品对比信息 — 无竞品相关文档
2. **[中]** 性能基准测试数据 — 仅有理论分析，无实测数据
3. **[低]** 客户反馈 — 无客户使用反馈记录
```
