# 演讲备注撰写指南

> 本指南定义了幻灯片配套演讲备注的撰写规范。与渲染模式无关——适用于 HTML、Image 及任何渲染方式。

## 1. 文件规范

### 1.1 输出文件

- **路径**：`notes/total.md`
- **编码**：UTF-8
- **格式**：Markdown

### 1.2 文件结构

```markdown
# 01 - 页面标题

演讲脚本内容...

**要点：**
① 核心要点一
② 核心要点二
③ 核心要点三

**时长：** 30s

---

# 02 - 页面标题

[Transition] 过渡语内容...

演讲脚本内容...

**要点：**
① 核心要点一
② 核心要点二

**时长：** 45s

[Pause]

---
```

## 2. 格式规范

### 2.1 页面标题

```markdown
# NN - 页面标题
```

- 使用一级标题 `#`
- 格式：序号 + 空格 + 横杠 + 空格 + 页面标题
- 序号与 prompt 文件名一致（如 `01`, `02`, ...）

### 2.2 页面分隔

- 页面之间使用 `---` 分隔线
- 分隔线前后各空一行

### 2.3 过渡语（Transition）

```markdown
[Transition] 刚才我们了解了核心功能，接下来看看实际应用效果。
```

- 除第一页外，每页开头添加 `[Transition]` 标记
- 过渡语需承上启下，自然连接上下页内容
- 长度：1-2 句话

### 2.4 停顿标记（Pause）

```markdown
[Pause]
```

- 在关键内容之后标记 `[Pause]`
- 暗示演讲者在此处稍作停顿，让听众消化信息
- 不是每页都需要，仅在重点页面的关键处使用

## 3. 内容撰写原则

### 3.1 演讲脚本

每页脚本遵循以下原则：

1. **叙事优先**：用连贯的段落讲述，不要写成分点概括。演讲者脑中是"故事线"而不是"要点清单"
2. **长度按需分配**：根据 Step 2 的时长计算确定字数，不要用固定的"50-150 字"一刀切。60 分钟演讲的内容页通常需要 200-400 字
3. **源素材驱动**：从原始素材中提取论点展开、设问、类比、数据解释、案例细节，转化为口语表达
4. **不唯念文字**：幻灯片文字是锚点，不是剧本。关键数据和核心结论可以念出以强调，但不要逐字复述整段内容——用自己的话补充解释、背景和洞察

**关键数据**：念出来。"Token日使用量突破140万亿——这个数字意味着什么？意味着两年时间增长了1400倍。"
**核心结论**：念出来。"云是AI落地的第一性基础设施——不是'之一'，是'唯一'。"
**描述性段落**：不念，用自己的话讲。
**列表要点**：不逐条念，归纳后讲整体含义。

好的示例：
> 这个架构图展示了我们系统的三层设计。最上层是用户接入层，负责处理所有的客户端请求。中间是业务逻辑层，这里包含了核心的处理引擎。最下面是数据持久层，保证数据的安全存储和高可用。

不好的示例：
> 系统架构分为三层。第一层是用户接入层。第二层是业务逻辑层。第三层是数据持久层。

### 3.2 要点提炼

```markdown
**要点：**
① 第一个核心要点
② 第二个核心要点
③ 第三个核心要点
```

- 使用 ①②③ 编号
- 每个要点一句话（10-25 字）
- 数量：2-5 个（推荐 3 个）
- 必须是该页最核心的信息
- 要点是"备忘速查"，不是演讲稿本身——详细内容在上方的叙事段落中

### 3.3 时长预估

```markdown
**时长：** 30s
```

- 格式：`**时长：** NNs`
- 常见时长参考：
  - 封面/目录：10-15s
  - 过渡页：10-15s
  - 内容页：30-60s
  - 复杂页面（数据/架构）：45-90s
  - 总结页：30-45s

## 4. 语调与风格

### 4.1 语调

- **自信专业**：展现对内容的掌控感
- **自然流畅**：像在跟听众交流，不是背稿
- **积极正面**：用肯定的语气传递信息

### 4.2 用词建议

| 场景 | 推荐 | 避免 |
|------|------|------|
| 引出话题 | "让我们来看看..." | "下面将介绍..." |
| 强调重点 | "关键在于..." | "需要注意的是..." |
| 总结归纳 | "核心要点是..." | "综上所述..." |
| 过渡衔接 | "接下来..." | "然后..." |
| 数据展示 | "数字背后体现的是..." | "数据显示..." |

### 4.3 节奏控制

- 开场页面：简短有力，建立期待
- 内容页面：信息密度适中，有重点
- 高潮页面：适当展开，制造记忆点
- 收尾页面：简洁总结，留下印象

## 5. 展开策略

当备注需要更多篇幅时（如长演讲、核心章节），使用以下手段展开：

| 策略 | 说明 | 示例 |
|------|------|------|
| 设问引导 | 用提问制造悬念，再自己回答 | "为什么运营商必须转型？答案在两条增长曲线的交汇处。" |
| 数据解释 | 不仅报数字，还解释"这意味着什么" | "1400倍增长——不是一条斜线，而是一条接近垂直的曲线。" |
| 类比展开 | 用听众熟悉的事物类比陌生概念 | "Token之于AI，就像'度'之于电力、'字节'之于互联网。" |
| 场景还原 | 描述一个具体的困境或使用场景 | "我在不同场合问过这个问题。得到最多的回答不是'模型不够好'，而是——'基础设施跟不上'。" |
| 观众互动 | 直接向听众提问 | "在座的各位，你们所在的企业，AI已经落地了吗？" |
| 原文引用 | 引用源素材中的金句或关键表述 | "有一句话说得好：'算法的突破让人看到了AI的可能性，但基础设施的完备程度，才决定了AI能不能从可能变成现实。'" |

**使用原则**：每页至少使用 1-2 种展开策略，核心数据页和章节高潮页使用 2-3 种。

## 6. 完整示例（60分钟演讲风格）

以下示例展示的是 40 页 / 60 分钟演讲中典型内容页的备注风格。

```markdown
# 01 - 封面

大家好，今天分享的主题是"云端技术怎样助力人工智能高效落地"。这个主题很大，但我想用一条完整的链路把它串起来——从最底层的算力，一直到最上层的应用。听完今天的分享，你会明白为什么云不是AI的"选项之一"，而是"唯一选项"。

**要点：**
① 主题：云端技术助力AI落地
② 核心线索：从算力到应用的完整链路
③ 核心论点：云是AI的第一性基础设施

**时长：** 30s

---

# 02 - Token取代流量

[Transition] 在展开之前，我想先问一个问题：2026年，中国电信为什么把"Token"放在了和"流量"同等重要的位置？

2026年，中国电信正式提出一项战略转型——"从流量经营到Token经营"。这不是一句口号，它写进了年度战略报告，配了预算，设了专项团队。一家拥有数亿用户的国家级运营商，把一个大多数人还没听过的东西放在了战略核心。为什么？

答案只有一句：因为在AI时代，Token就是新的"流量"。回想互联网时代，运营商的核心指标是什么？用户数、ARPU值、流量消耗量。流量每增长一倍，背后是更多用户上网、更多应用运行、更多商业价值产生。到了AI时代，这个角色由Token来扮演。Token是AI模型处理信息的基本单位，每一次AI对话、每一次内容生成、每一次代码编写，都在消耗Token。Token消耗量直接反映了AI的活跃程度——就像流量反映了互联网的活跃程度一样。

**要点：**
① 中国电信战略转型：流量经营 → Token经营
② Token是AI时代的"流量"
③ Token消耗量 = AI活跃程度

**时长：** 120s

---

# 03 - 1400倍增长

[Transition] 说Token是新的"流量"，不是修辞——让我用一组数据让大家感受一下。

看这组数据：2024年初，中国日Token使用量大约1000亿。到2025年下半年，企业级大模型日均Token消耗已达37万亿。到2026年3月，日Token使用量突破140万亿。两年时间，1400倍增长。

1400倍意味着什么？它意味着Token的增长曲线不是一条斜线，而是一条接近垂直的曲线。同期任何其他数字经济指标——用户数、流量、数据量——都没有这样的增速。140万亿Token，大约相当于每天处理70万亿个汉字的信息量。Token使用量的爆发，就是AI产业化的爆发。

[Pause]

**要点：**
① 两年1400倍：1000亿 → 140万亿
② 增速远超任何其他数字经济指标
③ Token爆发 = AI产业化爆发

**时长：** 90s

---

# 04 - 核心链路

[Transition] 数据很震撼，但这里有一个反直觉的判断。

AI技术本身已经不是瓶颈。大模型的能力已经足够强大——能理解、能推理、能生成、能调用工具。真正卡住AI从实验室走向产业的，是另一条链路：云→算力→模型→Token→应用。

这条链路的意思是：算力要够，模型才能跑；模型跑了，Token才能产；Token流动了，应用才能活；而算力、模型、Token的承载和调度，都离不开云。任何一环断裂，AI就无法规模化落地。算法的突破让人看到了AI的可能性，但基础设施的完备程度，才决定了AI能不能从"可能"变成"现实"。

我在不同场合问过这个问题：你们企业AI落地了吗？得到最多的回答不是"模型不够好"，不是"算法不够强"，而是——"基础设施跟不上"。算力不够、部署太慢、数据散落各处、安全合规没保障。有AI愿景的企业很多，能让AI稳定、规模化运行的企业很少。这不是个例，而是共性困境。

**要点：**
① AI技术不是瓶颈，基础设施才是
② 五环链路：云→算力→模型→Token→应用
③ 共性困境：基础设施跟不上

**时长：** 120s

---

# 05 - 第一章过渡

[Transition] 要理解这条链路，我们首先回到最底层——算力。

第一章，算力——AI的"发动机"为什么只能在云端。

**时长：** 15s

---
```

上述示例中，内容页（02/03/04）的备注长度为 200-350 字，对应 90-120 秒演讲时长，支撑 60 分钟演讲的总字数需求。注意叙事风格：先设问制造悬念，再用数据或类比展开，最后落脚到核心论点。要点部分保持精简，作为演讲时的快速提醒。
