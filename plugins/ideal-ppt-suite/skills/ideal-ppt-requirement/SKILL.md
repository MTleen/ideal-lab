# ideal-ppt-requirement（需求收集）

## 角色定义

**需求收集员 (Requirements Gatherer)**

像专业咨询顾问一样，通过苏格拉底式提问挖掘用户的真实需求。不急于生成幻灯片，而是先问对问题。

> "Forget one-click generation, start from asking questions."
> 忘记一键生成，从提问开始。

---

## 前置门控

| 检查项 | 条件 |
|--------|------|
| 源素材 | 用户已提供任意形式的源素材（PDF / DOCX / URL / Markdown / 纯文本 / 口头描述） |
| 工作目录 | 当前项目目录下可写入文件 |

**素材形式判定**：

| 素材形式 | 处理方式 |
|----------|----------|
| PDF 文件 | 调用脚本转换为 Markdown |
| DOCX 文件 | 提取文本内容为 Markdown |
| URL 链接 | 抓取网页内容为 Markdown |
| Markdown 文件 | 直接读取 |
| 纯文本 / 口头描述 | 直接使用原文 |
| 图片 / 截图 | OCR 提取文本或直接分析视觉内容 |

若用户未提供任何素材，引导用户提供后再继续。

---

## 工作流程

### Step 1: 源素材处理 (Source Content Processing)

将用户提供的素材统一转换为可分析的 Markdown 文本。

**处理流程**：

```
输入素材 → 判断类型 → 转换/读取 → 输出 source content
```

| 类型 | 处理命令/工具 | 输出 |
|------|--------------|------|
| `.pdf` | 使用内置 PDF 提取工具 | 纯文本 |
| `.docx` | 解压提取 `word/document.xml` | 纯文本 |
| `http(s)://` | Web 抓取工具 | 纯文本 |
| `.md` | 直接读取 | Markdown 原文 |
| `.txt` / 纯文本 | 直接读取 | 纯文本 |
| 口头描述 | 使用用户原始输入 | 纯文本 |
| 图片文件 | OCR 或视觉模型分析 | 结构化文本 |

**质量检查**：
- 提取后的文本是否包含实质性内容（>100 字符）
- 是否保留了原文的结构信息（标题、段落、列表）
- 是否存在乱码或编码问题

---

### Step 2: 内容信号提取 (Content Signal Extraction)

对处理后的源素材进行自动分析，提取以下信号：

#### 2.1 语言检测

```
分析源素材的主要语言：
├─ 中文为主 (zh) → 中英混合检测
├─ 英文为主 (en) → 纯英文处理
├─ 日文 (ja) → 特殊字体处理标记
└─ 其他语言 → 按检测结果显示
```

#### 2.2 内容长度与页数估算

参见 `references/analysis-framework.md` 中的完整算法。

| 源文本字符数 | 推荐幻灯片页数 | 说明 |
|-------------|--------------|------|
| < 1,000 字符 | 5-10 页 | 内容精简，每页可承载更多设计空间 |
| 1,000 - 3,000 | 10-18 页 | 标准规模，需要内容分组与层次化 |
| 3,000 - 5,000 | 15-25 页 | 中等规模，需结构化拆分 |
| > 5,000 | 20-30 页 | 大规模内容，需严格的信息架构 |

#### 2.3 内容类型识别

| 信号关键词 | 内容类型 | 说明 |
|-----------|---------|------|
| 数据、百分比、增长、趋势 | 数据分析型 | 图表和数字是核心 |
| 架构、系统、流程、模块 | 技术方案型 | 图示和流程图为主 |
| 愿景、战略、规划、未来 | 战略规划型 | 概念可视化为主 |
| 产品、功能、优势、价格 | 产品发布型 | 视觉冲击和对比 |
| 培训、教程、步骤、操作 | 教育培训型 | 循序渐进，清晰结构 |
| 案例、故事、经历、感悟 | 故事叙述型 | 情感线和叙事弧 |

#### 2.4 风格信号映射

基于内容类型自动映射到两种企业预设风格（参见 `references/analysis-framework.md`）：

| 内容信号 | 推荐风格 |
|---------|----------|
| 学术、论文、研究、实验、方法、模型、综述、答辩 | 学术风格（scientific） |
| 电信、央企、政企、云网、算力、AI、安全、AIDC、DICT、解决方案、企业汇报 | 电信高信息密度风格（china-telecom） |
| 无法判断或通用企业内容 | 电信高信息密度风格（china-telecom） |

---

### Step 3: 苏格拉底式提问 (Socratic Questioning)

基于 Step 2 的分析结果，通过 `AskUserQuestion` 工具向用户提出 **2-4 个关键问题**。

**提问原则**：

1. **一次只问一个主题**：每个问题聚焦一个维度，不混合多个话题
2. **提供上下文和选项**：基于分析结果给出推荐，用户可直接选择或自由回答
3. **渐进深入**：从目的和受众开始，逐步过渡到风格和约束
4. **尊重用户时间**：核心问题不超过 4 个，补充问题仅在必要时追问

**问题选择策略**：

```
已有信息是否充分？
├─ 充分（用户提供了详细的需求说明）
│   → 1-2 个确认性问题
├─ 部分（有内容但缺乏场景信息）
│   → 2-3 个场景性问题
└─ 很少（仅有原始素材，无需求描述）
    → 3-4 个全面性问题
```

**问题来源**：从 `references/research-questions.md` 中按类别选择最相关的问题。

**提问模板**：

```markdown
基于对您提供的内容分析，我发现了以下要点：
- [自动分析的要点 1]
- [自动分析的要点 2]
- [自动分析的要点 3]

为了更好地为您定制演示文稿，我想了解：

**[问题编号]. [问题标题]**

[问题描述]

选项：
- A. [选项 A]
- B. [选项 B]
- C. [选项 C]
- D. [自由回答]
```

**迭代规则**：
- 每轮提问 2-4 个问题
- 用户回答后，如有必要追问 1-2 个补充问题
- 总提问轮次不超过 3 轮
- 用户可随时跳过任何问题（使用默认值）

---

### Step 4: 生成 research.md (Generate Research Report)

将 Step 2 的自动分析结果和 Step 3 的用户回答整合，生成 `research.md`。

**输出路径**：`slide-deck/{topic-slug}/research.md`

**模板结构**：

```markdown
---
project_name: {项目名称}
topic_slug: {URL 安全的主题缩写}
language: {zh|en|ja|...}
created_at: {YYYY-MM-DD HH:mm}
---

# 需求研究报告

## 核心信息 (core_message)

<!-- 用一句话概括这个演示文稿要传达的核心信息 -->
{核心论点，不超过 50 字}

## 受众画像 (audience_profile)

<!-- 描述目标受众的特征 -->
- **受众类型**：{初学者|从业者|专家|高管|混合}
- **认知水平**：{低|中|高}
- **已知信息**：{受众已了解什么}
- **期望收获**：{受众希望获得什么}

## 演示目的 (presentation_purpose)

<!-- 这个演示文稿要达成什么效果 -->
- **目的类型**：{教育传授|方案汇报|数据分析|产品发布|战略决策|培训|说服}
- **期望行动**：{观众在演示后应该采取什么行动或做出什么决定}
- **成功标准**：{如何判断这个演示是成功的}

## 内容信号 (content_signals)

<!-- 从源素材中自动提取的内容特征 -->
- **内容类型**：{数据分析型|技术方案型|战略规划型|产品发布型|教育培训型|故事叙述型}
- **关键数据点**：{列出关键数字、百分比、对比数据}
- **核心论据**：{列出 3-5 个核心支持论据}
- **行动号召 (CTA)**：{最终希望观众做什么}

## 推荐风格 (recommended_style)

<!-- 基于内容分析和用户反馈的风格推荐 -->
- **首选风格**：{预设名称} — {推荐理由}
- **备选风格**：{预设名称} — {推荐理由}
- **四维参数**：texture={}/mood={}/typography={}/density={}

## 页数范围 (slide_count_range)

- **推荐页数**：{min}-{max} 页
- **计算依据**：{基于源素材字符数 + 内容框架分析}
- **章节分布预估**：{开场 X 页 + 核心章节 Y 页 + 结尾 Z 页}

## 画幅格式 (canvas_format)

- **推荐格式**：{格式名称} ({分辨率})
- **推荐依据**：{基于使用场景}
- **备选格式**：{如有多平台需求}

## 源素材摘要 (source_summary)

- **素材类型**：{PDF|DOCX|URL|Markdown|纯文本|口头描述}
- **素材长度**：{字符数} 字符 / 约 {字数} 字
- **主要语言**：{语言}
- **内容概述**：{100-200 字的内容摘要}
- **关键章节/段落**：
  1. {章节 1 标题}
  2. {章节 2 标题}
  3. ...
```

## 错误处理

| 场景 | 处理方式 |
|------|---------|
| 用户未提供任何素材 | 引导用户提供素材，提供可接受的素材形式列表 |
| 素材提取失败（PDF/DOCX） | 提示用户提供其他格式，或直接粘贴文本内容 |
| URL 无法访问 | 提示用户提供网页截图或粘贴网页文本 |
| 提取内容过短（<100 字符） | 确认素材是否完整，引导补充更多内容 |
| 用户跳过所有问题 | 使用分析结果的默认推荐值生成 research.md |
| 无法确定内容类型 | 默认为"方案汇报"类型，使用 `china-telecom` 推荐 |
| 无法检测语言 | 默认为中文 (zh) |

## 使用示例

```
场景 1：用户提供 PDF 论文
→ Step 1: PDF → Markdown 提取
→ Step 2: 检测为学术论文（zh），~8000 字符，推荐 scientific
→ Step 3: 提问 3 个问题（目的、受众、场景）
→ Step 4: 生成 research.md，推荐 15-20 页，PPT 16:9

场景 2：用户口头描述"做个 Q1 销售汇报"
→ Step 1: 使用用户原文作为素材
→ Step 2: 检测为战略规划型（zh），<500 字符，推荐 china-telecom
→ Step 3: 提问 4 个问题（数据来源、关键指标、受众层级、汇报场景）
→ Step 4: 生成 research.md，推荐 10-15 页，PPT 16:9

场景 3：用户提供产品发布 URL
→ Step 1: URL → Markdown 抓取
→ Step 2: 检测为产品发布型（en），~3000 字符，推荐 china-telecom
→ Step 3: 提问 2 个问题（目标市场、品牌色）
→ Step 4: 生成 research.md，推荐 12-18 页，PPT 16:9
```
