# Eight Confirmations Protocol — 八大确认详细协议

本文件定义八大确认的完整协议，包括呈现信息、决策树、推荐逻辑和默认值。

---

## 总则

1. **分阶段路由呈现**：八大确认按依赖关系分 4 阶段呈现，**不可越阶段**
   - 阶段 1（基础 3 问）：a 画幅 + b 页数 + c 受众与目的 — 一次性呈现
   - 阶段 2（路由点）：d 美业方式 — **单独询问，是后续问题的路由开关**
   - 阶段 3（风格 4 问）：e 配色 + f 图标 + g 字体 — 一次性呈现
   - 阶段 4（条件问）：h 图片形式 — **仅在 d=html-mode 时才问，d=image-mode 时跳过**
2. **推荐优先**：每项必须附带基于 research.md 的推荐值
3. **用户修改**：用户可接受全部推荐、部分修改或全部推翻
4. **确认闭环**：用户确认后生成摘要表格，再次确认后进入生成阶段
5. **阶段间允许简短澄清**：进入下一阶段前可基于上一阶段答案做 1-2 个澄清追问（例如 d 选了 image-mode 后问"是否提供风格参考图"）

### 路由示例

```
阶段 1 → 阶段 2(单独) → 阶段 3(批量)
                   ↓
              d = image-mode？
                   ├─ 是 → 跳过阶段 4，直接生成 design-spec
                   └─ 否 → 继续阶段 4(h 图片形式)
```

## 确认 a: 画幅格式 (Canvas Format)

### 呈现信息

| 格式名称 | 分辨率 (px) | 比例 | 典型场景 |
|----------|-------------|------|----------|
| PPT 16:9 | 1920×1080 | 16:9 | 会议演示、投影、屏幕共享 |
| PPT 4:3 | 1024×768 | 4:3 | 传统投影仪、兼容旧设备 |
| XHS (小红书) | 1080×1440 | 3:4 | 小红书图文笔记 |
| Story (故事) | 1080×1920 | 9:16 | Instagram Stories、微信视频号 |
| Square (方形) | 1080×1080 | 1:1 | 微信朋友圈、Instagram 帖子 |
| WeChat Header (微信头图) | 900×383 | 2.35:1 | 微信公众号文章头图 |

### 决策树

```
内容的使用场景是什么？
├─ 会议/报告/汇报
│   ├─ 设备兼容性要求？
│   │   ├─ 有旧设备 → PPT 4:3
│   │   └─ 无特殊要求 → PPT 16:9  ★默认
│   └─ 远程/线上 → PPT 16:9
├─ 社交媒体发布
│   ├─ 小红书 → XHS (1080×1440)
│   ├─ 朋友圈/Instagram → Square (1080×1080)
│   ├─ Stories/短视频 → Story (1080×1920)
│   └─ 微信公众号 → WeChat Header (900×383)
└─ 多平台复用
    ├─ 主版本 → PPT 16:9
    └─ 适配版本 → 根据目标平台分别生成
```

### 推荐逻辑

1. 检查 research.md 中的"使用场景"字段
2. 匹配场景到决策树
3. 无明确场景时，默认推荐 **PPT 16:9**
4. 如涉及多平台，推荐主版本 + 适配版本

### 默认值

**PPT 16:9 (1920×1080)**

## 确认 b: 页数范围 (Page Count Range)

### 呈现信息

向用户展示以下信息：
- 推荐的页数范围 `[min, max]`
- 页数计算依据
- 各部分的页数分配预估

### 页数计算模型

```
总页数 = 开场 + 目录/过渡页 + 核心内容 + 结尾

开场 = 1~2 页（封面 + 可选的引入页）
目录/过渡页 = 1 页（目录）+ 章节过渡页 × (章节数 - 1)
核心内容 = Σ(每章节要点数 × 每要点页数)
结尾 = 1~2 页（总结 + 可选的感谢页/联系方式）

每要点页数参考：
├─ 纯文字要点 → 0.5~1 页（可合并多个要点）
├─ 数据/图表要点 → 1~2 页
├─ 案例分析要点 → 1~2 页
└─ 流程/架构要点 → 1~3 页
```

### 推荐逻辑

1. 从 research.md 提取内容框架和章节结构
2. 按计算模型估算每章页数
3. 汇总得到范围
4. 附上页数分配明细

### 默认值

根据内容框架动态计算，无固定默认值。如无法计算，提供 `8-15 页` 作为通用参考。

## 确认 c: 目标受众与目的 (Target Audience & Purpose)

### 呈现信息

1. **受众画像**
   - 初学者 (Beginner)：需要概念引导、降低认知门槛
   - 从业者 (Practitioner)：关注实操、行业最佳实践
   - 专家 (Expert)：关注深度分析、前沿洞察
   - 高管 (Executive)：关注决策信息、关键结论
   - 混合 (Mixed)：需分层设计，渐进式信息呈现

2. **演示目的**
   - 教育传授 (Education)：知识传递、技能培养
   - 方案汇报 (Proposal Review)：方案展示、评审汇报
   - 数据分析 (Data Analysis)：数据呈现、趋势分析
   - 产品发布 (Product Launch)：新品展示、市场推广
   - 战略决策 (Strategic Decision)：决策支持、战略分析
   - 培训 (Training)：技能培训、流程讲解
   - 说服 (Persuasion)：观点说服、行动号召

3. **认知负荷等级**
   - 低：受众需引导，控制信息密度
   - 中：受众可自主消化信息
   - 高：受众期待深度、详尽的信息

### 推荐逻辑

1. 从 research.md 提取受众描述和使用目的
2. 映射到上述分类
3. 根据受众画像确定认知负荷等级：
   - 初学者 → 低
   - 从业者 → 中
   - 专家 → 高
   - 高管 → 中（关注结论，不需要所有细节）
   - 混合 → 中

### 默认值

- 受众：**从业者 (Practitioner)**
- 目的：根据 research.md 推断
- 认知负荷：**中**

## 确认 d: 美业方式 + 风格目标 (Render Mode + Style)

> **⚠️ 路由点**：本项单独询问，是阶段 4 (h 图片形式) 是否触发的开关。

### 呈现信息

#### 第一步：美业方式（先问这个）

| 模式 | 说明 | 每页 PPT 的本质 | 后续 |
|------|------|----------------|------|
| `image-mode` | 每页由 AI 生成一张完整大图，图就是页 | 图片即页面 | **跳过 h**，直接进入设计 |
| `html-mode` | 每页用 HTML 排版（SVG/占位符/排版），图只是页内元素 | 排版即页面 | **继续问 h** |

#### 第二步：风格目标（基于美业方式推荐）

只提供两种预设风格：

| 预设 | 中文名 | 典型场景 |
|------|--------|----------|
| scientific | 学术风格 | 论文汇报、课题答辩、研究综述、实验结果、算法/方法报告 |
| china-telecom | 电信高信息密度风格 | 中国电信、央企政企、云网算力、AI、安全、AIDC、DICT、行业解决方案 |

### 风格选择决策树

```
内容是否明显是学术/研究/论文/实验？
├─ 是 → scientific
└─ 否 → china-telecom

是否提供电信 PPT 背景图/母版/参考图？
├─ 是 → style_reference.type=image，作为背景/母版/版式锚点
└─ 否 → style_reference.type=none 或 description
```

### 推荐逻辑

1. **美业方式推荐**：
   - 内容以视觉/图为主（产品发布、品牌宣传、故事叙述）→ `image-mode`
   - 内容以数据/方案/结构为主（论文、方案汇报、培训、战略）→ `html-mode`
2. 从 research.md 提取内容类型和使用场景
3. 按决策树选择 scientific 或 china-telecom
4. 结合受众画像调整密度（dense / ultra-dense）
5. 推荐 1 个主预设，说明为什么不是另一个预设
6. 若有参考图，记录参考图用途和禁止复制项

### image-mode 下的特殊处理

- 每页 = 一张 AI 大图，无需询问"图片形式"（h 跳过）
- **配色 e 和字体 g 仍正常询问**，其值将注入每页 AI 图像的 prompt 作为视觉约束（色彩基调、文字风格留白区、字体关键词等）
- **图标 f 仍需问**（用于页内可叠加的小标签、章节标识），但 image-mode 下简化为 Binary 选择（Built-in Library / None），无需展开 AI-Generated / Emoji 选项

### 默认值

**美业方式：`html-mode`**
**风格：电信高信息密度风格 (china-telecom)**

四维参数：`clean + professional + geometric + ultra-dense`

## 确认 e: 配色方案 (Color Scheme)

### 呈现信息

1. 推荐配色方案的完整参数：
   - 主色 (Primary)：HEX 值 + 使用说明
   - 辅助色 (Secondary)：HEX 值 + 使用说明
   - 强调色 (Accent)：HEX 值 + 使用说明
   - 渐变定义（如适用）
2. 60-30-10 比例说明
3. 中性色层级
4. WCAG 2.1 AA 对比度合规性说明

### 配色决策树

```
内容属于什么行业/领域？
├─ 金融/银行 → Navy Blue 系 (#005587)
├─ 科技/互联网 → Bright Blue 系 (#2196F3)
├─ 医疗/健康 → Teal Green 系 (#009688)
├─ 政府/公共 → Red 系 (#D32F2F)
├─ 教育/培训 → Blue-Green 系 (#00897B)
├─ 咨询/专业服务 → Consulting Blue 系 (#0076A8)
├─ 消费/零售 → Orange/Warm 系 (#FF9800)
└─ 创意/媒体 → Purple/Vibrant 系 (#9C27B0)

颜色角色分配：
Primary (60%) → 行业主色的浅色/中性变体
Secondary (30%) → 行业主色
Accent (10%) → 互补色或对比色
```

### 推荐逻辑

1. 检查 research.md 中的行业/领域信息
2. 匹配到行业色彩映射（参见 `color-knowledge-base.md`）
3. 基于风格预设调整明暗
4. 生成 60-30-10 配色方案
5. 验证 WCAG 2.1 AA 对比度

### 默认值

```
Primary:   #F5F7FA (浅灰蓝背景)
Secondary: #005587 (深蓝主色)
Accent:    #FF9800 (橙色强调)
Text:      #212121
Sub-text:  #757575
Divider:   #E0E0E0
```

## 确认 f: 图标使用 (Icon Usage)

### 呈现信息

| 选项 | 视觉风格 | 优点 | 缺点 | 适用场景 |
|------|----------|------|------|----------|
| Emoji | 色彩丰富、风格统一 | 零成本、跨平台 | 不够专业、风格受限 | 轻松非正式、内部培训 |
| AI-Generated | 高度定制 | 完全匹配主题 | 风格一致性风险 | 品牌定制、独特需求 |
| Built-in Library | 线条简洁、专业 | 一致性好、快速 | 通用性强不够独特 | 商务演示、专业汇报 |
| None | 无图标 | 纯净视觉 | 可能单调 | 极简设计、纯数据展示 |

### 决策树

```
风格预设是什么？
├─ scientific → None 或 Built-in Library（图标只作小标签）
└─ china-telecom → Built-in Library（线性图标，小尺寸，严禁图标替代内容）
```

### 推荐逻辑

1. 基于风格预设推荐图标类型
2. 商务/咨询场景默认 Built-in Library
3. 创意/教育场景可考虑 Emoji 或 AI-Generated

### 默认值

**Built-in Library**

## 确认 g: 字体方案 (Typography Plan)

### 呈现信息

完整的字体方案，包含：

| 用途 | 中文字体 | 英文字体 | 字号层级 |
|------|----------|----------|----------|
| 大标题 (Hero) | 推荐 | 推荐 | 48-60px |
| 章节标题 (H1) | 推荐 | 推荐 | 32-40px |
| 小标题 (H2) | 推荐 | 推荐 | 24-28px |
| 正文 (Body) | 推荐 | 推荐 | 16-18px |
| 辅助文字 (Caption) | 推荐 | 推荐 | 12-14px |

### 字号层级体系

**基于 24px 基准（标准密度）**：

```
Hero:    48-60px (2.0x-2.5x)
H1:      32-40px (1.3x-1.7x)
H2:      24-28px (1.0x-1.2x)
Body:    16-18px (0.67x-0.75x)
Caption: 12-14px (0.5x-0.58x)
```

**基于 18px 基准（紧凑密度）**：

```
Hero:    36-44px (2.0x-2.4x)
H1:      24-30px (1.3x-1.7x)
H2:      18-22px (1.0x-1.2x)
Body:    14-16px (0.78x-0.89x)
Caption: 10-12px (0.56x-0.67x)
```

### 推荐逻辑

1. 基于风格预设的四维参数中的 Typography 维度
2. 映射到具体字体推荐（参见 `design-guidelines.md`）
3. 确保中英文字体风格协调
4. 基于密度维度确定字号基准

### 默认值

```
中文标题: MiSans Bold
中文正文: MiSans Regular
英文标题: Liter Bold
英文正文: HedvigLettersSans Regular
基准字号: 24px
```

## 确认 h: 图片使用 (Image Usage) — 仅 html-mode

> **⚠️ 条件性确认**：仅当阶段 2 的 d = `html-mode` 时才询问本项。
> - d = `image-mode` → 每页即一张图，无"图片形式"概念，**直接跳过本项**
> - d = `html-mode` → HTML 排版中可能嵌入图片元素，按下表选

### 呈现信息

| 选项 | 说明 | 成本 | 质量 | 适用场景 |
|------|------|------|------|----------|
| None | 不使用图片，纯图形/色块设计 | 无 | — | 数据展示、极简风格 |
| User-provided | 用户提供图片素材 | 无 | 取决于用户 | 有品牌素材、产品图 |
| AI-generated | 通过 AI 生成配图 | API 调用 | 高度可控 | 无素材、需定制视觉 |
| Placeholders | 使用占位图，后续替换 | 无 | — | 模板设计、快速出稿 |

### 决策树

```
d 是什么模式？
├─ image-mode → 【跳过本项，不询问】
└─ html-mode ↓

    内容是否需要配图？
    ├─ 不需要（纯数据/文字为主）→ None
    └─ 需要 ↓
        用户是否有现成素材？
        ├─ 有 → User-provided（询问素材路径）
        └─ 无 ↓
            是否需要高质量定制配图？
            ├─ 是 → AI-generated
            └─ 否 → Placeholders
```

### 推荐逻辑

1. **先判断 d**：d = image-mode 时**不要问这一题**（这是修复后的强制规则）
2. 仅在 d = html-mode 下：
   - 数据密集型 → 推荐 None 或最少图片
   - 品牌推广型 → 推荐 User-provided 或 AI-generated
   - 通用型 → 推荐 Placeholders

### 默认值

**Placeholders**（仅在 html-mode 下生效）
