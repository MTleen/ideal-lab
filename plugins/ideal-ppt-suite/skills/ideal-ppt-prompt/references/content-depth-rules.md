# Content Depth Rules — 反抽象内容规则

本文档定义提示词内容的具体化标准。所有 SVG 生成提示词必须通过这些规则检查后才能输出。

**核心原则**：提示词中禁止出现任何抽象标签、未具体化的内容、模糊描述。SVG 生成模型收到提示词后，应无需任何推理即可直接输出完整的 SVG 代码。

---

## Rule 1: 列表必须包含每项的具体内容

**规则**：列表不得只包含标签或类别名，每项必须附带具体的描述、数据或说明。

| 类型 | 示例 |
|------|------|
| FAIL | "三个核心功能" |
| FAIL | "功能列表：搜索、推荐、分析" |
| PASS | "3 个功能卡片横排：卡片 1 '智能搜索'（#2196F3 蓝色图标，支持自然语言查询，响应时间 < 200ms），卡片 2 '个性推荐'（#4CAF50 绿色图标，基于协同过滤算法，推荐准确率 92%），卡片 3 '数据分析'（#FF9800 橙色图标，实时仪表盘，支持 20+ 图表类型）" |

**检查方式**：搜索提示词中的列表，验证每个列表项是否有至少一个具体细节（数据、描述、特征）。

---

## Rule 2: 机制/流程必须有具体步骤

**规则**：涉及流程、机制、架构、管线等过程性内容时，必须列出每一步的具体名称和关键动作。

| 类型 | 示例 |
|------|------|
| FAIL | "端到端的数据处理流程" |
| FAIL | "从数据采集到模型部署的完整管线" |
| PASS | "4 步水平流程：Step 1 '数据接入'（Kafka + API Gateway，日均处理 5TB 原始数据）→ Step 2 '特征工程'（Spark 集群，200+ 特征维度自动提取）→ Step 3 '模型训练'（GPU 集群，XGBoost + Neural Net 双模型并行）→ Step 4 '在线服务'（Kubernetes 部署，P99 延迟 < 50ms）" |

**检查方式**：搜索提示词中的流程/机制描述，验证是否包含具体步骤名、工具/方法名、量化指标。

---

## Rule 3: 数据必须有实际数值和单位

**规则**：任何涉及数据的描述，必须给出具体数值和单位。禁止使用"显著增长"、"大幅提升"等模糊量词。

| 类型 | 示例 |
|------|------|
| FAIL | "市场规模持续扩大" |
| FAIL | "用户增长趋势图" |
| FAIL | "收入稳步上升" |
| PASS | "柱状图：2023 年营收 ¥4.2 亿（#2196F3 蓝柱），2024 年营收 ¥6.8 亿（#2196F3 蓝柱），2025 预测 ¥9.1 亿（#FF9800 橙柱带虚线边框）。Y 轴范围 0-10 亿，单位标注 '亿元'" |

**检查方式**：搜索提示词中的数据相关描述，验证是否有具体数值 + 单位。对"增长"、"提升"、"扩大"等词，要求后续紧跟具体数字。

---

## Rule 4: 对比必须有维度和具体内容

**规则**：对比类内容必须列出对比维度，每个维度在每个对比对象上有具体内容。

| 类型 | 示例 |
|------|------|
| FAIL | "新旧方案对比" |
| FAIL | "与竞品的差异化分析" |
| PASS | "对比表格 5 行 3 列：表头 ['维度', '方案 A', '方案 B']。Row 1: ['部署周期', '4-6 周', '3 天']。Row 2: ['运维人力', '5 人+', '1 人']。Row 3: ['月成本', '¥8 万+', '¥2.5 万']。Row 4: ['弹性扩展', '手动扩容', '自动弹性']。Row 5: ['SLA', '99.5%', '99.99%']。方案 B 列使用 #FF9800 强调色背景" |

**检查方式**：搜索提示词中的对比描述，验证是否有维度列表和每个维度的具体内容。

---

## Rule 5: 视觉描述必须描述具体布局和数据排列

**规则**：视觉布局描述必须使用具体的几何语言（位置、尺寸、颜色、排列），禁止使用"适当"、"合理"、"美观"等主观词。

| 类型 | 示例 |
|------|------|
| FAIL | "合理布局的卡片展示" |
| FAIL | "美观的数据可视化" |
| FAIL | "用适当的颜色区分" |
| PASS | "3 列等宽卡片，宽度 340px，间距 40px，起始 x: 80。每张卡片：白色背景 #FFFFFF，圆角 12px，阴影 `filter: drop-shadow(0 2px 8px rgba(0,0,0,0.1))`。卡片内部：顶部色条（高 6px，分别为 #2196F3 / #4CAF50 / #FF9800），标题区（y+20），正文区（y+56）" |

**检查方式**：搜索提示词中的视觉描述，验证是否包含位置/尺寸/颜色等具体参数。标记"适当"、"合理"、"美观"等主观词为 FAIL。

---

## 自检清单 (Self-Check Checklist)

每个提示词在输出前，Prompt Engineer 必须逐项检查：

```
内容深度自检清单（每项必须通过）

[ ] R1 — 列表具体化
    □ 所有列表项都有具体内容（非仅标签）
    □ 没有 "展示 N 个要点" 后面无具体内容的描述

[ ] R2 — 流程具体化
    □ 所有流程/机制描述有具体步骤名
    □ 每个步骤有工具名、方法名或量化指标

[ ] R3 — 数据具体化
    □ 所有数据点有具体数值和单位
    □ 没有 "显著增长" 等无量词的模糊描述
    □ 图表数据集完整（X 轴 + Y 轴全部数据点）

[ ] R4 — 对比具体化
    □ 对比描述包含完整维度列表
    □ 每个维度在每个对象上有具体内容
    □ 对比结果有明确的高亮规则

[ ] R5 — 视觉描述具体化
    □ 布局描述包含位置和尺寸参数
    □ 颜色使用 HEX 值，不使用 "品牌色" 等抽象引用
    □ 没有出现 "适当"、"合理"、"美观" 等主观词

[ ] 自包含性
    □ 提示词内不引用外部文件路径
    □ 提示词内不要求 "参考上一页"
    □ 提示词内不使用 "如前所述" 等上下文依赖表述
    □ STYLE_INSTRUCTIONS 块包含完整的风格参数
```

**检查不通过时**：返回对应规则，补充具体内容，重新检查，最多迭代 3 次。3 次后仍不通过则报告该页面需要人工补充内容。
