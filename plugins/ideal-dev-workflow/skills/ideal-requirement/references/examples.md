# 需求文档示例

## Example 1: Problem Statement

**好的示例：**
> **问题陈述:**
> "当前用户登录模块存在以下问题：
> 1. 登录响应时间过长，平均耗时 3.5 秒，超出系统设计目标（1 秒）
> 2. 高并发场景下错误率上升至 5%，主要表现为连接超时
> 3. 用户反馈统计显示，登录相关问题占总投诉量的 23%"
>
> **特点:** 具体数据、明确对比、量化影响

**坏的示例：**
> **问题陈述:**
> "现在用户登录很慢，经常报错，用户体验不好，需要优化一下。"
>
> **问题:** 缺乏数据、主观评价、无量化指标

---

## Example 2: Features Description

**好的示例：**
> | 功能编号 | 功能名称 | 功能描述 | 优先级 |
> |----------|----------|----------|--------|
> | F001 | 用户登录 | 支持邮箱/手机号登录，包含验证码验证 | P0 |
> | F002 | 密码找回 | 通过邮箱找回密码，发送重置链接 | P1 |
> | F003 | 第三方登录 | 支持微信、GitHub OAuth 登录 | P2 |
>
> **特点:** 结构化、可追踪、优先级明确

**坏的示例：**
> **功能清单:**
> - 登录功能
> - 找回密码
> - 第三方登录
> - 还有其他一些功能...
>
> **问题:** 无编号、无优先级、无描述

---

## Example 3: Acceptance Criteria

**好的示例：**
> **功能验收:**
> - [ ] 用户可使用邮箱和手机号登录
> - [ ] 验证码 60 秒内发送成功
> - [ ] 登录响应时间 < 1 秒（P95）
> - [ ] 并发 100 用户时错误率 < 0.1%
>
> **特点:** 可测试、可量化、覆盖边界

**坏的示例：**
> **验收标准:**
> - 功能正常
> - 性能良好
> - 无 Bug
>
> **问题:** 无法验证、无具体指标

---

## Example 4: 多模块需求

**场景：** 用户提出"前端重构"需求，涉及 DesignToken、组件、图标、布局四个方面。

**处理方式：**
- 多模块需求在 P1 阶段生成统一的需求文档
- 模块拆分由 P5 编码计划阶段的 story 机制处理
- 每个模块对应一组 story，通过 story 的 executor 标签区分执行环境

**询问话术：**
```
需求涉及多个模块，将在编码计划阶段按模块拆分为独立的开发故事。
请确认以下模块范围：
1. DesignToken 基础层     — CSS Token 系统建立
2. 核心组件度量值更新     — UI 组件尺寸对齐
3. 图标库迁移             — Phosphor → Lucide
4. 应用层一致性审计       — 业务组件 token 对齐

是否确认？
```

**用户响应后：**
- 确认 → 生成统一需求文档，模块拆分留待 P5 编码计划阶段处理

**生成目录结构示例：**
```
docs/迭代/2026-03-24-前端重构/
├── P1-需求文档.md
├── 流程状态.md
└── ...（后续 P5 阶段生成 stories/ 目录）
```

> 模块拆分在 P5 编码计划阶段通过 story 机制处理，P1 阶段仅生成单一需求文档。
