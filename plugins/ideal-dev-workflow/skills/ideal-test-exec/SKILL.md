# ideal-test-exec（P11 测试执行）

## 角色定位

Phase Skill — **执行协调者**。

职责：
1. 加载所需上下文（P7 测试用例 + 项目代码）
2. 调度子智能体执行测试
3. 将报告写入文件系统
4. 返回执行摘要

**不负责**：更新 flow state、验证前置条件，协调评审。

---

## 输入

| 来源 | 内容 |
|------|------|
| P7-测试用例.md | 功能/边界/异常/截图/真实应用端到端用例 |
| 项目代码 | 待测试的代码 |

## 输出

| 文件 | 路径 |
|------|------|
| 测试报告 | `docs/迭代/{需求名称}/P11-测试报告.md` |

## 子智能体调度

| 调用时机 | 子智能体 | 任务 |
|----------|----------|------|

| 调试修复 | `debug` | 测试失败时的根因分析和修复 |

## 前端截图测试（必须）

> **强制要求**：当 P7 测试用例中包含前端截图用例时，必须通过 Chrome DevTools MCP 工具驱动浏览器执行截图，不得仅依赖 TypeScript 编译检查或手工截图。

### 截图方式

**Chrome DevTools MCP**：Agent 直接通过 MCP 工具控制浏览器，执行导航、交互和截图操作。无需编写任何测试脚本，零额外依赖。

### 截图保存规范

| 项目 | 规范 |
|------|------|
| 截图保存目录 | `docs/迭代/{需求名称}/screenshots/` |
| 截图命名规则 | `{用例编号}_{截图名称}.png` |
| 视口大小 | 自动使用当前屏幕分辨率（无需配置） |
| 截图方式 | Agent 通过 MCP 工具操控浏览器，全页面截图或指定元素截图 |
| 工具 | Chrome DevTools MCP（Agent 直接调用，零代码） |

### 截图执行流程

```
1. 确认 P7 中存在前端截图用例（TC-S-* 类型）
2. 创建截图保存目录：docs/迭代/{需求名称}/screenshots/
3. Agent 通过 Chrome DevTools MCP 工具操控浏览器：
   - 导航到目标页面
   - 等待页面渲染完成
   - 执行交互操作（点击、输入等）
   - 调用截图功能，保存到指定路径
4. 验证截图文件已保存到 screenshots/ 目录（文件存在 + 大小 > 10KB）
5. 在测试报告中引用截图路径
```

### 截图验证规则

| 检查项 | 说明 |
|--------|------|
| 文件存在 | 每个 TC-S-{id} 用例必须有对应的截图文件 |
| 文件大小 | 截图文件 > 10KB（小于则可能为空白页） |
| 命名规范 | 符合 `{TC编号}_{名称}.png` 格式 |
| 视觉完整性 | 页面无空白区域、无渲染错误（人工抽检） |

## 真实应用端到端（核心闭环）测试（必须）

> **强制要求**：当 P7 测试用例中包含 TC-E2E 用例时，必须在**真实运行的应用实例**上执行（启动应用 → Playwright MCP / Chrome DevTools MCP 驱动 → 逐步操作 → 断言 → 留证）。严禁把 E2E 用例降级为单元测试、类型检查或 curl 调用——那些是辅助证据，不是 E2E 的通过依据。

### 执行环境

1. 按项目实际方式启动应用（如 `npm run dev`），等待应用就绪（端口可达 / 窗口加载完成）。
2. 通过 **Playwright MCP**（驱动浏览器或可接入的桌面/Electron 窗口）或 **Chrome DevTools MCP**（驱动浏览器页面）连接到运行中的应用。

### E2E 执行流程

```
1. 确认 P7 中存在 TC-E2E 用例，读取每条用例的闭环类型、执行环境、步骤、预期结果
2. 启动真实应用（如 npm run dev）并等待就绪
3. Agent 通过 Playwright/CDP MCP 连接应用，逐个执行 TC-E2E 用例：
   - 单轮主流程：输入 → 等待响应 → 断言输出
   - 连续多轮：连续多轮输入 → 断言模型记住上文
   - 会话切换：创建/切换会话 → 断言历史保留与可恢复
   - 工具调用：触发工具 → 断言调用成功、返回被消费、模型据此继续
   - 审批 gate：触发高风险动作 → 断言审批弹出、放行/拒绝行为正确
   - 错误恢复：注入异常/失败 → 断言应用可恢复继续使用
4. 每步留存证据（截图/日志/录屏）到 docs/迭代/{需求名称}/screenshots/ 或 e2e-logs/
5. 断言失败 → 调用 Task(debug) 根因分析修复 → 重新执行该 E2E 用例（最多 3 次）
6. 将 E2E 执行结果与证据路径写入测试报告
```

### E2E 验证规则

| 检查项 | 说明 |
|--------|------|
| 真机执行 | E2E 用例必须在真实运行的应用上执行，不得降级为单测 |
| 闭环完整 | 每个适用闭环（多轮/会话/工具/审批）都有对应 E2E 用例被执行 |
| 证据留存 | 每条 E2E 用例有截图/日志/录屏证据 |
| 失败处理 | E2E 失败调 debug 根因修复，不靠"看起来对"放行 |

---

## 测试结果状态

| 状态 | 说明 |
|------|------|
| PASS | 测试通过 |
| FAIL | 测试失败 |
| SKIP | 跳过测试 |
| BLOCK | 被阻塞无法执行 |

## 缺陷严重程度

| 级别 | 说明 |
|------|------|
| 致命 | 系统崩溃、数据丢失 |
| 严重 | 核心功能不可用 |
| 一般 | 功能受限、体验问题 |
| 轻微 | 界面问题、文字错误 |

## 执行流程

```
Step 0: 验证前置产物
  └─ 检查 P7-测试用例.md 存在 → 否则终止

Step 1: 读取 P7-测试用例.md
Step 2: 准备测试环境
Step 3: 按优先级执行用例
  └─ Task(qa) → 记录执行结果，截图/日志留存
Step 3.5: 前端截图测试（仅当 P7 包含截图用例时）
  ├─ 创建截图目录 docs/迭代/{需求名称}/screenshots/
  ├─ Agent 通过 Chrome DevTools MCP 操控浏览器
  ├─ 逐个执行截图用例：导航 → 交互 → 截图 → 保存
  ├─ 验证截图文件已保存（文件存在 + 大小 > 10KB）
  └─ 将截图路径写入测试报告
Step 3.6: 真实应用端到端测试（仅当 P7 包含 TC-E2E 用例时）
  ├─ 启动真实应用（如 npm run dev）并等待就绪
  ├─ Agent 通过 Playwright/CDP MCP 连接运行中的应用
  ├─ 逐个执行 TC-E2E 用例：按闭环类型操作 → 断言预期 → 留证
  ├─ 失败时 Task(debug) 根因修复后重试（最多 3 次）
  └─ 将 E2E 结果与证据路径写入测试报告
Step 4: 发现缺陷时
  └─ Task(debug) → 根因分析，修复后重新验证
Step 5: 生成测试报告
Step 6: 更新流程状态
```

详见 `references/test-exec-flow.md`。
