# ideal-ccworkflow-executor（CC-Workflow 执行器）

## Overview

这个 skill 指导 CEO（小毛虫）如何基于 CC-Workflow 执行任务，确保高质量的任务完成。

**核心原则**：
- 虚拟公司员工参与评审
- 关键节点由老大最终确认
- 评审交流文档化

## 当你不知道如何执行任务时

当你被分配一个新任务时，使用这个 skill 来确定正确的执行流程。

## 关键概念

### 1. 评审模式（mode）

通过 `docs/迭代/{需求名称}/流程状态.md` 中的 `mode` 字段决定：

| mode | 说明 | 流程 |
|------|------|------|
| `yolo` | YOLO 模式 | 我(CEO)推进，不需要老大每个阶段评审 |
| `normal` | 正常模式 | 每个执行阶段后，老大评审确认 |

**默认**：`normal`（需要老大确认）

### 2. 评审交流文档

每个评审阶段创建一个评审交流文档：
- **位置**：`docs/迭代/{需求名称}/P{阶段号}-评审交流.md`
- **格式**：类似论坛，有来有回
- **内容**：
  ```markdown
  # P1 评审交流

  ## 第 1 轮

  **评审专家**：
  - 意见1：...
  - 意见2：...

  **执行者（xxx）**：
  - 已修改...
  - 理由如下...

  ## 第 2 轮

  ...

  ## 评审结果
  - ✅ 通过 / ❌ 需要修改
  ```

### 3. 阶段流转

**执行阶段**（Claude 执行）：
- P1、P3、P5、P7、P9、P11、P13、P15

**评审阶段**（评审 + 确认）：
- P2、P4、P6、P8、P10、P12、P14

### 4. 关键节点

每个**评审阶段**完成后，需要老大最终确认：
- P2 需求评审完成后
- P4 方案评审完成后
- P8 计划评审完成后
- P12 测试评审完成后
- P14 维基评审完成后
- P15 成果提交完成后

### 5. Debug 流程

在 P9 开发执行过程中，如果遇到 bug：
1. 暂停当前开发
2. 调用 `ideal-debugging` skill 进行调试
3. 修复 bug 后继续开发
4. 如果 bug 复杂，可能需要多轮 debug

## 执行流程

### 步骤 0：判断是否需要 Init（开始新任务时）

1. 检查项目根目录是否存在必要文件（README.md、CLAUDE.md）
2. 检查是否有 `.claude/skills/` 目录
3. **如果缺少**，调用 `ideal-init` skill 初始化项目
4. **如果已有**，直接进入 P1

### 步骤 1：读取项目信息

1. 读取 `docs/迭代/{需求名称}/流程状态.md`
2. 确定当前阶段（current_phase）
3. 确定评审模式（mode）
4. 确定项目路径

### 步骤 2：调用 Skill 执行任务

**⚠️ 重要：必须严格按照对应 Skill 的流程执行！**

**在 Discord 环境中**，由于不支持子代理的交互式对话，需要调整：

**方案 A：直接由我（CEO）进行对话**
- 我直接以 PM 的角色与你进行苏格拉底式对话
- 逐个问问题，一次只问一个
- 你回答后继续下一个问题
- 收集完必要信息后让你确认
- 确认后生成需求文档

**方案 B：跳过交互式对话**（仅限简单任务）
- 如果任务很简单，可以跳过苏格拉底式对话
- 直接生成需求文档
- 但需要在评审阶段补充细节

---

**⚠️ 重要：必须严格按照每个 Skill 的流程执行！**

### 步骤 2：判断当前阶段

**如果当前是执行阶段**（P1/P3/P5/P7/P9/P11/P13）：
1. 调用对应的 skill 执行任务
2. 任务完成后，进入对应的评审阶段

**如果当前是评审阶段**（P2/P4/P6/P8/P10/P12/P14）：
1. 根据 mode 决定评审流程
2. 执行评审流程
3. 评审通过后，进入下一阶段

### 步骤 3：执行评审流程

**YOLO 模式**：
1. 我读取当前阶段的产出物
2. **我调用评审专家子代理**（@评审专家）进行评审
3. 评审专家评审（正常多轮，直到通过）
4. 创建评审交流文档（如 P1-评审交流.md）
5. 记录评审意见和结果
6. **不需要老大确认**，直接进入下一阶段

**Normal 模式**：
1. 我读取当前阶段的产出物（如 P1-需求文档.md）
2. **我调用评审专家子代理**（@评审专家）
3. 评审专家提意见 → 记录到评审交流文档
4. 执行者根据意见修改 → 反馈到文档
5. 循环直到评审通过
6. **关键节点**：通知老大最终确认
7. 进入下一阶段

### 步骤 4：创建评审交流文档

**每个评审阶段都必须创建评审交流文档**！

- 位置：`P{评审阶段号}-评审交流.md`
- 内容：评审轮次、评审意见、评审结果

**⚠️ 重要：评审阶段编号**

CC-Workflow 的阶段流转：
- 执行阶段：P1 → P3 → P5 → P7 → P9 → P11 → P13 → P15
- **评审阶段**：P2 → P4 → P6 → P8 → P10 → P12 → P14

**评审交流文档的编号 = 评审阶段的编号**

| 执行阶段 | 评审阶段 | 评审交流文档 |
|----------|----------|-------------|
| P1 完成 | 进入 P2 评审 | P2-评审交流.md |
| P3 完成 | 进入 P4 评审 | P4-评审交流.md |
| P5 完成 | 进入 P6 评审 | P6-评审交流.md |
| P7 完成 | 进入 P8 评审 | P8-评审交流.md |
| P9 完成 | 进入 P10 评审 | P10-评审交流.md |
| P11 完成 | 进入 P12 评审 | P12-评审交流.md |
| P13 完成 | 进入 P14 评审 | P14-评审交流.md |

### 步骤 4：更新流程状态

每次阶段完成后，更新 `docs/迭代/{需求名称}/流程状态.md`

### 步骤 5：Debug（如有需要）

在 P9 开发执行过程中，如果遇到 bug：
1. 暂停当前开发
2. 调用 `ideal-debugging` skill 进行调试
3. 修复 bug 后继续开发
4. 如果 bug 复杂，可能需要多轮 debug：
- 更新 `current_phase`
- 更新 `status`
- 更新完成时间

## 上下文注入

当你需要让评审专家评审时，需要先注入上下文：

**步骤**：
1. 读取当前阶段的产出物（如 P1-需求文档.md）
2. 读取相关的参考文档（如 P3-技术方案.md）
3. 将这些内容作为上下文发送给评审专家

**示例 prompt**：
```
你需要评审 P1-需求文档。

上下文：
- 需求文档：{需求文档内容}
- 技术方案：{技术方案内容}

请提出评审意见。
```

## 注意事项

1. **不要跳过评审阶段**：即使 YOLO 模式，也需要快速评审
2. **评审交流必须文档化**：每个评审轮次都要记录
3. **关键节点必须通知老大**：每个评审阶段完成后
4. **使用绝对路径**：告诉子智能体时使用绝对路径

## 如何调用子智能体

当需要调用子智能体执行任务时：

### 1. 使用 sessions_spawn

```python
# 正确方式：传入 cwd 参数
sessions_spawn(
    runtime="subagent",
    mode="run",
    cwd="/项目绝对路径",  # 重要！
    agentId="architect",
    task="任务描述..."
)
```

### 2. 绝对路径

告诉子智能体时，**始终使用绝对路径**：
```
# 错误
请读取 docs/迭代/需求名称/P1-需求文档.md

# 正确
请读取 /Users/.../cc-workflow/docs/迭代/需求名称/P1-需求文档.md
```

### 3. 子智能体响应处理

- 等待子智能体完成（不要轮询）
- 子智能体完成后，读取产出物
- 根据产出物决定下一步

## 评审循环终止

评审可能多轮进行，需要设置终止条件：

### 最大轮次
- **默认**：3 轮
- 如果 3 轮后仍未通过：
  - 记录当前问题
  - 通知老大决策

### 终止条件
- 评审专家说"✅ 通过"
- 达到最大轮次（3 轮）仍未达成共识
- 老大介入决定

### 终止处理
1. 记录所有未解决的问题
2. 通知老大介入
3. 根据老大决策执行

## 是否需要 Init

在开始新任务前，需要判断是否需要初始化项目：

**需要 init 的情况**：
- 新项目
- 项目结构不完整
- 缺少配置文件

**不需要 init 的情况**：
- 已有完整项目结构
- 只需要在现有基础上开发

**判断步骤**：
1. 检查项目根目录是否存在必要文件（如 README.md、CLAUDE.md）
2. 检查是否有 `.claude/skills/` 目录
3. 如果缺少，调用 `ideal-init` skill
