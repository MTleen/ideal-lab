# Technical Proposal Writing (P4)

## Overview

根据 P3 任务拆分计划，启动多个 Agent 并行撰写各章节。写作需遵循 `technical-proposal-style` 的规范。

## 输入

- P3-任务拆分计划.md
- P1-需求分析.md
- P2-技术方案大纲.md
- 项目目录和方案名称

## 输出

`{项目目录}/项目文档/技术方案/sections/*.md`

## 写作规范（参考 technical-proposal-style）

### 写作三避免
- **避免第一人称**：不用"我们"、"你"，采用无主句或以方案为主语
- **避免问句标题**：标题使用陈述句，不用"为什么"、"如何"等问句
- **避免负面表述**：用"方案选择依据"替代"放弃的原因"

### 技术调研要求（内部参考，不写入正文）

调研是写作前的**准备工作**，调研成果仅供写作者参考，**不得作为正文中的独立段落或章节输出**。

调研内容包括：
1. **技术原理调研**：该技术/方法的核心原理是什么？涉及哪些基础理论？
2. **行业最佳实践**：同类项目中通常采用哪些方案？有什么成熟经验可以借鉴？
3. **公式与模型**：涉及计算、评估、评分的技术内容，必须包含必要的公式或数学模型。
4. **数据支撑**：该技术在实际应用中的性能数据、效果数据。

写作时，将调研成果**内化**为正文内容——即把调研发现的原理、方案、数据融入正文叙述中，而非以"技术调研"标题段落的形式显式输出。

### 语言三要求
- **正式**：避免口语化、比喻化、情感化表述
- **准确**：数值带范围和来源，术语首次定义
- **客观**：避免主观判断和夸大表述

### 公式规范要求（强制）
技术方案中涉及计算、评估、评分的章节，必须包含必要的公式。公式格式规范：
- 使用标准 LaTeX 公式格式（行内公式 `$...$`，独立公式 `$$...$$`）
- 公式中每个变量必须明确定义
- 公式必须有应用说明（说明在什么场景下使用、如何计算）
- 公式的具体形式由写作者根据该章节所采用的技术方法动态确定

### 证据三要素
- **有对照**：说明与什么对比
- **有来源**：数据来自何处
- **可核验**：提供验证方法

### 格式五规范
- **标题层级清**：使用 X.Y.Z 格式，避免跳跃
- **图表编号明**：表 X-Y、图 X-Y 格式
- **分隔不滥用**：分隔线仅用于章节间
- **列表不过度**：无序列表 ≤ 全文行数 5%，优先段落叙述
- **表格不代文**：文字论述为主，表格仅辅助总结

## 执行步骤

### 1. 读取任务计划

读取 `P3-任务拆分计划.md`，获取任务列表：

```markdown
### 任务 1: 1. 项目概述
- **文件**：`sections/1-项目概述.md`
- **预估字数**：800字
- **章节层级**：1.1 ~ 1.3
- **上下文**：
  - 来源需求：{需求片段}
  - 上级概述：{大纲中的一句话概述}
- **写作要点**：
  - 项目背景需要交代行业背景和客户痛点
  - 项目目标要可量化、可衡量
```

### 2. 技术调研（内部参考，不写入正文）

**在撰写任何章节之前，必须完成技术调研。** 调研成果仅作为写作者的内部参考，**不得在正文中以独立章节或段落形式输出**。

#### 调研内容清单

1. **该技术领域的基础原理**：涉及的核心理论是什么？有哪些公认的技术模型？
2. **行业应用现状**：同类民航/航空 IT 项目中通常采用哪些方案？有什么成熟经验？
3. **必要的公式与数学模型**：该技术涉及计算、评估、评分吗？若有，必须包含公式。
4. **实际应用数据**：该技术在实际项目中的性能数据、效果数据。

调研完成后，将调研成果**内化**到正文叙述中——用原理说明替代泛泛而谈，用数据支撑替代主观论断，用公式模型替代模糊描述。

### 3. 并行启动 Agent

使用 `Agent` 工具并行启动多个写作 Agent：

```python
for task in tasks:
    Agent(
        subagent_type="general-purpose",
        description=f"写作: {task.title}",
        prompt=f"""
        你是一个技术方案撰写专家。请根据以下上下文撰写技术方案章节。

        ## 任务信息
        - 章节：{task.title}
        - 预估字数：约{task.word_count}字
        - 章节层级：{task.headings}

        ## 需求上下文
        {task.context}

        ## 写作要点
        {task.writing_points}

        ## 技术调研要求（内部参考，不写入正文）
        在撰写正文之前，必须先完成技术调研。调研成果仅作为写作者的内部参考，**不得在正文中以"技术调研"章节或段落形式输出**。调研完成后，将调研成果内化到正文叙述中：
        1. 该技术领域的基础原理和核心理论 → 融入正文原理说明
        2. 行业应用现状和成熟经验 → 融入正文方案描述
        3. 必要的公式与数学模型 → 以正规公式块形式直接出现在正文中
        4. 实际应用数据 → 融入正文数据支撑

        ## 写作规范（必须遵守）
        1. 避免第一人称：不用"我们"、"你"，采用无主句
        2. 避免问句标题：标题使用陈述句
        3. 避免负面表述：用"方案选择依据"替代"放弃的原因"
        4. 正式：避免口语化、比喻化
        5. 准确：数值带范围和来源
        6. 客观：避免主观判断和夸大
        7. 列表不过度：优先段落叙述
        8. 表格不代文：文字论述为主
        9. 公式规范：涉及计算/评估的章节必须包含公式，使用 $...$ 或 $$...$$ 格式

        ## 输出路径
        直接写入文件：{task.output_path}

        请直接输出内容，不要包含其他说明。
        """,
        run_in_background=True
    )
```

### 4. 并发控制

- **最大并发数**：建议 5 个并行任务（与 5 个写作组对应）
- **任务分组**：每组同时启动，组内可串行或并行

### 5. 结果收集

各 Agent 完成后：
1. 读取各 `sections/` 下的 md 文件
2. 检查是否有任务失败
3. 如有失败，重新执行该任务
4. **验证公式**：检查涉及计算/评估的章节是否包含必要公式

## 写作模式示例

### 技术方案模式
```
[技术名称] 是一种 [定义]。
本方案选择该技术，基于以下考量：
1. [考量1]：具体分析...
2. [考量2]：具体分析...
3. [考量3]：具体分析...

在实现上，通过 [具体方式] 实现 [功能]。
该方案的优势在于 [优势1]、[优势2]。
```

### 对比模式
```
传统方法：[传统方案] + 局限性分析
本方案：[推荐方案] + 优势分析
对比结果：[量化对比数据]
```

### 架构描述模式
```
整体架构分为三层：
第一层为 [展示层]，负责 [功能]，采用 [技术] 实现；
第二层为 [业务层]，负责 [功能]，采用 [技术] 实现；
第三层为 [数据层]，负责 [功能]，采用 [技术] 实现。
层间通过 [协议/格式] 进行交互。
```

## 质量检查

完成后，检查以下问题：

| 问题 | 检查方式 |
|------|---------|
| 第一人称 | 搜索"我们"、"你" |
| 问句标题 | 搜索"为什么"、"如何" |
| 口语化 | 搜索"打通"、"靠...补位" |
| 量化证据 | 检查数值是否有范围和来源 |
| 列表过度 | 统计无序列表占比 |
| 表格代文 | 检查是否仅有表格缺乏论述 |
| **公式完整性** | 检查涉及计算/评估/评分的章节是否包含相应公式；公式形式由写作者根据该章节技术方法动态确定，不是预设的 |
