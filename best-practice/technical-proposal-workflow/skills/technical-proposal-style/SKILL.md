# 技术标书写作风格规范

## 概述

本 skill 提供技术标书/投标文档的写作风格规范和质量检查标准，参考 NSFC 科研基金申请书的写作要求，确保文档正式、准确、客观。

## 核心原则

### 写作三避免
- **避免第一人称**：不用"我们"、"你"，采用无主句或以方案为主语
- **避免问句标题**：标题使用陈述句，不用"为什么"、"如何"等问句
- **避免负面表述**：用"方案选择依据"替代"放弃的原因"

### 语言三要求
- **正式**：避免口语化、比喻化、情感化表述
- **准确**：数值带范围和来源，术语首次定义
- **客观**：避免主观判断和夸大表述

### 证据三要素
- **有对照**：说明与什么对比
- **有来源**：数据来自何处
- **可核验**：提供验证方法

### 格式五规范
- **标题层级清**：使用X.Y.Z格式，避免跳跃
- **图表编号明**：表X-Y、图X-Y格式
- **分隔不滥用**：分隔线仅用于章节间
- **列表不过度**：无序列表≤全文行数5%，优先段落叙述
- **表格不代文**：文字论述为主，表格仅辅助总结

## 使用方式

### 1. 文档检查

当用户提供文档要求检查时：

1. 读取文档内容
2. 按 [quality_checklist.md](references/quality_checklist.md) 逐项检查
3. 输出问题清单（含位置和严重程度）
4. 提供改写建议

### 2. 内容优化

当用户要求优化特定内容时：

1. 识别问题类型（写作风格/语言风格/量化证据/格式）
2. 查阅对应参考文档
3. 提供具体改写方案

### 3. 新内容生成

当用户要求按规范生成新内容时：

1. 明确内容类型和用途
2. 按规范模板生成
3. 确保符合所有风格要求

## Markdown 转 Word

当需要将技术标书从 Markdown 转换为 Word 格式时：

### 使用转换脚本

```bash
# 基本用法
./scripts/markdown_to_word.sh 章节目录/

# 指定输出文件
./scripts/markdown_to_word.sh -o 投标文件.docx 章节目录/
```

### 转换要求

转换后的 Word 文档必须满足：

| 要求 | 标准 |
|-----|------|
| 正文字体 | 宋体，小四（12pt），黑色 |
| 标题字体 | 黑体，三号/四号，黑色 |
| 行距 | 1.5 倍 |
| 首行缩进 | 2 字符 |
| 段前段后 | 0 |
| 标题斜体 | 无 |
| 表格 | 居中对齐 |
| 目录 | 无（手动添加） |

详见 [markdown_to_word.md](references/markdown_to_word.md)

## 常见问题快速修复

### 问题1：第一人称过多

**识别**：文档中大量"我们选择"、"我们评估"

**修复模式**：
```
原文：我们选择微服务架构，因为...
改为：微服务架构被选定，其优势在于...
```

### 问题2：问句标题

**识别**：标题为"为什么需要X？"、"如何实现Y？"

**修复模式**：
```
原文：##### 为什么需要这些数据？
改为：#### X.Y.Z 数据需求分析
```

### 问题3：口语化表述

**识别**：使用"打通"、"靠...补位"、"够用"等

**修复模式**：
```
原文：靠卫星补位
改为：通过遥感技术补充监测覆盖
```

### 问题4：量化证据不足

**识别**：数值无来源、无范围、无对比

**修复模式**：
```
原文：准确率约90%
改为：准确率达到88%-92%（基于测试集验证，n=1200）
```

### 问题5：标题层级不规范

**识别**：五级标题作为段落引导、编号跳跃

**修复模式**：
```
原文：##### 为什么需要这些数据？
改为：整合到四级标题下的段落首句
```

### 问题6：无序列表过度使用（重要）

**识别**：大量使用 `-` 列表替代段落论述

**修复模式**：
```
❌ 原文：
Web端技术架构：
- 框架：Vue 3.x + TypeScript
- UI：Element Plus / Ant Design Vue
- 状态：Pinia
- 构建：Vite 5.x

✅ 改为：
Web端采用Vue 3.x框架结合TypeScript语言开发，UI组件库选用Element Plus或Ant Design Vue。状态管理通过Pinia实现，构建工具采用Vite 5.x以提升开发效率。
```

### 问题7：以表代文（重要）

**识别**：技术选型仅用表格呈现，缺乏文字论证

**修复模式**：
```
❌ 原文：
经评估：

| 维度 | 方案A | 方案B | 选择 |
|-----|-------|-------|-----|
| 性能 | 高 | 中 | A |
| 成本 | 高 | 低 | B |

**选择方案A**：性能优先。

✅ 改为：
本系统技术选型需综合考虑性能、成本、可维护性等因素。

**方案A分析**：该方案采用高性能架构，响应时间可达50ms以内，吞吐量提升40%。代价是硬件成本较高，约需增加30%预算。适用于对实时性要求较高的核心业务场景。

**方案B分析**：该方案采用经济型架构，成本可控，部署简单。但响应时间约为200ms，吞吐量受限。适用于非核心业务场景。

**综合评估与选择**：基于核心业务对实时性的严格要求，选择方案A作为主系统架构。虽然初期投入较高，但从业务连续性和用户体验角度，该投资是值得的。

（表格作为辅助总结）
**表X-X 方案对比总结**

| 维度 | 方案A | 方案B | 选择 |
|-----|-------|-------|-----|
| 性能 | 高 | 中 | A |
| 成本 | 高 | 低 | - |
```

## 检查输出格式

### 问题清单格式

```markdown
## 检查结果

### 问题统计
| 类别 | 问题数量 | 严重程度 |
|-----|---------|---------|
| 写作风格 | X | 高/中/低 |
| 语言风格 | X | 高/中/低 |
| 量化证据 | X | 高/中/低 |
| 格式规范 | X | 高/中/低 |

### 问题详情
1. **[类别]** [问题描述] - 位置：X.Y节 - 严重程度：高/中/低
   - 原文：...
   - 建议：...

2. ...
```

### 优先级说明

- **高**：必须修改，影响文档专业性和可信度
- **中**：建议修改，提升文档质量
- **低**：可选修改，锦上添花
